# 数学 / 理论 / 证明型论文写作指南

## 1. 适用范围

本指南适用于以**定义、假设、理论模型、定理、命题、引理、证明、反例、构造和理论解释**为核心知识生产方式的论文。其目标是帮助 Agent 在辅助用户撰写、重构、审稿、润色此类论文时，判断论文是否不仅“证明正确”，而且清楚说明“为什么重要”。

适用领域包括但不限于：

- 数学；
- 应用数学；
- 理论计算机科学；
- 统计理论；
- 概率论与随机过程；
- 优化理论；
- 信息论；
- 经济理论；
- 博弈论；
- 逻辑；
- 形式化方法；
- 理论物理；
- 复杂系统理论；
- 机器学习理论；
- 控制理论；
- 数理金融；
- 密码学理论；
- 其他以形式化模型和证明为主要贡献的研究。

本指南适合以下论文类型：

- 提出新定义或新理论框架的论文；
- 证明新定理的论文；
- 推广已有定理的论文；
- 收紧上下界的论文；
- 给出存在性、唯一性或构造性结果的论文；
- 证明不可能性、下界或反例的论文；
- 建立等价关系、对偶关系或表示定理的论文；
- 分析算法、模型、机制或系统的理论性质的论文；
- 给出新的证明方法、技术工具或理论解释的论文；
- 将已有理论统一到一个更一般框架中的论文。

这类论文的核心不是展示大量实验结果，而是建立一条可被同行逐步检验的形式化论证链：

```text
问题背景
→ 概念与符号
→ 假设与模型
→ 主结果陈述
→ 证明策略
→ 形式证明
→ 例子 / 反例 / 特例
→ 与已有理论的关系
→ 理论意义与边界
```

理论论文不仅要证明“命题为真”，还要让读者理解：

```text
为什么这个问题值得研究？
为什么这些假设合理？
定理相对于已有结果新在哪里？
证明技术的关键思想是什么？
结果的适用边界在哪里？
该结果改变了我们对什么问题的理解？
```

## 2. 不适用范围

本指南不应机械套用于以下论文或文本。

### 2.1 以实验数据为主要证据的自然科学论文

如果论文主要通过实验、观测、测量和表征解释自然现象，核心证据是实验数据而非形式证明，则应使用实验 / 观察型自然科学论文指南。

理论结果可以作为解释工具，但不能替代实验设计、误差分析和数据质量说明。

### 2.2 以算法 benchmark 为主要贡献的计算论文

如果论文主要提出算法或模型，并通过数据集、baseline、指标和实验结果证明有效，则应使用计算 / 算法 / 数据驱动型论文指南。

理论分析可以是其中一部分，但如果贡献主要依赖实验比较，则不应写成纯理论论文。

### 2.3 工程系统或设计型论文

如果论文主要贡献是系统架构、装置、工具、协议、平台或工程实现，则应使用工程 / 系统 / 设计型论文指南。

理论分析可以说明系统性质，但系统论文的核心证据通常是实现与评价，而不是定理证明。

### 2.4 教材式讲义或综述

如果文本主要介绍已有理论、整理经典证明或教学说明，而没有原创问题、原创结果或新的组织框架，则不构成理论研究论文。

### 2.5 纯计算验证或数值猜想

如果论文主要通过数值实验、仿真或计算机搜索提出猜想，但没有形式证明，则不能套用证明型论文的标准。  
这类论文可以提出 conjecture，但不能把数值证据写成 theorem。

### 2.6 形式上有证明但贡献不清的技术笔记

有些文本包含正确证明，但只是微小变形、已知结果的重述或缺少理论意义说明。  
这类文本需要先明确贡献位置，否则不应包装成完整理论论文。

## 3. 知识生产逻辑

数学 / 理论 / 证明型论文通过**形式化、抽象、推导和证明**生产可信知识。

其基本逻辑是：

```text
现实问题 / 理论问题 / 已有结果限制
→ 抽象为形式化模型
→ 给出定义、符号和假设
→ 提出可精确陈述的命题
→ 证明命题在假设下成立或不成立
→ 用例子、反例和特例说明边界
→ 解释结果相对于已有理论的贡献
```

### 3.1 这类论文依靠什么产生可信知识

理论论文的可信性不来自样本量或实验重复，而来自以下因素：

| 可信性来源 | 写作中应体现什么 | Agent 判断重点 |
|---|---|---|
| 清晰定义 | 对象、符号、概念、空间、映射、关系清楚 | 定义是否无歧义 |
| 明确假设 | 定理在哪些条件下成立 | 是否存在隐藏假设 |
| 精确陈述 | 定理、命题、引理、推论条件和结论清楚 | 是否可被检验 |
| 严密证明 | 每一步推理可追踪 | 是否有逻辑跳跃 |
| 证明结构 | 读者知道证明路线 | 是否只堆技术细节 |
| 例子与反例 | 展示结果适用范围和边界 | 假设是否必要 |
| 与已有结果比较 | 说明新结果相对已有理论的改进 | novelty 是否清楚 |
| 直觉解释 | 帮助读者理解结果为何成立 | 是否只给形式推导 |
| 边界说明 | 结果何时不成立或尚未覆盖 | 是否过度推广 |

### 3.2 理论论文与实验论文、算法实验论文的差异

| 维度 | 理论 / 证明型论文 | 实验自然科学论文 | 算法 / 数据驱动论文 |
|---|---|---|---|
| 主要证据 | 定义、定理、证明、反例 | 实验、观测、测量 | benchmark、baseline、指标 |
| 核心问题 | 在何种假设下什么命题成立 | 自然现象如何发生 | 方法在任务上是否有效 |
| 可验证性 | 同行逐步检查证明 | 重复实验或复核数据 | 复现实验和比较结果 |
| 贡献形式 | 定理、界、等价、存在性、不可能性 | 发现、机制、测量 | 新模型、算法、数据、协议 |
| 失败风险 | 证明正确但意义不清 | 数据多但机制不清 | SOTA 但解释不足 |
| 写作重点 | 直觉、条件、证明路线、结果边界 | 方法、误差、图表、讨论 | 任务、实验协议、分析 |

理论论文常问：

```text
在精确定义的对象和假设下，什么命题可以被证明？
```

实验论文常问：

```text
在可观察世界中，什么现象被数据支持？
```

算法实验论文常问：

```text
在给定任务、数据和指标下，方法是否优于已有方案？
```

### 3.3 理论论文的核心闭环

理论论文必须形成以下闭环：

```text
问题为什么重要
→ 抽象模型是否合适
→ 假设是否清楚且合理
→ 主结果是否精确且有新意
→ 证明是否完整
→ 例子 / 反例是否说明边界
→ 与已有理论的关系是否清楚
→ 读者是否理解结果意义
```

如果只完成“证明是否完整”，但没有解释“为什么重要”，论文仍然可能被认为贡献不足。

### 3.4 Claim-Evidence-Warrant 在理论论文中的形式

在理论论文中：

| 元素 | 理论论文中的对应 |
|---|---|
| Claim | 定理、命题、引理、推论、构造、不可能性结果 |
| Evidence | 形式证明、反例、构造、归约、推导 |
| Warrant | 证明策略、关键思想、已有定理的适用条件、逻辑连接 |
| Boundary | 假设、特例、反例、紧性说明、适用范围 |

示例：

```text
Claim：在假设 A、B、C 下，所有满足条件 X 的对象都具有性质 Y。
Evidence：定理 1 的证明。
Warrant：证明先将问题归约到紧性引理，再利用不变量控制极限过程，最后排除边界退化情形。
Boundary：反例 2 表明，如果去掉假设 B，结论 Y 不再成立。
```

Agent 检查规则：

```text
每当论文出现“显然”“容易得到”“由此可知”“不难证明”“直接推出”时，必须检查：
1. 中间推理是否真的清楚？
2. 是否调用了未陈述的定理？
3. 是否需要额外假设？
4. 是否存在边界情况？
5. 是否需要补一个 lemma？
```

## 4. 典型论文结构

数学 / 理论 / 证明型论文常见结构如下：

```text
Title
Abstract
Keywords / MSC / CCS / JEL codes, if applicable

1. Introduction
2. Related Work / Background
3. Preliminaries
4. Problem Setup / Model
5. Main Results
6. Proof Overview / Proof Sketch
7. Proofs
8. Examples / Counterexamples
9. Applications / Implications
10. Discussion / Limitations
11. Conclusion

References
Appendix
```

不同领域会有变体。例如：

- 数学论文可能将 proof 直接放在 main results 后；
- 理论计算机科学论文常包含 model、algorithm、theorem、proof、lower bound；
- 统计理论论文常包含 assumptions、main theorem、rates、proof outline、technical lemmas；
- 经济理论论文常包含 model、equilibrium definition、propositions、comparative statics；
- 逻辑或形式化方法论文常包含 syntax、semantics、soundness、completeness；
- 理论物理论文可能包含模型构造、对称性分析、极限情形和物理解释。

### 4.1 章节功能总览

| 章节 | 核心功能 | 不应写成 |
|---|---|---|
| Introduction | 解释问题重要性、已有缺口、主结果直觉和贡献 | 只列定理编号 |
| Related Work | 定位已有理论结果和本文差异 | 文献罗列 |
| Preliminaries | 给读者进入证明所需工具 | 教材式大百科 |
| Problem Setup | 精确定义对象、符号、假设和问题 | 模糊背景描述 |
| Main Results | 清楚陈述定理、命题、推论 | 条件混乱的公式堆 |
| Proof Sketch | 给出证明路线和关键思想 | 证明的重复 |
| Proofs | 完整、可检查地证明结果 | 技术细节无结构 |
| Examples / Counterexamples | 解释结果边界和假设必要性 | 装饰性例子 |
| Applications / Implications | 展示结果如何使用或改变理解 | 泛泛意义 |
| Discussion | 讨论假设强弱、边界、开放问题 | 重复定理 |
| Conclusion | 收束贡献与意义 | 口号式总结 |

### 4.2 推荐结构一：标准定理论文

```text
1. Introduction
2. Preliminaries
3. Main Theorems
4. Proof Strategy
5. Proof of Main Theorem
6. Examples and Counterexamples
7. Applications
8. Conclusion
```

适合数学、概率、统计理论、优化理论等论文。

### 4.3 推荐结构二：理论计算机科学论文

```text
1. Introduction
2. Related Work
3. Model and Problem Definition
4. Main Results
5. Upper Bound / Algorithm
6. Lower Bound / Hardness
7. Proofs
8. Discussion
9. Conclusion
```

适合复杂度、算法理论、密码学、学习理论、形式化方法。

### 4.4 推荐结构三：统计 / 机器学习理论论文

```text
1. Introduction
2. Problem Setup
3. Assumptions
4. Main Results
5. Interpretation of Rates / Bounds
6. Proof Sketch
7. Technical Lemmas
8. Experiments or Simulations, if illustrative
9. Limitations
10. Conclusion
```

注意：如果有仿真实验，通常是说明或补充直觉，不是证明主结果的主要证据。

### 4.5 推荐结构四：理论模型 / 经济理论论文

```text
1. Introduction
2. Model
3. Equilibrium / Solution Concept
4. Main Propositions
5. Comparative Statics
6. Extensions
7. Examples
8. Implications
9. Conclusion
```

重点是模型假设、均衡概念、命题、解释和经济含义。

## 5. 各章节写作指南

## 5.1 标题 Title

### 写作目标

标题应准确传达论文的理论对象、主结果类型或核心问题，使目标读者能够判断论文属于哪个理论方向以及贡献是什么。

### 常见标题类型

| 类型 | 结构 | 示例形式 |
|---|---|---|
| 对象 + 结果 | Bounds for X under Y | 界、率、估计 |
| 理论关系 | Equivalence between X and Y | 等价、表示、对偶 |
| 不可能性 | Impossibility of X under Y | 下界、不可能性 |
| 存在性 | Existence of X in Y systems | 存在性、构造 |
| 推广型 | A Generalization of X to Y | 推广经典定理 |
| 技术型 | A New Proof of X via Y | 新证明方法 |
| 模型型 | A Theory of X with Y | 理论模型 |
| 直觉型主副标题 | When X Fails: A Lower Bound for Y | 强调边界 |

### 常见错误

- 标题过宽，如“关于某理论的研究”；
- 标题只写对象，不写结果类型；
- 标题声称 general，但结果条件很强；
- 标题声称 optimal，但正文没有紧性或下界；
- 标题声称 theory，但只是一个 lemma；
- 标题使用“突破性”“全新”等宣传词。

### Agent 质量判断

```text
1. 标题是否体现理论对象？
2. 是否体现结果类型：定理、界、等价、存在性、不可能性等？
3. 是否与主结果范围一致？
4. 是否避免过度声称？
5. 是否便于同行检索？
```

---

## 5.2 摘要 Abstract

### 写作目标

理论论文摘要应压缩以下链条：

```text
问题背景
→ 已有理论缺口
→ 本文主结果
→ 证明思想或技术
→ 与已有结果关系
→ 意义或应用
```

### 推荐结构

```text
1. 问题句：说明理论问题或对象。
2. 缺口句：说明已有结果的限制。
3. 结果句：陈述本文证明了什么。
4. 条件句：说明结果在哪些假设或模型下成立。
5. 技术句：概括证明方法或关键工具。
6. 意义句：说明结果如何推广、收紧、统一或解释已有理论。
```

### 示例骨架

```text
本文研究 X 空间中满足 Y 条件的对象是否具有 Z 性质。已有结果在 A 假设下证明了该性质，但其是否可推广到 B 情形仍不清楚。本文证明，在条件 C 和 D 下，Z 性质依然成立，并给出与参数 n 相关的最优上界。证明的核心是构造新的能量泛函，并结合紧性引理排除边界退化情形。本文结果推广了 E 的经典定理，同时反例表明条件 D 在一般情形下不可去除。
```

### 常见错误

- 摘要只说“本文研究……并证明若干结果”；
- 不说明已有缺口；
- 不陈述主结果内容；
- 只列 theorem number；
- 不说明假设或适用范围；
- 过度声称“彻底解决”；
- 不说明证明技术或结果意义。

### Agent 质量判断

```text
1. 摘要是否明确理论问题？
2. 是否说明已有理论不足？
3. 是否清楚陈述主结果？
4. 是否说明结果条件？
5. 是否说明证明思想或技术贡献？
6. 是否说明意义和边界？
```

## 5.3 Introduction 引言

### 写作目标

理论论文的 Introduction 必须完成两件事：

```text
1. 让读者相信问题重要；
2. 让读者理解主结果为何有意义。
```

这比单纯列出定理更重要。理论论文常被拒，不是因为证明错误，而是因为读者看不出结果为何值得发表。

### 推荐组织方式

```text
1. 背景：介绍理论对象或问题来源。
2. 核心问题：用尽量少的技术语言说明研究问题。
3. 已有结果：概括已有理论进展。
4. 缺口：指出尚未解决的关键限制、条件或边界。
5. 本文主结果：用非完全技术化语言解释。
6. 结果直觉：说明为什么结果合理或令人意外。
7. 技术贡献：说明证明方法或新工具。
8. 例子 / 应用：说明结果如何使用。
9. 贡献列表和文章结构。
```

### 引言中的问题重要性写法

弱写法：

```text
X 是一个重要问题，已有很多研究。
```

强写法：

```text
X 问题处于 A 理论和 B 理论的交界处。一方面，A 理论中的经典结果依赖紧性假设；另一方面，B 理论的典型对象往往不满足该假设。因此，理解 X 是否仍在弱条件下成立，是判断 A 理论能否扩展到 B 场景的关键。
```

### 已有理论缺口写法

弱写法：

```text
已有研究没有讨论一般情况。
```

强写法：

```text
已有结果证明了有限维情形下的收敛率，但证明依赖于谱间隙下界。当维度随样本量增长时，该谱间隙可能退化，因此现有证明无法直接推出高维情形下的同类结论。
```

### 主结果直觉写法

理论论文应在正式定理前解释直觉。

```text
从直觉上看，虽然假设 A 被弱化后局部控制不再成立，但条件 B 提供了一种全局补偿机制。本文的证明正是通过构造一个同时捕捉局部振荡和全局约束的势函数来实现这一点。
```

### 贡献列表写法

弱贡献：

```text
本文证明了几个新定理。
```

强贡献：

```text
本文的贡献有三点：
1. 将经典 X 定理从紧空间推广到满足弱局部紧性条件的度量空间。
2. 给出参数依赖的最优上界，并通过反例证明该界不可进一步改善。
3. 提出一种基于能量分解的新证明方法，可用于处理 Y 类问题中的边界退化。
```

### 常见错误

- 引言过早进入符号；
- 只列定理，不解释意义；
- 背景像教材；
- 没有说明已有结果具体限制；
- 不说结果为什么 surprising 或 useful；
- 证明技术贡献不清；
- 贡献只写“证明了新结论”；
- 文章结构预告过长但不服务理解。

### Agent 质量判断

```text
1. 引言是否说明问题来源？
2. 是否说明已有结果和限制？
3. 是否清楚表达本文主结果？
4. 是否给出主结果直觉？
5. 是否说明理论贡献类型？
6. 是否让非最窄领域读者理解为什么重要？
7. 是否避免只堆符号和定理编号？
```

## 5.4 Related Work / Background 相关工作与背景

### 写作目标

Related Work 在理论论文中用于定位本文结果相对于已有定理、证明技术和理论框架的位置。

它应回答：

```text
已有结果证明了什么？
它们依赖哪些假设？
本文放宽、加强、推广、收紧或反驳了什么？
本文证明技术与已有技术有什么关系？
```

### 推荐组织方式

#### 方式一：按结果强弱组织

```text
经典结果
→ 后续推广
→ 当前最强结果
→ 本文改进
```

#### 方式二：按假设类型组织

```text
强凸性假设下的结果
→ 弱凸性假设下的结果
→ 非凸情形的已知困难
→ 本文处理的条件
```

#### 方式三：按证明技术组织

```text
紧性方法
→ 对偶方法
→ 概率耦合方法
→ 本文的新分解技术
```

#### 方式四：按问题变体组织

```text
有限维情形
→ 无限维情形
→ 随机情形
→ 本文统一框架
```

### 好的 Related Work 段落

```text
经典的 X 定理在假设 A、B 下给出性质 Y。后续工作将条件 A 弱化为 A'，但仍保留了 B 中的有界性要求。最近的研究表明，如果完全去掉 B，则结论在一般情形下失败。本文位于这两类结果之间：我们不要求 B 的全局有界性，但引入局部增长条件 C，并证明该条件足以恢复 Y 性质。
```

### 常见错误

- 只罗列作者和年份；
- 不说明已有定理条件；
- 不比较结果强弱；
- 不说明证明技术差异；
- 忽略最接近结果；
- 把已有结果说得过弱；
- 对已有定理的适用条件描述不准确；
- 没有说明本文结果是否 strict generalization。

### Agent 质量判断

```text
1. 是否准确陈述已有结果条件和结论？
2. 是否说明本文与已有结果的逻辑关系？
3. 是否比较假设强弱和结论强弱？
4. 是否说明证明技术差异？
5. 是否避免文献罗列？
6. 是否处理最接近的相关工作？
```

## 5.5 Preliminaries 预备知识

### 写作目标

Preliminaries 的目标是让读者具备理解主结果和证明所需的最小工具。

它不应写成教材章节，而应服务于后文证明。

### 应包含内容

- 基础符号；
- 标准定义；
- 本文采用的约定；
- 已知定理；
- 引理或技术工具；
- 记号说明；
- 证明中反复使用的概念；
- 与标准用法不同的定义说明。

### 符号写作原则

好的符号系统应：

```text
1. 一致；
2. 不过度复杂；
3. 不重复使用同一符号表示不同对象；
4. 明确变量取值范围；
5. 明确常数是否依赖参数；
6. 明确随机量、确定量、函数、集合和空间的区别；
7. 重要符号首次出现时定义。
```

### 定义写作原则

好的定义应满足：

```text
1. 必要：后文确实使用；
2. 精确：没有歧义；
3. 最小：不包含无关条件；
4. 可检查：读者能判断对象是否满足；
5. 与已有定义关系清楚；
6. 如为新定义，应解释动机。
```

### 定义示例结构

```text
Definition 2.1 (名称).
Let ... be ...
We say that ... is ... if ...
```

定义后应加解释：

```text
The condition above can be viewed as a local version of ...
It is weaker than ... but stronger than ...
Example 2.2 shows that ...
```

### 假设写作原则

假设不应藏在正文中。应集中编号并解释。

```text
Assumption A1 (Regularity).
...
Assumption A2 (Growth condition).
...
Assumption A3 (Identifiability).
...
```

每个假设最好说明：

```text
1. 它用于证明哪一步；
2. 它是否标准；
3. 它是否可弱化；
4. 它是否必要；
5. 是否有例子满足它；
6. 是否有反例说明缺失时结论失败。
```

### 常见错误

- 预备知识太多，像教材；
- 关键符号没有定义；
- 定义和假设混在正文中；
- 假设过强但不解释；
- 使用已有定理但不说明版本；
- 不说明常数依赖；
- 后文证明调用了未引入工具；
- 定义后没有例子，读者无法形成直觉。

### Agent 质量判断

```text
1. Preliminaries 是否只包含必要内容？
2. 符号是否统一？
3. 定义是否精确？
4. 假设是否集中、编号、解释？
5. 已知定理是否准确引用？
6. 是否为主结果和证明做好铺垫？
```

## 5.6 Problem Setup / Model 问题设定与模型

### 写作目标

Problem Setup 将研究对象从自然语言问题转化为可证明的形式问题。

它应回答：

```text
研究对象是什么？
在哪个空间或模型中？
需要证明什么性质？
参数、变量和约束是什么？
哪些对象是给定的，哪些对象是待构造或待证明存在的？
```

### 应包含内容

- 对象类别；
- 空间或结构；
- 映射、算子、函数或关系；
- 参数范围；
- 约束条件；
- 目标性质；
- 问题陈述；
- 与已有模型的差异；
- 典型例子。

### 问题设定模板

```text
Let X be ...
For each parameter θ ∈ Θ, consider ...
The central question is whether ...
More precisely, we ask:
(Q1) ...
(Q2) ...
```

### 模型假设写法

理论模型中的假设应避免“看起来合理但无法检验”。应说明其角色。

弱写法：

```text
我们假设函数足够光滑。
```

强写法：

```text
我们假设 f 为二阶连续可微，并且 Hessian 满足 Lipschitz 条件。该条件用于控制 Taylor 展开中的余项；如果只假设一阶光滑，则 Lemma 3.2 中的误差界不再成立。
```

### 常见错误

- 模型设定分散在多处；
- 变量和参数混淆；
- 没有说明对象属于哪个空间；
- 假设隐藏在证明中；
- 不说明问题与已有模型的区别；
- 模型过于抽象但没有例子；
- 模型过于具体导致结果意义有限。

### Agent 质量判断

```text
1. 研究对象是否形式化？
2. 所有变量和参数是否定义？
3. 假设是否清楚？
4. 问题是否可以被定理回答？
5. 是否提供典型例子帮助理解？
6. 模型与已有设定差异是否说明？
```

## 5.7 Main Results 主结果

### 写作目标

Main Results 是理论论文的核心。定理、命题、引理和推论必须精确、可读、有层次。

### 定理、命题、引理、推论的区别

| 类型 | 功能 | 写作重点 |
|---|---|---|
| Theorem 定理 | 论文主要理论贡献 | 条件、结论、意义清楚 |
| Proposition 命题 | 较局部但独立的重要结果 | 支撑主线 |
| Lemma 引理 | 证明工具或中间步骤 | 说明用途 |
| Corollary 推论 | 主结果的直接后果 | 展示应用或特例 |
| Definition 定义 | 引入对象或概念 | 精确无歧义 |
| Example 例子 | 展示适用性或直觉 | 服务理解 |
| Counterexample 反例 | 展示边界或假设必要性 | 服务条件解释 |
| Conjecture 猜想 | 尚未证明的可能方向 | 明确非定理 |

### 主结果陈述原则

好的定理陈述应包含：

```text
1. 对象；
2. 假设；
3. 参数范围；
4. 结论；
5. 常数依赖；
6. 是否最优；
7. 是否存在 / 唯一 / 构造；
8. 是否概率性陈述；
9. 是否 asymptotic；
10. 与已有结果差异。
```

### 定理陈述模板

```text
Theorem 3.1 (Main theorem).
Let ... be ...
Assume A1–A3.
Then, for every ..., there exists ...
such that ...
Moreover, the bound is sharp in the sense that ...
```

### 定理后的解释

定理之后不应立刻进入证明。应解释：

```text
1. 定理说了什么；
2. 为什么条件合理；
3. 与已有结果相比新在哪里；
4. 结论是否最优；
5. 哪个例子说明适用范围；
6. 证明将在何处给出。
```

示例：

```text
Theorem 3.1 extends the classical result of X from compact spaces to locally compact spaces satisfying A2. The dependence on ε is optimal, as shown by the family of examples in Section 5. The main difficulty is that compactness can no longer be used to extract a convergent subsequence; our proof replaces it with the tightness estimate in Lemma 4.2.
```

### 常见错误

- 定理条件太长且难读；
- 结论不够精确；
- 未说明常数依赖；
- 定理之间关系不清；
- 主结果和技术引理混在一起；
- 定理后没有解释；
- 不说明结果是否加强已有结果；
- 用“显然推广”但没有比较条件强弱；
- 定理陈述中隐藏定义。

### Agent 质量判断

```text
1. 主结果是否一眼可定位？
2. 条件和结论是否清楚分开？
3. 是否说明常数、参数和范围？
4. 是否解释定理意义？
5. 是否说明与已有结果关系？
6. 是否避免把技术细节塞进主结果陈述？
```

## 5.8 Proof Sketch / Intuition 证明概览与直觉

### 写作目标

Proof Sketch 的作用不是替代正式证明，而是帮助读者理解证明路线。

理论论文应避免直接把读者扔进长证明。尤其是主证明复杂时，应先说明：

```text
证明分几步？
关键障碍是什么？
核心新想法是什么？
哪些 lemma 解决哪些障碍？
为什么该策略能工作？
```

### 推荐结构

```text
1. 证明目标；
2. 主要困难；
3. 关键构造或归约；
4. 主要引理及其作用；
5. 如何组合引理得到主结果；
6. 与已有证明的不同。
```

### Proof Sketch 示例骨架

```text
The proof has three steps. First, we reduce the general case to a normalized setting by rescaling the operator. Second, we establish a stability estimate that controls the error uniformly over the parameter space. This is the main technical step and relies on a new decomposition lemma. Finally, we combine the stability estimate with a compactness argument to pass to the limit. The key difference from previous proofs is that we do not require global boundedness; instead, local growth is controlled through the Lyapunov function introduced in Lemma 4.1.
```

### 证明直觉写法

直觉应解释“为什么结果应该成立”。

```text
Intuitively, condition A prevents mass from escaping to infinity, while condition B rules out local oscillations. Together they provide the two ingredients that compactness supplied in earlier proofs.
```

### 常见错误

- 没有 proof sketch，直接进入技术证明；
- proof sketch 只是重复 theorem；
- proof sketch 太长，变成半个证明；
- 不说明关键困难；
- 不说明新技术；
- 不说明各 lemma 的作用；
- 证明概览与正式证明结构不一致。

### Agent 质量判断

```text
1. 是否说明证明整体路线？
2. 是否指出主要困难？
3. 是否解释关键新想法？
4. 是否说明 lemma 的作用？
5. 是否帮助读者理解正式证明？
6. 是否避免重复完整证明？
```

## 5.9 Proofs 正式证明

### 写作目标

Proofs 部分需要完整、严密、可检查。  
但严密不等于堆砌技术。好的证明应有清晰结构、局部目标和逻辑路线。

### 证明组织原则

```text
1. 先给路线，再给细节；
2. 长证明分步骤；
3. 每一步说明目标；
4. 关键中间结论单独命名为 lemma 或 claim；
5. 避免隐藏假设；
6. 处理边界情况；
7. 明确调用已有结果；
8. 结尾回到定理结论。
```

### 长证明结构模板

```text
Proof of Theorem 3.1.
We divide the proof into four steps.

Step 1: Normalization.
...

Step 2: Establishing the key estimate.
...

Step 3: Passing to the limit.
...

Step 4: Verifying the sharpness.
...

Combining Steps 1–4 yields the desired conclusion.
```

### 引理使用原则

好的引理应：

```text
1. 有明确功能；
2. 不只是任意中间计算；
3. 能减少主证明复杂度；
4. 条件和主定理一致或更清楚；
5. 证明位置合理；
6. 在使用前说明用途。
```

引理前可写：

```text
The following lemma is the technical core of the proof. It replaces the compactness argument used in earlier work.
```

### 常见证明技术及写作关注点

| 技术 | 写作重点 |
|---|---|
| 归纳法 | 明确基例、归纳假设、归纳步 |
| 反证法 | 明确否定命题和矛盾来源 |
| 构造法 | 明确构造对象及其满足条件 |
| 紧性 / 极限 | 明确收敛类型和极限保性质 |
| 对偶性 | 明确 primal、dual 和强对偶条件 |
| 归约 | 明确源问题、目标问题和保持性质 |
| 不变量 | 明确不变量定义和保持机制 |
| 概率界 | 明确事件、概率空间、常数和 union bound |
| 耦合 | 明确联合构造和边缘分布 |
| 能量估计 | 明确能量函数、下降性质和误差项 |
| 谱分析 | 明确谱条件、算子域和正则性 |
| 变分法 | 明确泛函、紧性、下半连续性 |
| 对角化 | 明确序列选择和极限控制 |
| 形式系统证明 | 明确语法、语义、推导规则、soundness / completeness |

### 常见错误

- 使用“显然”掩盖关键步骤；
- 长证明没有结构；
- lemma 条件与主定理不一致；
- 调用已有定理但条件未验证；
- 常数依赖不清；
- 边界情况未处理；
- 变量作用域混乱；
- 证明结尾没有明确回到定理；
- 技术计算正确但读者不知道为什么这样做；
- 将重要证明放入附录但正文没有 proof sketch。

### Agent 质量判断

```text
1. 证明是否分步骤？
2. 每一步目标是否清楚？
3. 是否存在逻辑跳跃？
4. 调用定理的条件是否验证？
5. 符号和变量是否一致？
6. 是否处理边界情况？
7. 是否说明关键 lemma 的意义？
8. 是否能从证明末尾回到主定理结论？
```

## 5.10 Examples / Counterexamples 例子与反例

### 写作目标

理论论文中的例子和反例不是装饰，而是帮助读者理解结果的适用范围、假设必要性和结论紧性。

### 例子的功能

| 类型 | 功能 |
|---|---|
| 基本例子 | 帮助理解定义 |
| 非平凡例子 | 说明定理适用范围不空 |
| 特例 | 连接经典结果 |
| 极端例子 | 展示边界行为 |
| 反例 | 说明假设必要 |
| 构造例 | 证明存在性或紧性 |
| 数值例 | 辅助直觉，不替代证明 |
| 应用例 | 展示理论如何使用 |

### 好例子写法

```text
Example 4.1 shows that Assumption A2 is satisfied by a broad class of non-compact spaces, including ...
This demonstrates that Theorem 3.1 is not merely a compact reformulation of the classical theorem.
```

### 好反例写法

```text
The next example shows that Assumption A2 cannot be removed. We construct a sequence satisfying A1 but not A2 for which the conclusion of Theorem 3.1 fails.
```

### 反例应说明

```text
1. 反例违反哪个假设；
2. 其他假设是否仍满足；
3. 结论如何失败；
4. 这说明主定理边界在哪里。
```

### 紧性说明

如果论文声称 bound 是 tight / optimal，需要说明：

```text
1. 上界是什么；
2. 下界或构造是什么；
3. 两者是否同阶；
4. 常数是否最优；
5. optimality 是在什么类中成立。
```

### 常见错误

- 定义没有例子；
- 反例不清楚违反了哪个条件；
- 例子与主定理无关；
- 特例没有说明如何退化到经典结果；
- 声称 tight 但只给直觉；
- 数值例子被写成证明；
- 例子太复杂，反而阻碍理解。

### Agent 质量判断

```text
1. 是否有例子帮助理解新定义？
2. 是否有例子说明主定理适用范围？
3. 是否有反例说明假设必要？
4. 是否有特例连接已有结果？
5. 是否清楚说明边界和 tightness？
```

## 5.11 Applications / Implications 应用与含义

### 写作目标

Applications / Implications 用于说明理论结果如何被使用，或它对已有理论、模型、算法、经济机制、物理解释等有什么影响。

理论论文不一定必须有应用，但应说明结果的理论意义。

### 常见应用形式

| 类型 | 示例 |
|---|---|
| 推出经典结果 | 旧定理作为新定理特例 |
| 解决开放问题 | 回答已有文献提出的问题 |
| 改进界 | 收紧 rate、constant、sample complexity |
| 解释现象 | 给出模型行为的理论解释 |
| 指导算法 | 说明算法何时收敛或失败 |
| 证明限制 | 给出 lower bound 或 impossibility |
| 统一框架 | 将多个定理纳入同一模型 |
| 预测边界 | 说明某性质在何种条件下不可能 |

### 应用写法

弱写法：

```text
本结果有很多应用。
```

强写法：

```text
As a first application, Theorem 3.1 recovers the classical compact-space result by setting γ = 0. More importantly, it also applies to locally compact spaces with polynomial growth, which were not covered by previous theorems.
```

### 理论含义写法

```text
The lower bound indicates that no algorithm using only first-order information can achieve a rate faster than ... under Assumption A. Thus, the upper bound in Theorem 2 is optimal up to logarithmic factors.
```

### 常见错误

- 只说“有重要理论意义”；
- 应用与主结果关系不清；
- 推论条件没有验证；
- 把 speculative implication 写成 theorem；
- 没有说明特例如何对应已有结果；
- 不区分已证明结果和可能应用。

### Agent 质量判断

```text
1. 应用是否由主定理严格推出？
2. 是否说明如何验证应用条件？
3. 是否展示相对于已有结果的改进？
4. 是否避免把推测写成结论？
5. 是否说明理论意义具体在哪里？
```

---

## 5.12 Discussion / Limitations 讨论与局限

### 写作目标

理论论文的 Discussion 应讨论：

```text
假设是否强？
结论是否最优？
证明技术是否可迁移？
结果未覆盖什么？
还有哪些开放问题？
```

### 应包含内容

- 假设强弱；
- 结论是否 tight；
- 证明方法的局限；
- 与已有结果差异；
- 反例说明的边界；
- 未解决的问题；
- 可能推广方向；
- conjecture；
- 对相关领域的启示。

### 好的局限写法

弱写法：

```text
本文结果还可以进一步推广。
```

强写法：

```text
本文证明依赖 Assumption A2 中的局部增长条件。反例 5.1 表明完全去除该条件时结论失败，但尚不清楚是否可以将 A2 弱化为平均增长条件。这构成一个自然的后续问题。
```

### conjecture 写法

如果提出猜想，应明确其地位。

```text
Based on the counterexample in Section 5 and the proof of Theorem 3.1, we conjecture that Assumption A2 can be replaced by the weaker condition A2'. We do not prove this statement here.
```

### 常见错误

- 不讨论假设是否必要；
- 不说明证明方法局限；
- 未来工作泛泛；
- 将猜想写得像已证明；
- 不承认结果只覆盖特殊情形；
- 不解释反例的意义。

### Agent 质量判断

```text
1. 是否讨论假设强弱？
2. 是否说明结果边界？
3. 是否处理反例和 tightness？
4. 是否提出合理开放问题？
5. 是否区分 theorem、conjecture 和 speculation？
```

## 6. 常见贡献类型

理论论文的贡献形式多样。Agent 应帮助用户明确贡献类型，而不是只写“本文证明了新结果”。

### 6.1 推广 Generalization

将已有结果从特殊情形推广到更一般情形。

示例：

```text
本文将经典 X 定理从有限维空间推广到可分 Banach 空间。
```

质量要求：

- 说明原结果条件；
- 说明新结果条件；
- 比较条件强弱；
- 证明原结果是特例；
- 说明推广为何非平凡。

### 6.2 收紧界 Sharpening Bounds

改进上界、下界、收敛率、样本复杂度或常数。

示例：

```text
本文将已有 O(n^{-1/2}) 的误差界改进为 O(n^{-1})，并证明该速率在给定模型类中最优。
```

质量要求：

- 说明旧界；
- 说明新界；
- 说明是否 tight；
- 说明改进来自什么技术；
- 明确常数和参数依赖。

### 6.3 建立等价 Equivalence

证明两个看似不同的性质、模型、问题或定义等价。

示例：

```text
本文证明稳定性条件 A 与可表示性条件 B 在假设 C 下等价。
```

质量要求：

- 双向证明；
- 明确等价条件；
- 说明等价带来的理解或应用；
- 给出非等价边界或反例。

### 6.4 存在性 Existence

证明满足某性质的对象存在。

示例：

```text
本文证明在弱正则条件下均衡解存在。
```

质量要求：

- 明确对象类别；
- 明确存在条件；
- 说明是否构造性；
- 讨论唯一性或非唯一性；
- 给出例子。

### 6.5 唯一性 Uniqueness

证明满足条件的对象唯一。

质量要求：

- 明确唯一性在何种意义下成立；
- 处理等价类或规范化；
- 说明缺少条件时是否非唯一。

### 6.6 构造 Construction

显式构造对象、反例、编码、图、模型或解。

质量要求：

- 构造步骤清楚；
- 验证构造满足所有条件；
- 说明构造的意义；
- 说明是否最优或极端。

### 6.7 不可能性 / 下界 Impossibility / Lower Bound

证明某目标无法实现，或任何方法都不能超过某界。

示例：

```text
本文证明在只允许一阶查询的模型下，不存在收敛率优于 O(1/T) 的算法。
```

质量要求：

- 明确模型限制；
- 明确对所有方法成立；
- 说明 lower bound 与 upper bound 的关系；
- 避免只证明某类方法失败却声称所有方法不可能。

### 6.8 分类 Classification

对一类对象给出完整分类或结构定理。

质量要求：

- 分类标准清楚；
- 覆盖所有对象；
- 不重不漏；
- 提供典型例子；
- 说明分类意义。

### 6.9 新证明 New Proof

给出已有结果的新证明。

质量要求：

- 说明为什么新证明有价值；
- 是否更短、更概念化、更一般化、更可迁移；
- 是否产生新推论；
- 避免只是技术重写。

### 6.10 理论解释 Theoretical Explanation

解释某算法、模型或现象为何出现。

质量要求：

- 模型抽象合理；
- 结论与现象对应；
- 假设清楚；
- 不把理论模型过度推广到实际系统。

### 6.11 统一框架 Unified Framework

将多个已有结果纳入一个更一般的理论框架。

质量要求：

- 统一不是简单并列；
- 说明共同结构；
- 证明旧结果如何作为特例；
- 新框架应产生新结果或新理解。

## 7. 证据与论证标准

## 7.1 强理论证据

理论论文中的强证据包括：

```text
1. 定义清楚；
2. 假设明确；
3. 定理陈述精确；
4. 证明完整；
5. 调用已有结果时条件全部验证；
6. 例子说明定理非空；
7. 反例说明假设必要或边界；
8. 与已有结果比较充分；
9. 结论不超过定理范围；
10. 证明直觉帮助理解。
```

### 强证据组合示例

| 主张 | 强证据组合 |
|---|---|
| 结果推广已有定理 | 新定理 + 旧定理作为特例 + 假设比较 |
| 界是最优的 | 上界证明 + 匹配下界 / 构造 |
| 假设必要 | 主定理 + 去掉假设后的反例 |
| 新定义有用 | 定义 + 非平凡例子 + 主定理使用 |
| 证明技术可迁移 | 技术 lemma + 多个应用 / 推论 |
| 不可能性结果 | 明确模型 + 对任意方法的 lower bound |

## 7.2 弱理论证据

弱证据常表现为：

- 定义模糊；
- 假设隐藏；
- 定理条件不清；
- 证明跳步；
- 只证明特殊情形却声称一般结论；
- 没有处理边界情况；
- 没有例子说明结果非空；
- 没有反例说明假设必要；
- 与已有结果关系不清；
- 只说“容易推广”但不证明；
- 数值例子替代理论证明；
- conjecture 写得像 theorem。

## 7.3 理论论证强度判断

Agent 可使用以下表格：

| 问题 | 诊断问题 | 修改方式 |
|---|---|---|
| 假设过强 | 是否比已有结果还强？ | 解释必要性或弱化假设 |
| 贡献不清 | 相对哪个定理新？ | 加入比较段 |
| 证明跳步 | 哪一步无法验证？ | 增加 lemma 或详细推导 |
| 定理过宽 | 是否所有条件都证明了？ | 限定陈述 |
| 常数不清 | 常数依赖哪些参数？ | 明确写出 |
| 结果无直觉 | 读者为何相信？ | 增加 proof sketch |
| 边界不明 | 假设能否去掉？ | 增加例子 / 反例 |
| 应用空泛 | 如何从定理推出？ | 写 corollary |

## 7.4 结论强度与证明范围匹配

| 证明结果 | 可写 | 不应写 |
|---|---|---|
| 在假设 A 下成立 | 在假设 A 下…… | 一般情况下…… |
| 存在性 | 存在至少一个…… | 可以构造有效算法，除非已证明 |
| 非构造存在性 | 存在…… | 给出可计算方法 |
| 某类方法下界 | 对该类方法不可能…… | 对所有方法不可能…… |
| 渐近界 | 当 n → ∞ 时…… | 对所有有限 n 均精确…… |
| 一阶近似 | 局部近似成立 | 全局结论成立 |
| 特例证明 | 在该特例中…… | 该理论普遍成立 |

## 8. 文献综述写法

### 8.1 理论文献综述的功能

理论论文的文献综述应建立“结果地图”。

它不是“谁研究过什么”，而是说明：

```text
已有定理之间是什么关系？
它们依赖哪些假设？
它们解决了哪些情形？
哪些情形仍未解决？
本文结果如何改变这张地图？
```

### 8.2 文献综述的组织方式

#### 方式一：按定理强弱组织

```text
最早结果：条件强，结论基本
后续结果：条件弱化
当前结果：仍缺少某情形
本文结果：填补该情形
```

#### 方式二：按假设组织

```text
有界情形
紧情形
光滑情形
非光滑情形
本文弱化条件
```

#### 方式三：按证明技术组织

```text
紧性方法
对偶方法
概率方法
构造方法
本文新技术
```

#### 方式四：按 open problem 组织

```text
已解决的问题
仍开放的问题
本文解决其中哪一部分
剩余问题
```

### 8.3 文献综述段落模板

```text
已有研究在 A 条件下证明了 B 结论。随后工作将 A 弱化为 A'，但仍依赖 C 条件。与这些结果不同，本文在不假设 C 的情况下证明 B 的一个弱化 / 强化版本。该结果说明 C 并非获得 B 所必需，但反例 4.2 表明完全去除 A' 仍不可能。
```

### 8.4 常见错误

- 文献只按时间排序；
- 不比较条件和结论；
- 不说明已有定理是否包含本文结果；
- 对已有结果描述不精确；
- 不说明 open problem；
- 不讨论最接近工作；
- 将技术差异写得过于模糊；
- 夸大本文相对已有工作的差异。

### Agent 质量判断

```text
1. 文献综述是否形成结果地图？
2. 是否比较假设和结论？
3. 是否说明本文结果的精确位置？
4. 是否处理最接近结果？
5. 是否避免按作者罗列？
6. 是否准确而不夸大？
```

## 9. 结果、分析或论证主体的组织方式

理论论文主体通常由 Main Results、Proofs、Examples 和 Applications 组成。其组织原则是：

```text
先让读者知道要证明什么；
再让读者理解为什么可能成立；
然后给出证明；
最后说明结果边界和意义。
```

### 9.1 推荐主体结构

```text
3. Main Results
   3.1 Main theorem
   3.2 Consequences
   3.3 Comparison with known results

4. Proof Overview
   4.1 Key difficulty
   4.2 Main technical lemma
   4.3 How lemmas combine

5. Proofs
   5.1 Proof of technical lemma
   5.2 Proof of main theorem
   5.3 Proof of corollaries

6. Examples and Counterexamples
   6.1 Examples satisfying assumptions
   6.2 Necessity of assumptions
   6.3 Tightness of bounds

7. Applications
```

### 9.2 定理之间的层级

理论论文应避免把所有结果平铺。可按以下层级组织：

```text
Main Theorem：全文核心结果；
Supporting Lemmas：证明工具；
Corollaries：主结果直接后果；
Examples：帮助理解；
Counterexamples：边界说明；
Extensions：在额外条件下的推广；
Applications：结果使用。
```

### 9.3 证明主体的叙事

证明叙事应遵循：

```text
目标
→ 难点
→ 技术工具
→ 关键步骤
→ 组合
→ 结论
```

弱写法：

```text
下面证明定理。由公式……得到……，再由……得……
```

强写法：

```text
The main obstacle is the lack of compactness. We first establish a uniform tightness estimate, which allows us to recover enough compactness to pass to the limit. The proof then proceeds by applying this estimate to the normalized sequence constructed in Step 1.
```

### 9.4 附录的使用

附录适合放置：

- 过长技术计算；
- 标准但繁琐的引理；
- 辅助证明；
- 扩展结果；
- 额外例子；
- 背景定理；
- 形式化细节。

但主文必须包含：

```text
1. 主结果陈述；
2. 证明路线；
3. 关键 lemma 的作用；
4. 结果意义。
```

不能把理解论文所必需的全部证明逻辑放入附录而正文只留定理。

### 9.5 常见主体失败

| 失败模式 | 表现 | 修正 |
|---|---|---|
| 定理堆砌 | 一连串 theorem，无解释 | 增加结果地图 |
| 技术先行 | 先给复杂 lemma，不知用途 | 先写 proof overview |
| 主次不清 | 主定理和小 lemma 同等呈现 | 分层组织 |
| 证明孤立 | 不解释为什么这么证明 | 增加证明直觉 |
| 无边界 | 没有例子和反例 | 增加 boundary section |
| 附录依赖过强 | 正文无法理解证明 | 主文补 proof sketch |

## 10. 讨论与结论写法

## 10.1 Discussion 的功能

理论论文 Discussion 应从形式结果上升到理论意义：

```text
结果解决了什么缺口？
假设为何合理或仍然强？
证明技术带来什么新可能？
哪些边界由反例说明？
哪些开放问题仍然存在？
```

### 推荐结构

```text
1. 回顾主结果；
2. 解释其相对已有理论的意义；
3. 讨论假设和 tightness；
4. 讨论证明技术的可迁移性；
5. 说明应用或推论；
6. 提出开放问题。
```

### 10.2 从定理到意义的提升

定理层：

```text
Theorem 3.1 proves that property P holds under assumptions A1–A3.
```

意义层：

```text
This shows that global compactness is not essential for P; a local growth condition is sufficient.
```

理论地图层：

```text
Together with the counterexample in Section 5, the result places the boundary of P between local growth and arbitrary non-compactness.
```

### 10.3 结论写法

Conclusion 应压缩：

```text
研究问题
→ 主结果
→ 技术贡献
→ 理论意义
→ 边界和未来方向
```

示例：

```text
本文研究了在弱紧性条件下性质 P 是否仍然成立的问题。我们证明，在局部增长条件和正则性假设下，P 仍然成立，并给出最优参数依赖。证明的核心是一个新的能量分解引理，它替代了经典证明中的全局紧性论证。反例表明局部增长条件在一般情形下不可完全去除。未来工作可进一步研究该条件是否能弱化为平均增长条件。
```

### 常见错误

- Conclusion 只重复 theorem；
- 不说明理论意义；
- 不提边界；
- 不提证明技术贡献；
- 把猜想写成结论；
- 过度声称“解决了整个问题”；
- 忽略反例和限制。

## 11. 审稿人关注点

理论论文审稿人通常会重点质疑以下问题。

### 11.1 问题是否重要

审稿人会问：

```text
1. 这个问题为什么值得证明？
2. 它是否连接已有理论缺口？
3. 是否解决了已知 open problem 或重要特例？
4. 结果是否只是技术变形？
5. 读者能否理解其意义？
```

### 11.2 结果是否已有

审稿人会问：

```text
1. 是否已有文献证明了相同或更强结果？
2. 本文是否只是已有定理的直接推论？
3. 作者是否准确比较了最近结果？
4. 是否遗漏了最接近工作？
```

### 11.3 假设是否过强

审稿人会问：

```text
1. 假设是否比已有结果还强？
2. 假设是否自然？
3. 是否有例子满足假设？
4. 假设是否为了证明方便而人为设置？
5. 是否有反例说明假设必要？
```

### 11.4 定理是否清楚

审稿人会问：

```text
1. 条件和结论是否精确？
2. 常数依赖是否明确？
3. 定理是否有隐藏条件？
4. 结论是否与标题和摘要一致？
5. 是否区分 theorem、corollary、conjecture？
```

### 11.5 证明是否完整

审稿人会问：

```text
1. 是否存在逻辑跳跃？
2. 是否调用了不适用的已知定理？
3. 边界情况是否处理？
4. 是否验证所有假设？
5. 变量和符号是否一致？
6. 技术引理是否正确？
```

### 11.6 贡献是否足够

审稿人会问：

```text
1. 结果是否足够新？
2. 是否只是小幅推广？
3. 推广是否有意义？
4. 新证明是否带来新理解？
5. 反例或下界是否足够强？
```

### 11.7 写作是否可读

审稿人会问：

```text
1. 是否有证明路线？
2. 是否有直觉解释？
3. 是否有例子帮助理解？
4. 符号是否过重？
5. 读者能否看出主线？
```

## 12. 常见失败模式

## 12.1 证明正确但贡献不清楚

表现：

- 证明很多；
- 定理成立；
- 但不知道新在哪里。

修正：

```text
1. 增加已有结果比较表；
2. 明确本文放宽了哪个假设或加强了哪个结论；
3. 在 Introduction 中解释理论缺口；
4. 在定理后说明结果意义；
5. 用例子展示新结果覆盖的情形。
```

## 12.2 定理陈述过于技术化

表现：

- 定理条件过长；
- 读者读不出核心结论；
- 主结果被 technical assumptions 淹没。

修正：

```text
1. 将假设编号；
2. 在定理前解释假设；
3. 定理后用自然语言解释；
4. 将次要技术条件放入 Assumption 部分；
5. 增加 simplified version。
```

## 12.3 引言只列定理

表现：

```text
本文的主要结果是 Theorem 1.1、Theorem 1.2、Theorem 1.3。
```

问题：

- 没有问题动机；
- 没有理论地图；
- 没有直觉。

修正：

```text
1. 先解释问题和已有缺口；
2. 再用非技术语言说明主结果；
3. 最后列形式贡献。
```

## 12.4 假设看起来人为

表现：

- 假设复杂；
- 没有解释；
- 不知道是否有对象满足。

修正：

```text
1. 给每个假设命名；
2. 解释用途；
3. 给例子；
4. 给反例或必要性说明；
5. 讨论能否弱化。
```

## 12.5 证明路线不清

表现：

- 证明很长；
- 公式连续出现；
- 不知道每步目的。

修正：

```text
1. 增加 proof sketch；
2. 将证明分步骤；
3. 给 lemma 命名；
4. 在每步开头说明目标；
5. 在关键步骤后解释意义。
```

## 12.6 过度使用“显然”

表现：

```text
显然可得……
不难证明……
直接推出……
```

修正：

```text
1. 判断是否真的显然；
2. 若非显然，补充中间推理；
3. 若常用结果，明确引用；
4. 若需要新工具，增加 lemma。
```

## 12.7 没有例子或反例

表现：

- 定义抽象；
- 假设复杂；
- 没有展示适用对象；
- 不知道条件是否必要。

修正：

```text
1. 增加基本例子；
2. 增加非平凡例子；
3. 增加去掉关键假设后的反例；
4. 增加经典定理作为特例。
```

## 12.8 结论超出证明

表现：

- 只证明特定模型，却声称一般理论；
- 只证明渐近结果，却声称有限样本；
- 只证明某类算法下界，却声称所有算法不可能。

修正：

```text
1. 限定结论范围；
2. 明确模型和假设；
3. 区分 theorem、conjecture 和 implication；
4. 增加 Limitations。
```

## 13. Agent 写作辅助策略

## 13.1 Agent 的总体任务

Agent 辅助理论论文时，应同时检查：

```text
1. 形式正确性；
2. 贡献清晰度；
3. 证明可读性；
4. 理论意义；
5. 假设边界。
```

Agent 不应只润色语言，而应帮助作者回答：

```text
这个定理为什么重要？
它相对已有结果新在哪里？
读者能否理解证明路线？
假设是否合理？
是否需要例子或反例？
结论是否超过证明范围？
```

## 13.2 Agent 诊断流程

```text
1. 识别论文类型：定理、下界、构造、理论模型、证明技术、统一框架等。
2. 提取核心问题。
3. 提取主结果。
4. 比较已有结果。
5. 检查定义和假设。
6. 检查定理陈述。
7. 检查证明路线。
8. 检查证明细节是否有跳步。
9. 检查例子、反例和边界。
10. 检查 Introduction 是否说明意义。
11. 检查 Conclusion 是否不过度声称。
```

## 13.3 Agent 如何改写理论论文 Introduction

Agent 应将 Introduction 从“定理列表”改写为“问题—缺口—主结果—直觉—贡献”。

模板：

```text
本文研究的问题是……
该问题在……中重要，因为……
已有结果表明……，但它们依赖……
这留下了一个关键问题：……
本文证明……
直观上，结果成立是因为……
证明的主要困难是……
本文的技术贡献在于……
```

## 13.4 Agent 如何改写主结果陈述

检查表：

```text
1. 定理对象是否明确？
2. 假设是否集中编号？
3. 结论是否精确？
4. 常数依赖是否清楚？
5. 是否需要 simplified theorem？
6. 是否需要定理后解释？
7. 是否需要比较已有结果？
```

Agent 可建议：

```text
在正式 Theorem 前增加 Informal Theorem。
```

示例：

```text
Informal Theorem.
Under a local growth condition, the compactness assumption in the classical theorem can be replaced while preserving the same convergence rate.
```

然后再给正式定理。

## 13.5 Agent 如何改写证明叙事

Agent 应把长证明拆成：

```text
Proof overview
Step 1: ...
Step 2: ...
Step 3: ...
Conclusion
```

每个步骤开头补：

```text
The goal of this step is ...
```

对关键 lemma 补：

```text
This lemma is needed because ...
```

对技术计算后补：

```text
This estimate is the point where Assumption A2 is used.
```

## 13.6 Agent 如何检查假设

Agent 应建立假设表：

| 假设 | 内容 | 用于证明哪一步 | 是否标准 | 是否有例子 | 是否必要 |
|---|---|---|---|---|---|

若某假设没有用途，应提醒：

```text
该假设可能冗余，或需要在证明中明确使用位置。
```

若某假设很强，应提醒：

```text
需要解释该假设为何自然，或提供满足该假设的非平凡例子。
```

## 13.7 Agent 如何检查贡献

Agent 应要求作者明确填写：

```text
本文相对已有结果：
- 放宽了哪个假设？
- 加强了哪个结论？
- 收紧了哪个界？
- 证明了哪个下界？
- 给出了哪个反例？
- 统一了哪些结果？
- 新证明方法能否用于其他问题？
```

## 13.8 Agent 如何检查证明跳步

高风险词：

```text
显然
容易得到
不难证明
直接推出
标准论证
类似可得
由常规方法
显而易见
```

Agent 应逐句检查：

```text
1. 是否需要补充推导？
2. 是否需要引用定理？
3. 是否需要验证条件？
4. 是否需要处理边界情形？
5. 是否应改成 lemma？
```

## 13.9 Agent 如何降级过度声称

| 过强表达 | 更稳妥表达 |
|---|---|
| 本文解决了该问题 | 本文在假设 A–C 下解决了该问题 |
| 本文给出最优界 | 本文给出与下界同阶的界 |
| 该结论普遍成立 | 该结论在本文模型中成立 |
| 该假设是必要的 | 反例表明在一般情形下不能完全去除该假设 |
| 本文理论解释了实际现象 | 本文模型为该现象提供了一种理论解释 |

## 14. 检查清单

## 14.1 全文总检清单

```markdown
## 理论问题
- [ ] 论文是否明确研究什么理论问题？
- [ ] 问题是否相对已有理论有意义？
- [ ] Introduction 是否解释为什么重要？
- [ ] 是否说明已有结果的限制？
- [ ] 是否清楚表述本文贡献？

## 定义与假设
- [ ] 所有核心对象是否定义？
- [ ] 符号是否统一？
- [ ] 假设是否集中编号？
- [ ] 每个假设是否解释用途？
- [ ] 是否有例子满足假设？
- [ ] 是否有反例说明关键假设必要？

## 主结果
- [ ] 主定理是否容易定位？
- [ ] 定理条件是否清楚？
- [ ] 定理结论是否精确？
- [ ] 常数和参数依赖是否说明？
- [ ] 是否说明与已有结果关系？
- [ ] 是否说明结果是否 tight / optimal？

## 证明
- [ ] 是否有 proof sketch？
- [ ] 长证明是否分步骤？
- [ ] 每步目标是否清楚？
- [ ] 是否存在“显然”式跳步？
- [ ] 调用已有定理时是否验证条件？
- [ ] 是否处理边界情况？
- [ ] 引理是否有明确功能？

## 例子与反例
- [ ] 是否有例子帮助理解定义？
- [ ] 是否有非平凡例子说明适用范围？
- [ ] 是否有反例说明假设边界？
- [ ] 是否有特例连接经典结果？
- [ ] 是否说明 tightness？

## 理论意义
- [ ] 是否说明结果推广、收紧、统一或反驳了什么？
- [ ] 是否说明证明技术贡献？
- [ ] 是否说明应用或推论？
- [ ] 是否提出合理开放问题？
- [ ] Conclusion 是否避免过度声称？
```
