# 工程 / 系统 / 设计型论文写作指南

## 1. 适用范围

本指南适用于以**工程问题、系统设计、技术实现、方案优化和系统评价**为核心的科研论文。其目标不是单纯报告“做了一个系统”，而是说明：

> 面对一个真实且重要的问题，作者如何在明确约束下做出设计选择，并通过评价证据证明这些选择确实有效。

适用对象包括但不限于：

- 工程技术论文；
- 机器人系统论文；
- HCI / 人机交互系统论文；
- 软件系统论文；
- 网络系统论文；
- 芯片与硬件系统论文；
- 控制系统论文；
- 能源系统论文；
- 智能制造系统论文；
- 仪器装置与测量系统论文；
- 嵌入式系统论文；
- 工程方法与应用系统论文；
- 平台、工具链、框架、协议、架构、装置、算法系统结合型论文。

这类论文通常围绕以下对象展开：

- 一个系统；
- 一个工程架构；
- 一个装置；
- 一个工具；
- 一个协议；
- 一个平台；
- 一个控制策略；
- 一个设计方法；
- 一个软硬件协同方案；
- 一个工作流；
- 一个面向真实场景的应用方案。

本指南特别适用于以下写作任务：

- 从项目报告中提炼学术论文；
- 将系统实现转化为可发表的工程贡献；
- 诊断工程论文的设计逻辑；
- 重构系统架构与评价章节；
- 检查 baseline、消融实验、压力测试是否充分；
- 判断系统论文的 novelty 是否成立；
- 防止论文写成产品说明书、开发日志或功能清单。

工程 / 系统论文的核心是：

> 设计选择与评价证据之间的闭环。

也就是说，论文必须清楚回答：

```text
真实问题是什么？
为什么已有方案不够？
系统必须满足哪些需求和约束？
作者为什么这样设计？
系统如何实现？
评价如何证明这些设计选择有效？
结果如何回到最初的问题和设计目标？
```

## 2. 不适用范围

本指南不应机械套用于以下文本或论文类型。

### 2.1 纯自然科学实验论文

自然科学实验论文通常通过实验、观测和测量解释自然现象，例如材料结构、化学反应、生物机制或环境过程。其核心是“自然现象解释”。

工程 / 系统论文虽然也做实验，但实验的目的通常是评价系统是否满足设计目标，例如：

- 系统是否更快；
- 是否更稳定；
- 是否更易用；
- 是否能部署；
- 是否在真实约束下工作；
- 是否优于已有方案。

### 2.2 纯理论或形式化证明论文

如果论文主要依靠定理、证明、模型推导、复杂度分析或形式化验证，而没有具体系统实现与系统评价，则应使用理论 / 方法型论文指南。

### 2.3 纯算法论文

如果论文主要贡献是一个算法本身，并通过标准数据集和指标评价算法性能，而没有明显系统设计、架构实现、部署约束或工程集成问题，则更接近算法 / 方法论文。

但如果算法被嵌入系统架构中，并且论文重点在端到端系统、实时性、部署、可靠性、可扩展性或工程约束，则本指南适用。

### 2.4 产品说明书

产品说明书强调功能、使用方式和市场价值；工程论文强调问题、设计理据、实现细节、评价证据和学术贡献。

产品说明书常写：

```text
本系统具有 A、B、C 三大功能，界面友好，性能优越。
```

工程论文应写：

```text
为满足低延迟、高并发和边缘部署约束，本文采用 A 架构，将 B 任务下沉到 C 模块，并通过 D 机制降低通信开销。评价结果显示……
```

### 2.5 开发日志或项目总结

开发日志记录开发过程，项目总结记录完成情况。  
工程论文不是“我们做了什么”，而是“为什么这样做、解决了什么难题、证据如何证明有效”。

### 2.6 需求文档或技术方案书

需求文档和技术方案书可以说明目标、功能和模块，但通常不包含学术贡献、与已有研究比较和严格评价。

### 2.7 单纯应用案例

如果文本只是把现有工具应用到一个场景中，没有新问题、新架构、新组合、新评价或新设计洞见，则不构成强工程论文。

## 3. 知识生产逻辑

工程 / 系统 / 设计型论文的知识生产逻辑不同于“发现自然规律”，它主要通过**设计、实现和评价**生产可信知识。

其基本链条是：

```text
真实问题 / 场景痛点
→ 工程约束与需求
→ 现有方案不足
→ 设计目标
→ 设计选择与 rationale
→ 系统架构与实现
→ 评价设计
→ 实验证据
→ 设计主张成立或被限定
→ 工程 / 系统贡献
```

### 3.1 这类论文如何产生可信知识

工程系统论文的可信知识来自以下几个方面：

| 知识来源 | 论文中应体现什么 | Agent 判断重点 |
|---|---|---|
| 真实问题 | 问题来自真实场景、用户、部署、性能瓶颈或系统需求 | 问题是否真实而非作者自造 |
| 工程约束 | 延迟、成本、功耗、带宽、内存、可靠性、安全、可维护性等 | 约束是否具体 |
| 设计目标 | 系统要优化什么、保证什么、取舍什么 | 目标是否可评价 |
| 设计理据 | 为什么采用该架构、模块、策略或流程 | 是否有 rationale，而非只描述模块 |
| 系统实现 | 架构、模块、接口、数据流、控制流、部署方式 | 是否足以理解和复现 |
| 评价实验 | 指标、baseline、对比、消融、压力测试、案例分析 | 评价是否支撑主张 |
| 证据闭环 | 每个设计目标都有评价证据回应 | 是否形成闭环 |
| 边界说明 | 系统适用和不适用条件 | 是否避免过度推广 |

### 3.2 工程系统论文与自然科学实验论文的区别

| 维度 | 自然科学实验论文 | 工程 / 系统 / 设计型论文 |
|---|---|---|
| 核心问题 | 自然现象为什么发生 | 系统如何解决真实问题 |
| 知识对象 | 自然机制、规律、结构、过程 | 设计方案、架构、方法、工程取舍 |
| 证据来源 | 实验、观测、测量、表征 | 系统实现、性能评价、用户研究、部署测试 |
| 方法重点 | 控制变量、重复、误差、机制解释 | 需求、约束、架构、baseline、指标、消融 |
| 结果重点 | 数据揭示现象或机制 | 评价证明设计有效 |
| 贡献形式 | 发现、机制、测量、规律 | 新系统、新架构、新设计理据、新工程经验 |
| 审稿关注 | 数据质量、机制解释、实验设计 | 问题真实性、novelty、baseline、公平评价、可复现性 |
| 失败模式 | 数据堆砌、机制过度 | 项目报告化、产品说明书化、评价不足 |

自然科学论文常问：

```text
为什么会出现这个现象？
```

工程系统论文常问：

```text
在真实约束下，什么样的设计能更好地解决这个问题？
```

### 3.3 工程系统论文的核心闭环

工程论文必须形成以下闭环：

```text
Problem
→ Requirements
→ Design Rationale
→ System Architecture
→ Implementation
→ Evaluation
→ Discussion
→ Back to Problem
```

每个环节都应相互对应。

| 前文提出 | 后文必须回应 |
|---|---|
| 低延迟需求 | 延迟实验、端到端时延分解、压力测试 |
| 高可靠性需求 | 故障注入、异常恢复、长时间运行测试 |
| 可扩展需求 | 节点数、数据量、用户量扩展实验 |
| 易用性需求 | 用户研究、任务完成率、学习成本 |
| 低功耗需求 | 功耗测量、能效指标、硬件对比 |
| 可部署需求 | 部署场景、资源占用、维护成本 |
| 新架构主张 | 消融实验、模块替换、baseline 对比 |
| 新应用场景 | 案例研究、现场测试、用户反馈 |

如果前文声称“系统可扩展”，但 Evaluation 中没有扩展性实验，则闭环断裂。  
如果前文声称“部署友好”，但没有部署成本、资源占用或现场运行证据，则主张不足。

### 3.4 工程论文中的 Claim-Evidence-Warrant

工程论文中的主张也需要 Claim-Evidence-Warrant 结构。

```text
Claim：本文系统相较已有方案降低了端到端延迟。
Evidence：在相同硬件、相同负载、相同数据规模下，本文系统延迟比 baseline 低 35%。
Warrant：由于消融实验显示主要收益来自异步调度模块，因此性能提升可归因于该设计选择。
Limitation：该结论目前在中等负载和特定网络条件下验证，极端弱网环境仍需进一步测试。
```

Agent 检查规则：

```text
每当论文写“有效”“可靠”“可扩展”“实时”“轻量”“易用”“优于”“适合部署”时，必须追问：
1. 用什么指标衡量？
2. 与谁比较？
3. 在什么条件下比较？
4. baseline 是否公平？
5. 是否有消融证明设计选择的作用？
6. 是否有压力测试或真实场景证据？
7. 是否说明适用边界？
```

## 4. 典型论文结构

工程 / 系统 / 设计型论文不一定采用 IMRaD，而更常采用围绕问题、需求、设计、实现和评价展开的结构。

典型结构如下：

```text
Title
Abstract
Keywords

1. Introduction / Problem / Motivation
2. Background and Related Work
3. Requirements / Design Goals
4. Design Rationale
5. System Architecture
6. Implementation
7. Evaluation
8. Case Study / Deployment / User Study
9. Discussion
10. Conclusion

References
Appendices / Artifact / Supplementary Materials
```

不同领域可有变体：

- HCI 论文可能强调 User Needs、Design Process、User Study；
- 网络系统论文可能强调 Protocol Design、System Implementation、Performance Evaluation；
- 机器人论文可能强调 Task Setting、Hardware Platform、Control Architecture、Experiments；
- 芯片论文可能强调 Architecture、Circuit Design、Implementation、Benchmarking；
- 软件系统论文可能强调 System Design、Implementation、Evaluation、Ablation；
- 仪器装置论文可能强调 Design Principle、Hardware, Calibration, Validation；
- 能源系统论文可能强调 System Modeling、Control Strategy、Field Test、Efficiency Evaluation。

### 4.1 章节功能总览

| 章节 | 核心功能 | 不应写成 |
|---|---|---|
| Problem / Motivation | 说明真实问题和研究必要性 | 泛泛行业背景 |
| Requirements | 明确系统必须满足的需求和约束 | 功能愿望清单 |
| Design Rationale | 解释为什么这样设计 | 模块说明书 |
| System Architecture | 展示系统整体结构和模块关系 | 截图或框图堆砌 |
| Implementation | 说明关键实现细节和工程决策 | 开发日志 |
| Evaluation | 用证据证明系统主张 | 简单 demo |
| Discussion | 分析取舍、边界、失败情况和部署意义 | 重复实验结果 |
| Conclusion | 回答问题并压缩贡献 | 宣传总结 |

### 4.2 推荐结构一：标准系统论文结构

```text
1. Introduction
2. Background and Related Work
3. Problem Statement and Requirements
4. System Design
5. Implementation
6. Evaluation
7. Discussion
8. Conclusion
```

适合软件系统、网络系统、工程工具、智能制造平台等。

### 4.3 推荐结构二：设计研究型结构

```text
1. Introduction
2. Motivation and User / Scenario Analysis
3. Design Goals
4. Design Process and Rationale
5. Prototype / System
6. Evaluation / User Study
7. Reflection and Implications
8. Conclusion
```

适合 HCI、交互设计、设计型系统、用户场景系统。

### 4.4 推荐结构三：硬件 / 装置型结构

```text
1. Introduction
2. Design Requirements
3. Hardware / Device Architecture
4. Control / Signal Processing / Calibration
5. Implementation and Fabrication
6. Experimental Validation
7. Comparison and Deployment
8. Discussion
9. Conclusion
```

适合芯片、仪器、机器人、嵌入式装置、能源设备。

### 4.5 推荐结构四：方法 + 系统结合型结构

```text
1. Introduction
2. Problem and Existing Limitations
3. Method Overview
4. System Integration
5. Implementation Details
6. Experimental Setup
7. Results and Ablation
8. Case Study
9. Discussion
10. Conclusion
```

适合提出新方法并集成到系统中的论文。

## 5. 各章节写作指南

## 5.1 标题 Title

### 写作目标

标题应准确传达系统对象、应用场景、核心技术或主要贡献，使目标读者能够判断论文属于什么问题、什么系统、什么贡献。

### 应包含内容

标题可包含以下元素：

- 系统名称；
- 目标问题；
- 应用场景；
- 核心技术；
- 关键性能目标；
- 架构特征；
- 部署场景；
- 用户群体或任务。

### 常见标题类型

| 类型 | 结构 | 示例形式 |
|---|---|---|
| 系统名 + 功能 | X: A System for Y | 某系统用于某任务 |
| 架构 + 场景 | A Lightweight Architecture for Edge Y | 强调架构特征 |
| 问题 + 方案 | Enabling X under Y Constraints | 强调约束 |
| 设计目标 + 系统 | Scalable X for Real-Time Y | 强调性能目标 |
| 方法 + 系统 | Integrating A and B for C | 强调技术组合 |
| 设计洞见型 | Rethinking X for Y Systems | 强调重新设计 |

### 常见错误

- 标题像产品广告；
- 只写系统名称，不说明做什么；
- 使用“高效”“智能”“先进”但不具体；
- 标题声称“实时”“可扩展”“低功耗”，正文没有评价；
- 标题过长，堆叠所有模块；
- 标题无法区分与已有系统的差异。

### Agent 质量判断

```text
1. 标题是否说明系统解决什么问题？
2. 是否体现核心贡献或设计特点？
3. 是否避免宣传词？
4. 是否与评价结果一致？
5. 是否能被目标领域读者检索到？
```

## 5.2 摘要 Abstract

### 写作目标

摘要应压缩完整工程论证链：

```text
问题
→ 现有方案不足
→ 需求 / 约束
→ 设计方案
→ 实现
→ 评价结果
→ 贡献
```

### 推荐结构

```text
1. 背景 / 问题句：指出真实工程问题或应用场景。
2. 缺口句：说明现有方案为何不足。
3. 设计目标句：说明系统需要满足哪些关键约束。
4. 方案句：概括本文系统的核心设计。
5. 评价句：报告关键实验结果和对比结果。
6. 贡献句：说明本文提供的新架构、新系统、新经验或新能力。
```

### 示例骨架

```text
在 X 场景中，系统需要在 Y 约束下完成 Z 任务。然而，现有方案通常依赖 A，导致 B 问题，难以满足 C 需求。本文提出 D，一个面向 E 的系统，通过 F 架构、G 调度策略和 H 接口设计实现低延迟与可扩展部署。我们在 I 数据集 / 平台 / 真实场景中评估 D，并与 J 个 baseline 进行比较。结果显示，D 在 K 指标上提升了 L，同时保持 M 资源开销。本文的贡献在于提出了 N 设计，并证明其在 O 条件下能够有效解决 P 问题。
```

### 摘要必须避免

- 只写“本文设计并实现了一个系统”；
- 不说明现有方案不足；
- 不报告评价结果；
- 用“效果良好”“性能优越”替代具体指标；
- 夸大部署范围；
- 说“系统可扩展”但不给扩展实验；
- 说“用户体验好”但没有用户研究或任务指标。

### Agent 质量判断

```text
1. 摘要是否说明真实问题？
2. 是否说明现有方案不足？
3. 是否给出核心设计，而非功能列表？
4. 是否报告关键评价结果？
5. 是否说明贡献类型？
6. 是否限定系统适用范围？
```

## 5.3 Problem / Motivation 问题与动机

### 写作目标

Problem / Motivation 的目标是让读者相信：

```text
这个问题真实存在；
这个问题重要；
已有方案没有充分解决；
本文提出系统是必要的。
```

### 应包含内容

- 真实场景描述；
- 用户、系统或应用痛点；
- 当前方案的限制；
- 工程约束；
- 问题的学术或工程价值；
- 本文要解决的具体问题；
- 解决该问题的难点。

### 好的问题写法

弱写法：

```text
随着人工智能的发展，智能制造系统越来越重要。
```

强写法：

```text
在中小型制造车间中，质量检测系统通常需要在低成本边缘设备上完成实时缺陷识别。然而，现有云端检测方案依赖稳定网络连接，难以满足生产线毫秒级响应和数据本地化要求。
```

弱写法：

```text
现有系统存在效率低的问题。
```

强写法：

```text
现有系统在高并发请求下主要瓶颈来自同步任务调度：当请求数超过每秒 500 个时，队列等待时间占端到端延迟的 60% 以上，导致交互式应用无法满足 100 ms 响应目标。
```

### 问题真实性的证据来源

工程问题可以通过以下方式证明真实：

- 真实部署场景；
- 用户访谈或需求调研；
- 系统日志；
- 性能 profiling；
- 工业标准或应用需求；
- 现有系统失败案例；
- 实际任务约束；
- 数据规模增长趋势；
- 资源受限环境；
- 现场实验观察；
- benchmark 中暴露的瓶颈。

### 常见错误

- 动机过于宏大；
- 问题是作者人为设定的，不像真实场景；
- 只说“现有方法不足”，不说明不足在哪里；
- 没有工程约束；
- 没有说明为什么不能直接用已有系统；
- 只强调应用价值，不说明研究问题；
- 问题与后文设计目标不一致。

### Agent 质量判断

```text
1. 问题是否具体到场景和任务？
2. 是否说明真实约束？
3. 是否说明已有方案为什么不够？
4. 是否有数据、案例或场景证据支持问题真实性？
5. 是否能自然推出系统需求？
6. 是否避免泛泛行业背景？
```

---

## 5.4 Requirements / Design Goals 需求与设计目标

### 写作目标

Requirements / Design Goals 用于将模糊问题转化为可设计、可评价的目标。

它回答：

```text
系统必须满足什么？
为什么这些需求重要？
这些需求如何影响设计？
这些需求如何在 Evaluation 中被验证？
```

### 需求类型

| 类型 | 示例 | 评价方式 |
|---|---|---|
| 功能需求 | 支持任务 A、B、C | 功能覆盖、案例验证 |
| 性能需求 | 低延迟、高吞吐、高精度 | 指标测试 |
| 可靠性需求 | 故障恢复、鲁棒性 | 故障注入、长期运行 |
| 可扩展需求 | 支持更多用户、节点、数据 | 扩展实验 |
| 资源需求 | 低功耗、低内存、低成本 | 资源测量 |
| 部署需求 | 边缘部署、跨平台、易维护 | 部署实验 |
| 可用性需求 | 易学、易用、低负担 | 用户研究 |
| 安全 / 隐私需求 | 权限隔离、数据本地化 | 安全分析、攻击测试 |
| 实时性需求 | 控制周期、响应时间 | 实时测试 |
| 兼容性需求 | 与现有系统集成 | 集成案例 |

### 好的设计目标写法

弱写法：

```text
系统应具有高性能和良好扩展性。
```

强写法：

```text
G1：系统应在 100 ms 内完成端到端响应，以满足交互式控制任务的实时要求。
G2：系统应在 8 GB 内存限制下支持至少 1,000 个并发连接，以适应边缘服务器部署。
G3：系统应在单个节点故障时保持核心服务可用，以满足现场连续运行需求。
```

### 需求与评价的对应关系

| 设计目标 | 对应评价 |
|---|---|
| G1：低延迟 | 延迟分布、P95 / P99 latency |
| G2：高吞吐 | 吞吐量随负载变化曲线 |
| G3：低资源占用 | CPU、内存、功耗测量 |
| G4：可扩展 | 节点数 / 用户数 / 数据量扩展实验 |
| G5：可靠 | 故障恢复时间、错误率、长时间运行 |
| G6：易用 | 用户任务完成率、时间、错误、满意度 |
| G7：可部署 | 部署步骤、依赖、现场运行案例 |

### 常见错误

- 需求写成愿望清单；
- 需求无法评价；
- 需求与真实问题脱节；
- 需求之间存在冲突但不讨论取舍；
- 需求没有优先级；
- 后文 Evaluation 没有回应需求；
- 写了“实时”“可扩展”“鲁棒”，但没有量化或测试。

### Agent 质量判断

```text
1. 每个需求是否来自真实问题？
2. 每个需求是否可评价？
3. 是否说明需求之间的 trade-off？
4. 是否能从需求推出设计选择？
5. Evaluation 是否逐项回应这些需求？
```

## 5.5 Design Rationale 设计理据

### 写作目标

Design Rationale 是工程系统论文最重要但最常被写弱的部分。  
它不是描述“系统有什么模块”，而是解释：

```text
为什么选择这种设计？
为什么不选择其他设计？
该选择回应了哪个需求？
该选择带来什么取舍？
后文如何评价它是否有效？
```

### Design Rationale 应包含内容

- 关键设计挑战；
- 可选设计路径；
- 作者选择的方案；
- 选择理由；
- 与需求的对应关系；
- 取舍分析；
- 预期收益；
- 可能代价；
- 后续评价方式。

### 设计理据写法

弱写法：

```text
系统包括数据采集模块、处理模块和展示模块。
```

强写法：

```text
为降低端到端延迟，系统将数据预处理从中心服务器下沉到边缘节点。该设计减少了原始数据传输量，但增加了边缘设备的计算负担。为避免资源开销过高，本文采用轻量级流式预处理算法，并在 Evaluation 中分别测试延迟、带宽和 CPU 占用。
```

弱写法：

```text
我们使用微服务架构实现系统。
```

强写法：

```text
本文采用微服务架构并非为了增加模块数量，而是为了满足现场部署中的异构扩展需求。由于数据采集、模型推理和可视化服务的负载变化不同，将其拆分为独立服务允许系统单独扩展瓶颈模块。后文的扩展性实验将验证该设计在高负载下是否优于单体架构。
```

### 设计选择与需求映射

推荐表格：

| 需求 / 挑战 | 设计选择 | 理由 | 预期收益 | 评价方式 |
|---|---|---|---|---|
| 低延迟 | 边缘预处理 | 减少传输和中心排队 | 降低端到端延迟 | 延迟实验 |
| 高并发 | 异步队列 | 解耦任务阻塞 | 提升吞吐 | 压力测试 |
| 可扩展 | 微服务拆分 | 独立扩展瓶颈模块 | 支持负载增长 | 扩展实验 |
| 可靠性 | 心跳与故障转移 | 发现并恢复故障 | 降低停机时间 | 故障注入 |
| 易用性 | 可视化流程编排 | 降低配置复杂度 | 提高任务完成率 | 用户研究 |

### 常见错误

- 只描述架构，不解释为什么这样设计；
- 不讨论 alternative design；
- 没有 trade-off；
- 设计选择与需求不对应；
- 设计理据与 Evaluation 没有闭环；
- 把设计理据写成技术名词堆砌；
- 没有说明设计的新意。

### Agent 质量判断

```text
1. 是否解释了每个关键设计选择的理由？
2. 是否说明该选择回应哪个需求？
3. 是否讨论替代方案和取舍？
4. 是否说明预期收益？
5. 是否在 Evaluation 中验证该设计选择？
6. 是否避免只写模块说明？
```

## 5.6 System Architecture 系统架构

### 写作目标

System Architecture 负责说明系统整体如何工作。  
读者应能理解：

```text
系统由哪些部分组成？
模块之间如何交互？
数据如何流动？
控制逻辑如何运行？
接口如何定义？
关键路径在哪里？
设计如何体现前文 rationale？
```

### 应包含内容

- 系统总览图；
- 模块划分；
- 数据流；
- 控制流；
- 接口定义；
- 状态管理；
- 运行流程；
- 部署拓扑；
- 软硬件边界；
- 输入与输出；
- 异常处理路径；
- 与外部系统的交互。

### 架构图写作要求

一张好的系统架构图应能回答：

```text
1. 输入从哪里来？
2. 输出到哪里去？
3. 系统分为哪些模块？
4. 模块之间传递什么数据？
5. 哪些模块是本文贡献？
6. 哪些模块使用已有工具？
7. 关键路径和瓶颈在哪里？
8. 设计目标如何体现在架构中？
```

### 模块说明写法

弱写法：

```text
系统由采集模块、处理模块、存储模块和展示模块组成。
```

强写法：

```text
系统包含四个模块。采集模块负责从传感器流式接收数据，并将时间戳和设备状态一并写入消息队列。预处理模块在边缘节点执行降采样和异常检测，以减少中心服务器负载。调度模块根据任务优先级和节点负载分配推理请求。展示模块仅订阅处理后的事件流，从而避免直接访问原始数据。
```

### 接口和流程

工程论文应在必要时说明：

- API 输入输出；
- 消息格式；
- 协议；
- 数据结构；
- 调用顺序；
- 状态转换；
- 错误码；
- 同步 / 异步机制；
- 时间约束；
- 并发控制。

### 常见错误

- 架构图像产品框图；
- 模块名称抽象但功能不清；
- 没有数据流和控制流；
- 不区分已有组件和本文贡献；
- 不说明模块接口；
- 不说明部署环境；
- 架构和设计目标脱节；
- 图中出现模块，正文不解释；
- 正文解释模块，图中看不到。

### Agent 质量判断

```text
1. 架构图是否能独立传达系统逻辑？
2. 模块边界是否清楚？
3. 数据流和控制流是否清楚？
4. 关键设计是否在架构中体现？
5. 是否区分本文贡献和已有组件？
6. 是否足以让读者理解系统如何运行？
```

## 5.7 Implementation 实现

### 写作目标

Implementation 说明系统如何被具体实现。  
它不应写成开发日志，而应交代影响系统能力、性能、复现和评价的关键实现细节。

### 应包含内容

根据论文类型，可包括：

- 编程语言、框架和依赖；
- 硬件平台；
- 操作系统和运行环境；
- 通信协议；
- 数据库或存储系统；
- 关键算法实现；
- 调度策略；
- 缓存机制；
- 并发模型；
- 内存管理；
- 设备驱动；
- 控制周期；
- 部署脚本；
- 参数设置；
- 安全机制；
- 代码规模；
- 开源或 artifact 信息；
- 与现有系统集成方式。

### 实现细节选择原则

不是所有实现细节都要写。应优先写：

```text
1. 影响性能的实现；
2. 影响可复现性的实现；
3. 体现设计创新的实现；
4. 与工程约束相关的实现；
5. 评价实验需要理解的实现；
6. 与 baseline 公平比较相关的实现。
```

### 好的实现写法

弱写法：

```text
系统采用 Python 开发，前端使用 Vue，后端使用 Flask。
```

强写法：

```text
系统后端采用异步事件循环处理传感器流，以避免同步 I/O 导致的队列阻塞。前端仅接收聚合后的事件摘要，而不渲染原始数据流，从而将浏览器端刷新延迟控制在 50 ms 以内。该实现选择对应 G1 中的实时反馈需求。
```

弱写法：

```text
我们实现了一个调度模块。
```

强写法：

```text
调度模块维护每个边缘节点的任务队列长度、最近 30 秒平均处理时间和心跳状态。每次任务到达时，调度器根据加权负载分数选择目标节点。该策略避免将任务分配给暂时在线但已过载的节点。
```

### Implementation 与 Architecture 的区别

| Architecture | Implementation |
|---|---|
| 系统由什么组成 | 系统如何具体实现 |
| 模块关系 | 数据结构、协议、线程、参数 |
| 设计层级 | 工程实现层级 |
| 解释为什么这样组织 | 解释如何落地 |
| 面向理解系统 | 面向复现和评价 |

### 常见错误

- 写成“用了哪些技术栈”；
- 只列框架名称；
- 不写关键参数；
- 不说明硬件环境；
- 不说明实现与设计目标关系；
- 省略影响性能的细节；
- 过度描述普通 CRUD 功能；
- 不说明 baseline 是否同等优化；
- 不说明开源、数据或复现方式。

### Agent 质量判断

```text
1. Implementation 是否包含影响结果的关键细节？
2. 是否足以支持复现或评价？
3. 是否说明运行环境和参数？
4. 是否解释实现如何服务设计目标？
5. 是否避免技术栈清单化？
6. 是否说明与已有组件的关系？
```

## 5.8 Evaluation 评价

### 写作目标

Evaluation 是工程系统论文的证据核心。  
它必须证明：

```text
系统是否解决了问题？
设计目标是否达成？
关键设计选择是否有效？
系统是否优于已有方案？
系统在真实约束下是否可靠？
系统边界在哪里？
```

### Evaluation 应回答的问题

```text
Q1：系统是否有效完成目标任务？
Q2：系统是否优于已有方案或 baseline？
Q3：关键设计模块是否真的带来收益？
Q4：系统在压力、规模、异常情况下表现如何？
Q5：系统是否可部署或可用于真实场景？
Q6：系统的代价、限制和失败情况是什么？
```

### 评价指标设计

评价指标必须对应设计目标。

| 设计主张 | 常用指标 |
|---|---|
| 有效性 | 准确率、成功率、任务完成率、产出质量 |
| 低延迟 | 平均延迟、P95 / P99 延迟、端到端响应时间 |
| 高吞吐 | 每秒请求数、处理速率、队列长度 |
| 可扩展 | 随节点数、用户数、数据量变化的性能曲线 |
| 可靠性 | 故障恢复时间、错误率、可用性、MTBF |
| 鲁棒性 | 噪声、扰动、异常输入下性能 |
| 资源效率 | CPU、GPU、内存、功耗、带宽、成本 |
| 可部署 | 安装时间、依赖数量、维护复杂度、现场运行时长 |
| 易用性 | 任务完成时间、错误数、学习成本、主观评分 |
| 安全性 | 攻击成功率、泄露风险、权限隔离效果 |
| 可维护性 | 模块耦合、配置复杂度、更新成本 |

### Baseline 选择

baseline 是审稿人最关注的问题之一。  
合理 baseline 应包括：

- 当前主流方法；
- 领域标准系统；
- 最接近问题设置的已有方案；
- 简单但强的 baseline；
- 商业或开源系统；
- 无关键模块版本；
- 作者系统的不同变体；
- 人工流程或传统流程。

### 公平 baseline 的要求

```text
1. 相同任务；
2. 相同数据；
3. 相同硬件或合理归一化；
4. 相同负载条件；
5. 相同评价指标；
6. 合理调参；
7. 不故意弱化 baseline；
8. 说明 baseline 的实现来源；
9. 对无法直接比较的差异进行解释。
```

### 对比实验

对比实验用于回答：

```text
本文系统是否优于已有方案？
在哪些指标上优于？
代价是什么？
在什么条件下不优于？
```

写法：

```text
在相同硬件和输入负载下，本文系统与 Baseline A、Baseline B 和 Baseline C 进行比较。结果显示，本文系统在 P95 延迟上分别降低……，但内存占用增加……。这表明异步调度设计在延迟方面带来收益，但也引入额外缓存开销。
```

### 消融实验 Ablation

消融实验用于证明关键设计选择真的有用。

常见消融方式：

- 去掉模块；
- 替换策略；
- 关闭优化；
- 改变参数；
- 比较不同架构版本；
- 比较有无缓存；
- 比较同步与异步；
- 比较本地处理与云端处理；
- 比较单模块与端到端系统。

消融实验应对应 Design Rationale。

| 设计选择 | 消融方式 | 证明什么 |
|---|---|---|
| 边缘预处理 | 关闭边缘预处理 | 证明其降低带宽和延迟 |
| 异步调度 | 改为同步调度 | 证明其提升吞吐 |
| 缓存机制 | 禁用缓存 | 证明其降低重复计算 |
| 模块化架构 | 改为单体部署 | 证明其可扩展性 |
| 自适应控制 | 固定参数策略 | 证明其鲁棒性 |

### 压力测试 Stress Test

压力测试用于证明系统在极端或高负载情况下的边界。

常见压力测试：

- 高并发请求；
- 大数据量；
- 长时间运行；
- 网络延迟或丢包；
- 节点故障；
- 传感器异常；
- 输入噪声；
- 低电量或低算力；
- 多用户同时操作；
- 峰值负载；
- 真实现场干扰。

压力测试不一定要求系统“永远不失败”，但要说明：

```text
系统何时开始退化？
退化方式是什么？
是否可恢复？
瓶颈在哪里？
设计是否优雅降级？
```

### 案例分析 Case Study

案例分析适合证明系统在真实任务中的使用价值。

好的 case study 应说明：

- 场景；
- 使用者或任务；
- 输入；
- 系统运行过程；
- 输出；
- 与原流程相比的变化；
- 用户或系统收益；
- 失败或限制。

Case study 不应替代系统性评价。  
它通常补充说明“系统如何实际发挥作用”。

### 用户研究

HCI、工具系统、交互系统常需要用户研究。

常见指标：

- 任务完成率；
- 完成时间；
- 错误率；
- 学习成本；
- 认知负担；
- 主观满意度；
- 可理解性；
- 信任度；
- 用户偏好；
- 访谈反馈。

写作重点：

```text
1. 用户是谁？
2. 任务是什么？
3. 对比条件是什么？
4. 指标是什么？
5. 样本量是多少？
6. 如何控制顺序效应？
7. 结果是否支持设计目标？
8. 主观结果是否与行为指标一致？
```

### 部署评价

工程系统论文如果声称可部署，应提供：

- 部署环境；
- 部署时间；
- 依赖条件；
- 硬件资源；
- 维护成本；
- 运行时长；
- 故障情况；
- 用户或运营反馈；
- 与实验室环境差异。

### 常见错误

- 只做 demo，不做系统评价；
- 只与弱 baseline 比较；
- 没有消融实验；
- 没有压力测试；
- 指标与设计目标不对应；
- 只报告平均值，不报告尾延迟或分布；
- 只报告最优结果；
- 不说明硬件和负载；
- 不报告失败案例；
- 用单一案例证明普遍有效；
- 评价没有覆盖系统主张。

### Agent 质量判断

```text
1. Evaluation 是否逐项回应 Requirements？
2. baseline 是否合理且公平？
3. 指标是否对应系统主张？
4. 是否有消融证明关键设计？
5. 是否有压力测试证明边界？
6. 是否有真实场景或案例支持应用价值？
7. 是否报告代价和失败情况？
8. 是否避免只展示成功 demo？
```

---

## 5.9 Discussion 讨论

### 写作目标

Discussion 负责解释评价结果背后的意义，讨论设计取舍、边界条件、失败情况和可推广经验。

它回答：

```text
评价结果说明了什么？
哪些设计选择最关键？
系统在哪些条件下有效？
哪些条件下失效？
与已有系统相比，本文贡献是什么？
这些设计经验能否迁移到其他系统？
```

### 应包含内容

- 关键结果综合；
- 设计选择的有效性解释；
- trade-off 分析；
- 与已有系统比较；
- 系统边界；
- 失败案例；
- 部署经验；
- 安全、伦理、维护或成本问题；
- 可推广的设计启示；
- 未来改进方向。

### 设计取舍写法

弱写法：

```text
本系统仍有一些不足，未来需要改进。
```

强写法：

```text
本文系统通过边缘预处理降低了网络带宽需求，但代价是边缘设备 CPU 占用增加。实验结果显示，在负载低于每秒 800 个事件时，该代价可接受；当负载继续上升时，预处理模块成为新的瓶颈。因此，该设计适合网络带宽受限但边缘计算资源相对充足的场景。
```

### 失败案例写法

工程论文不应隐藏失败。  
好的失败分析可以提高可信度。

```text
在弱网条件下，系统的同步状态更新模块表现不稳定，导致部分任务重复执行。该失败说明当前一致性机制仍依赖最低网络质量保证。未来版本可采用本地日志缓冲和延迟合并策略以改善该问题。
```

### 常见错误

- 只是重复 Evaluation；
- 不讨论系统代价；
- 不说明什么时候不适用；
- 不处理失败案例；
- 只说“未来会优化”；
- 不把评价结果上升为设计经验；
- 对 novelty 解释不足；
- 回避 baseline 中表现不佳的结果。

### Agent 质量判断

```text
1. Discussion 是否解释评价结果？
2. 是否讨论设计取舍？
3. 是否说明系统边界？
4. 是否处理失败和负面结果？
5. 是否提炼可迁移的设计经验？
6. 是否避免重复 Evaluation？
```

---

## 5.10 Conclusion 结论

### 写作目标

Conclusion 应压缩全文，回到问题、设计、评价和贡献。

### 推荐结构

```text
本文针对 X 场景中的 Y 问题，提出了 Z 系统。
该系统基于 A、B 和 C 设计选择，满足了 D、E 和 F 需求。
评价结果显示，系统在 G 指标上优于 baseline，并在 H 测试中表现出 I 特性。
本文的贡献在于 J。
同时，系统仍受 K 条件限制，未来可进一步研究 L。
```

### 常见错误

- 只总结系统功能；
- 不回到真实问题；
- 不提评价证据；
- 贡献写成“实现了一个系统”；
- 结论比实验更强；
- 不说明边界；
- 写成产品宣传语。

### Agent 质量判断

```text
1. 是否回答了最初问题？
2. 是否压缩了核心设计选择？
3. 是否引用评价证据支持贡献？
4. 是否说明 novelty？
5. 是否限定适用范围？
6. 是否避免空泛宣传？
```

## 6. 常见贡献类型

工程 / 系统论文的 novelty 可以来自多个方面。Agent 应帮助用户从系统实现中提炼真正贡献，而不是停留在“做了一个系统”。

### 6.1 新问题贡献

论文识别并定义了一个此前未被充分研究但真实存在的工程问题。

示例：

```text
本文首次系统化定义了边缘制造环境中“低带宽、低算力、强实时”三重约束下的缺陷检测调度问题。
```

质量要求：

- 问题必须真实；
- 需要场景证据或数据支持；
- 不能只是换一个应用名；
- 需要说明已有系统为何不适用。

### 6.2 新架构贡献

论文提出新的系统架构以满足特定约束。

示例：

```text
本文提出一种边缘—云协同的分层推理架构，将低延迟决策留在边缘端，将高成本模型更新放在云端，从而同时满足实时性和持续学习需求。
```

质量要求：

- 架构选择要有 rationale；
- 要有对比和消融；
- 不能只是把常规模块重新命名。

### 6.3 新组合贡献

论文将已有技术以新的方式组合，解决此前组合不能解决的问题。

示例：

```text
本文将事件驱动调度、轻量级缓存和局部一致性协议组合，用于解决弱网环境下多机器人协作任务的实时协调问题。
```

质量要求：

- 说明组合为什么非平凡；
- 说明单个组件为什么不够；
- 评价组合带来的系统级收益。

### 6.4 新性能贡献

系统在关键指标上显著优于已有方案。

示例：

```text
在相同硬件和负载下，本文系统将 P95 延迟降低 40%，同时保持内存开销增加低于 8%。
```

质量要求：

- baseline 公平；
- 指标与需求对应；
- 结果不是微小优化；
- 需要说明性能提升来自什么设计。

### 6.5 新应用场景贡献

系统使某项技术在新场景中可用。

示例：

```text
本文展示了实时视觉检测系统在低成本边缘设备上的可部署性，使该技术可用于中小型制造车间。
```

质量要求：

- 场景有真实约束；
- 需要现场或近真实评价；
- 不能只是换数据集；
- 要说明对场景的适配设计。

### 6.6 新方法 / 工作流贡献

论文提出一种工程流程、设计方法、开发工具链或运行机制。

示例：

```text
本文提出一种面向多传感器系统的自动校准工作流，将人工配置时间从小时级降低到分钟级。
```

质量要求：

- 工作流要可复用；
- 需要与现有流程比较；
- 需要评价效率、错误率或可用性。

### 6.7 新装置 / 仪器贡献

论文设计并验证了一种仪器、硬件装置或测量平台。

示例：

```text
本文设计了一种低成本便携式环境监测装置，并通过标定实验和现场部署验证其测量精度和稳定性。
```

质量要求：

- 说明硬件设计 rationale；
- 报告校准、误差和可靠性；
- 与标准设备或已有装置比较；
- 说明适用范围。

### 6.8 设计经验贡献

论文从系统构建和评价中提炼可迁移设计原则。

示例：

```text
本文表明，在弱网协作系统中，局部一致性与延迟容忍机制的组合比全局同步更适合实时控制任务。
```

质量要求：

- 不能只写经验感想；
- 必须由评价证据支撑；
- 需要说明适用条件。

## 7. 证据与论证标准

## 7.1 强证据

工程系统论文中的强证据通常包括：

```text
1. 问题来自真实场景或可靠需求分析；
2. 设计目标具体且可评价；
3. 系统实现足够完整；
4. baseline 合理且公平；
5. 指标与主张直接对应；
6. 消融实验证明关键模块有效；
7. 压力测试揭示系统边界；
8. 真实案例或部署证明场景价值；
9. 报告资源开销、失败情况和 trade-off；
10. 结论限定在评价覆盖范围内。
```

### 强证据示例

| 主张 | 强证据组合 |
|---|---|
| 系统更快 | 公平 baseline + 端到端延迟 + 尾延迟 + 负载变化 |
| 系统可扩展 | 节点数 / 用户数 / 数据量扩展实验 |
| 系统更可靠 | 故障注入 + 长时间运行 + 恢复时间 |
| 系统易用 | 用户研究 + 任务完成率 + 错误率 + 访谈 |
| 系统可部署 | 真实环境部署 + 资源占用 + 维护记录 |
| 架构设计有效 | 消融实验 + 替代架构比较 |
| 系统优于已有方案 | 合理 baseline + 相同条件 + 多指标对比 |
| 系统适合新场景 | 场景约束说明 + 现场案例 + 指标达标 |

## 7.2 弱证据

弱证据常见表现：

- 只有功能 demo；
- 只展示成功案例；
- 没有 baseline；
- baseline 不公平；
- 只比较单一指标；
- 没有消融实验；
- 没有压力测试；
- 没有说明硬件环境；
- 没有报告资源开销；
- 声称可部署但没有部署证据；
- 声称易用但没有用户研究；
- 声称可靠但没有故障或长期测试；
- 只写“用户反馈良好”；
- 评价无法对应设计目标。

## 7.3 评价主张匹配原则

| 论文主张 | 必要评价 |
|---|---|
| 有效 | 任务成功率、准确率、产出质量、案例验证 |
| 高效 | 延迟、吞吐、资源消耗、运行时间 |
| 可扩展 | 规模变化实验 |
| 鲁棒 | 噪声、异常输入、弱网、故障测试 |
| 可靠 | 长时间运行、故障恢复、错误率 |
| 易用 | 用户研究、任务时间、错误率、主观反馈 |
| 可部署 | 真实环境、资源、依赖、运维成本 |
| 优于已有方案 | 公平 baseline 对比 |
| 设计选择有效 | 消融实验 |
| 适合真实场景 | case study / field deployment |

## 7.4 baseline 公平性判断

Agent 应使用以下问题检查 baseline：

```text
1. baseline 是否是领域内合理选择？
2. 是否遗漏了明显更强的已有方法？
3. 是否在相同硬件、数据、负载下比较？
4. baseline 是否经过合理调参？
5. 作者系统是否得到额外优势？
6. 指标是否对所有系统公平？
7. 是否解释无法完全公平比较的地方？
```

## 7.5 结论强度分级

| 证据情况 | 合适表达 | 避免表达 |
|---|---|---|
| 多 baseline + 多指标 + 消融充分 | 结果表明系统在……条件下优于…… | 全面优于所有方案 |
| 只有实验室测试 | 在实验环境中表现出…… | 可直接部署于所有真实场景 |
| 有案例但无系统评价 | 案例显示潜在可用性 | 证明系统普遍有效 |
| 有用户反馈但样本小 | 初步用户研究表明…… | 用户一致认为…… |
| 有单一负载实验 | 在该负载下…… | 具有高可扩展性 |
| 有间接指标 | 结果提示…… | 证明…… |

## 8. 文献综述写法

## 8.1 工程系统论文中的相关工作功能

Related Work 的目标不是展示作者读了多少文献，而是说明：

```text
已有系统如何解决类似问题？
它们的设计假设是什么？
它们为什么不能满足本文场景需求？
本文系统在哪些设计维度上不同？
```

### Related Work 应回答

- 现有系统有哪些类型？
- 它们解决了什么问题？
- 它们依赖什么假设？
- 它们在本文场景下为什么不足？
- 本文不是简单重复在哪里？
- 本文系统相较已有方案的设计位置是什么？

### 推荐组织方式

#### 方式一：按系统类别组织

```text
云端集中式系统
边缘部署系统
混合架构系统
本文系统定位
```

适合网络、边缘、制造、机器人系统。

#### 方式二：按设计策略组织

```text
同步调度方法
异步调度方法
分层调度方法
本文设计
```

适合系统架构、调度、控制、协议论文。

#### 方式三：按约束组织

```text
低延迟系统
低功耗系统
高可靠系统
本文同时处理多重约束
```

适合多约束工程系统。

#### 方式四：按应用场景组织

```text
实验室场景系统
工业现场系统
移动 / 边缘场景系统
本文目标场景
```

适合应用系统和部署型论文。

#### 方式五：按评价局限组织

```text
已有系统在标准 benchmark 上表现良好
但缺乏真实部署评价
或未测试高负载 / 弱网 / 长时间运行
本文补充……
```

### 好的文献综述写法

弱写法：

```text
A 系统提出了某方法。B 系统实现了某功能。C 系统提高了性能。
```

强写法：

```text
现有系统大致可分为集中式和边缘式两类。集中式系统便于统一管理，但依赖稳定网络连接，因此难以满足低延迟现场控制需求。边缘式系统降低了通信开销，但通常缺少跨节点协调机制，在任务负载不均时容易产生局部瓶颈。本文采用分层协同架构，在保留边缘实时响应能力的同时，引入轻量级全局调度以缓解负载不均问题。
```

### 常见错误

- 按作者逐篇罗列；
- 只说别人做了什么，不说不足；
- 不区分系统假设和场景差异；
- 批评已有系统不公平；
- 忽略最强 baseline；
- 不说明本文与已有系统的关系；
- Related Work 与 Evaluation baseline 脱节；
- 引用很多算法论文，但没有系统相关工作。

### Agent 质量判断

```text
1. Related Work 是否围绕系统问题组织？
2. 是否比较已有系统的设计假设？
3. 是否说明已有系统为何不满足本文需求？
4. 是否自然引出本文架构或设计选择？
5. Evaluation 中的 baseline 是否来自 Related Work？
6. 是否避免贬低式、不公平式批评？
```

---

## 9. 结果、分析或论证主体的组织方式

工程 / 系统论文的主体通常由 Design、Implementation 和 Evaluation 组成。  
主体组织的核心不是“模块 A、模块 B、模块 C”，而是：

```text
设计目标
→ 设计选择
→ 系统实现
→ 评价证据
```

## 9.1 设计主体的组织方式

弱结构：

```text
4.1 用户模块
4.2 数据模块
4.3 算法模块
4.4 展示模块
```

强结构：

```text
4.1 分层架构降低实时任务的中心依赖
4.2 异步调度缓解高并发下的队列阻塞
4.3 边缘缓存减少重复数据传输
4.4 故障转移机制保证现场任务连续性
```

好的系统设计小节应采用：

```text
设计挑战
→ 设计选择
→ 设计理由
→ 实现概览
→ 预期评价
```

## 9.2 Evaluation 的组织方式

Evaluation 不应按“我们做了哪些实验”组织，而应按“主张如何被验证”组织。

推荐结构：

```text
7.1 Experimental Setup
7.2 End-to-End Performance
7.3 Comparison with Baselines
7.4 Ablation Study
7.5 Scalability Test
7.6 Robustness / Reliability Test
7.7 Resource Overhead
7.8 Case Study / Deployment
7.9 Summary of Findings
```

或按研究问题组织：

```text
RQ1：系统是否满足实时性要求？
RQ2：系统是否优于现有方案？
RQ3：关键设计模块是否有效？
RQ4：系统在规模扩大时是否稳定？
RQ5：系统在真实场景中是否可用？
```

### Evaluation 开头的推荐写法

```text
本节评价系统是否满足第三节提出的设计目标。具体而言，我们回答以下问题：
EQ1：系统能否在目标负载下满足实时性要求？
EQ2：与现有系统相比，本文系统在延迟和吞吐上是否具有优势？
EQ3：边缘预处理、异步调度和故障转移三个设计分别贡献多少？
EQ4：系统在节点数、数据量和请求量增加时如何扩展？
EQ5：系统在真实部署场景中表现如何？
```

### 结果段落结构

```text
评价问题
→ 实验设置
→ 结果描述
→ 与 baseline 比较
→ 解释结果
→ 对设计主张的回应
```

示例：

```text
为验证异步调度对延迟的影响，我们在相同负载下比较完整系统和禁用异步调度的变体。结果显示，在每秒 1,000 个请求时，完整系统的 P95 延迟为 82 ms，而同步变体为 147 ms。这说明异步调度显著降低了队列等待时间，支持第四节中关于解耦任务阻塞的设计判断。
```

## 9.3 图表组织

工程系统论文中的图表通常包括：

| 图表类型 | 功能 |
|---|---|
| 架构图 | 展示系统组成和模块关系 |
| 数据流图 | 展示输入、处理、输出路径 |
| 时序图 | 展示交互和控制流程 |
| 部署图 | 展示节点、设备和网络拓扑 |
| 性能曲线 | 展示负载变化下性能 |
| 对比表 | 与 baseline 或已有系统比较 |
| 消融图 | 展示模块贡献 |
| 资源开销图 | CPU、内存、功耗、带宽 |
| 故障恢复图 | 展示可靠性 |
| 用户研究图 | 展示任务和体验指标 |
| 案例流程图 | 展示系统在真实任务中如何使用 |

每张图应有明确论证功能。

```text
Figure 1：系统目标场景与问题
Figure 2：系统整体架构
Figure 3：关键模块流程
Figure 4：端到端性能对比
Figure 5：消融实验
Figure 6：扩展性和压力测试
Figure 7：真实部署或案例分析
```

## 9.4 如何证明“有效、可靠、可扩展、可部署、优于已有方案”

### 有效

需要证明系统能完成目标任务。

证据包括：

- 成功率；
- 精度；
- 任务完成率；
- 目标功能覆盖；
- 端到端案例；
- 用户任务结果；
- 与人工流程或现有系统比较。

### 可靠

需要证明系统在异常或长期运行中保持可用。

证据包括：

- 故障注入；
- 长时间运行；
- 异常输入；
- 网络波动；
- 传感器故障；
- 错误恢复；
- 日志分析；
- 可用性指标。

### 可扩展

需要证明系统在规模增长时性能退化可控。

证据包括：

- 用户数增长；
- 请求数增长；
- 节点数增长；
- 数据量增长；
- 模型规模增长；
- 设备数量增长；
- 多场景扩展。

### 可部署

需要证明系统能在目标环境中运行。

证据包括：

- 真实硬件；
- 真实网络；
- 真实用户；
- 部署时长；
- 依赖环境；
- 运维成本；
- 现场案例；
- 资源占用；
- 维护记录。

### 优于已有方案

需要证明在公平条件下超过 baseline。

证据包括：

- 多 baseline；
- 相同数据；
- 相同硬件；
- 多指标；
- 统计检验或重复；
- 代价分析；
- 失败条件说明。

## 10. 讨论与结论写法

## 10.1 Discussion 的四层提升

工程论文的 Discussion 应把评价结果提升为设计洞见。

```text
结果层：系统在哪些指标上表现如何？
设计层：哪些设计选择导致这些结果？
取舍层：这些收益以什么代价获得？
迁移层：这些经验适用于哪些其他场景？
```

示例：

```text
结果层：系统在高并发负载下显著降低 P95 延迟。
设计层：消融实验表明主要收益来自异步调度。
取舍层：该设计增加了内存缓存开销。
迁移层：该设计适合任务可排队且延迟敏感的边缘系统，但不适合严格顺序一致性的交易系统。
```

## 10.2 如何写设计启示

弱写法：

```text
本文系统对实际应用具有一定参考意义。
```

强写法：

```text
本文结果表明，在弱网和实时性并存的系统中，将短周期决策保留在本地、将长周期优化放在中心节点，是一种有效的分层设计策略。这一经验可能适用于多机器人协作、边缘视频分析和工业传感器网络等场景。
```

## 10.3 如何写限制

限制不应只是“系统还有不足”，而应说明边界。

弱写法：

```text
本系统还有一些性能问题，未来会优化。
```

强写法：

```text
本文系统的调度策略假设任务之间相互独立，因此不适用于具有强依赖关系的工作流。当任务需要全局一致性时，当前异步策略可能产生状态冲突。未来可引入依赖感知调度以扩展适用范围。
```

## 10.4 如何写未来工作

未来工作应从评价中发现的限制推出。

弱写法：

```text
未来将进一步完善系统功能。
```

强写法：

```text
由于当前评价主要覆盖 20 个边缘节点以内的部署规模，未来工作将研究跨区域节点管理和多级调度，以验证该架构在更大规模工业场景中的适用性。
```

## 10.5 Conclusion 写法

好的 Conclusion 应包含：

```text
问题
→ 系统
→ 核心设计
→ 关键评价结果
→ 贡献
→ 边界
```

示例：

```text
本文针对低带宽边缘制造场景中的实时缺陷检测问题，提出了 X 系统。该系统通过边缘预处理、异步调度和轻量级故障转移机制，在资源受限环境下实现了低延迟和高可用。实验结果表明，与三个 baseline 相比，X 在 P95 延迟和带宽占用上均有明显改善，消融实验进一步验证了关键模块的作用。本文的贡献在于展示了一种面向现场部署约束的分层系统设计。当前系统仍主要面向独立任务场景，未来需要进一步支持具有复杂依赖关系的生产流程。
```

## 11. 审稿人关注点

工程 / 系统论文的审稿人通常会从以下角度质疑论文。

### 11.1 问题是否真实

审稿人会问：

```text
1. 这个问题是真实存在的吗？
2. 有用户、场景、日志、部署或数据支撑吗？
3. 是否只是作者为了展示系统而构造的问题？
4. 问题是否足够重要？
5. 为什么不能用已有方案解决？
```

### 11.2 需求和约束是否清楚

审稿人会问：

```text
1. 系统目标是否明确？
2. 工程约束是否具体？
3. 需求是否可评价？
4. 需求之间的取舍是否讨论？
5. 后文评价是否对应这些需求？
```

### 11.3 novelty 是否成立

审稿人会问：

```text
1. 本文新在哪里？
2. 是新问题、新架构、新组合、新性能还是新场景？
3. 是否只是普通工程集成？
4. 与已有系统相比差异是否实质性？
5. 设计选择是否有可迁移价值？
```

### 11.4 设计 rationale 是否充分

审稿人会问：

```text
1. 作者为什么这样设计？
2. 是否考虑过替代设计？
3. 设计选择与需求是否对应？
4. 是否讨论 trade-off？
5. 消融实验是否验证关键设计？
```

### 11.5 系统实现是否可信

审稿人会问：

```text
1. 系统是否真的实现了？
2. 实现是否完整而非概念 demo？
3. 关键细节是否清楚？
4. 是否能复现？
5. 是否说明硬件、软件、参数和环境？
6. 是否公开 artifact 或提供足够补充材料？
```

### 11.6 baseline 是否公平

审稿人会问：

```text
1. baseline 是否代表当前强方法？
2. 是否遗漏重要系统？
3. baseline 是否合理调参？
4. 硬件和负载是否相同？
5. 作者系统是否得到额外优化？
6. 是否解释无法公平比较的因素？
```

### 11.7 评价是否充分

审稿人会问：

```text
1. 指标是否覆盖所有主张？
2. 是否有对比实验？
3. 是否有消融实验？
4. 是否有压力测试？
5. 是否报告资源开销？
6. 是否有真实场景或用户研究？
7. 是否展示失败情况？
8. 是否只做了小规模 demo？
```

### 11.8 结论是否过度

审稿人会问：

```text
1. 实验条件是否支持作者结论？
2. 是否把实验室结果说成真实部署结果？
3. 是否把单一场景推广到所有场景？
4. 是否把小样本用户研究推广为普遍可用性？
5. 是否把性能提升说成全方位优越？
```

## 12. 常见失败模式

## 12.1 写成项目报告

表现：

```text
本文完成了需求分析、系统设计、模块开发和测试。
```

问题：

- 缺少学术问题；
- 缺少设计理据；
- 缺少与已有系统比较；
- 缺少评价闭环。

修正：

```text
1. 将“项目目标”改写为“研究问题 / 工程问题”。
2. 将“模块功能”改写为“设计选择”。
3. 将“测试结果”改写为“评价证据”。
4. 明确 contribution。
```

## 12.2 写成产品说明书

表现：

- 强调功能丰富；
- 界面友好；
- 系统先进；
- 缺少 baseline 和评价。

修正：

```text
1. 删除宣传语。
2. 增加真实问题和约束。
3. 解释为什么这些功能是设计必需。
4. 用指标和实验证据替代主观评价。
```

## 12.3 写成开发日志

表现：

```text
首先开发登录模块，然后开发数据模块，最后完成可视化模块。
```

修正：

```text
1. 不按开发时间写。
2. 按系统设计逻辑写。
3. 只保留影响贡献和评价的实现细节。
4. 将普通工程实现移入补充材料或省略。
```

## 12.4 只有架构，没有 rationale

表现：

- 架构图很复杂；
- 模块很多；
- 但不知道为什么这样设计。

修正：

```text
1. 在架构前增加 Design Rationale。
2. 为每个关键模块说明对应需求。
3. 说明替代方案和取舍。
4. 在 Evaluation 中验证关键模块。
```

## 12.5 评价与主张不匹配

表现：

- 声称可扩展，但只测试单机；
- 声称可靠，但没有故障测试；
- 声称易用，但没有用户研究；
- 声称可部署，但没有部署证据。

修正：

```text
1. 列出所有主张。
2. 为每个主张指定指标和实验。
3. 无法评价的主张降级或删除。
4. 在 Discussion 中说明限制。
```

## 12.6 baseline 弱或不公平

表现：

- 只与过时方法比较；
- baseline 没调参；
- 硬件不同；
- 负载不同；
- 只比较对自己有利指标。

修正：

```text
1. 增加强 baseline。
2. 统一硬件、数据和负载。
3. 说明 baseline 配置。
4. 报告多维指标。
5. 承认不公平比较的限制。
```

## 12.7 缺少消融实验

表现：

- 系统包含多个创新模块；
- 但不知道哪个模块产生收益。

修正：

```text
1. 构造去除关键模块的系统变体。
2. 比较完整系统与变体。
3. 将消融结果对应 Design Rationale。
4. 如果无法做消融，降低设计贡献声称。
```

## 12.8 缺少压力和边界测试

表现：

- 只在小规模、理想条件下运行；
- 却声称可扩展、可靠、可部署。

修正：

```text
1. 增加负载、规模、弱网、异常输入或故障测试。
2. 报告系统退化点。
3. 讨论瓶颈。
4. 限定结论条件。
```

## 12.9 novelty 不清

表现：

- 论文只是集成已有组件；
- 不说明新在哪里；
- 贡献写成“设计并实现了一个系统”。

修正：

```text
1. 判断 novelty 类型：新问题、新架构、新组合、新性能、新场景、新经验。
2. 在 Introduction 明确贡献。
3. 在 Related Work 中定位差异。
4. 在 Evaluation 中证明差异带来的收益。
```

## 13. Agent 写作辅助策略

## 13.1 Agent 的总体任务

Agent 辅助工程 / 系统论文时，应围绕“设计选择与评价证据闭环”进行诊断。

工作流程：

```text
1. 识别系统类型和目标场景。
2. 提取真实问题和工程约束。
3. 判断需求是否清楚且可评价。
4. 提取关键设计选择。
5. 检查 design rationale 是否充分。
6. 检查系统架构是否可理解。
7. 检查实现细节是否支撑复现和评价。
8. 检查 Evaluation 是否覆盖所有主张。
9. 检查 baseline、公平性、消融和压力测试。
10. 从系统实现中提炼贡献。
11. 降级超出评价证据的结论。
12. 重构论文故事线。
```

## 13.2 从系统实现中提炼论文贡献

Agent 可按以下步骤工作：

```text
1. 列出系统解决的问题。
2. 列出系统必须满足的约束。
3. 列出系统中与常规方案不同的设计。
4. 判断这些不同是否有 rationale。
5. 检查是否有评价证据证明这些设计有效。
6. 将“功能”改写为“贡献”。
```

### 功能到贡献的转换

弱：

```text
系统实现了数据采集、处理和展示功能。
```

强：

```text
本文贡献在于提出一种面向弱网环境的边缘数据处理架构，通过本地预处理和异步同步机制，在降低带宽占用的同时保持可接受的端到端延迟。
```

弱：

```text
系统提供了可视化界面。
```

强：

```text
本文设计了一种任务导向的可视化工作流，使非专家用户能够在不编写脚本的情况下完成传感器配置和异常诊断。用户研究显示，该设计降低了任务完成时间和配置错误率。
```

## 13.3 检查工程评价是否支撑设计主张

Agent 应建立“主张—证据矩阵”。

| 设计主张 | 对应证据 | 是否充分 | 修改建议 |
|---|---|---|---|
| 低延迟 | 延迟实验 | 部分充分 | 增加 P95 / P99 |
| 可扩展 | 无 | 不充分 | 增加规模实验或删除主张 |
| 可靠 | 故障恢复实验 | 较充分 | 报告恢复时间 |
| 易用 | 只有主观反馈 | 不充分 | 增加任务完成率 |
| 优于 baseline | 与一个弱 baseline 比较 | 不充分 | 增加强 baseline |

### Agent 判断规则

```text
如果一个主张没有指标，就无法评价。
如果一个指标没有 baseline，就难以证明优越。
如果一个设计没有消融，就难以证明它有效。
如果一个系统没有压力测试，就难以声称可扩展或可靠。
如果一个系统没有真实场景证据，就应谨慎声称可部署。
```

## 13.4 重构系统论文故事线

Agent 可将初稿重构为：

```text
在 X 场景中，现有系统因为 A 和 B 约束无法满足 C 需求。
为此，本文提出 D 系统，并设定 G1、G2、G3 三个设计目标。
为满足这些目标，系统采用 E、F、H 三项关键设计。
实现上，系统通过 I、J、K 机制落地。
评价结果表明，系统在 L、M、N 指标上达到目标，并优于 baseline。
消融实验进一步证明 E 和 F 是主要收益来源。
因此，本文贡献在于……
```

## 13.5 Agent 诊断问题真实性

Agent 应追问：

```text
1. 这个系统面向谁？
2. 用户或部署方为什么需要它？
3. 现有方案具体哪里失败？
4. 有没有日志、案例、访谈、性能 profiling 或真实场景？
5. 问题是否比“我们想做一个系统”更具体？
```

## 13.6 Agent 诊断 design rationale

Agent 应为每个模块生成诊断：

```text
模块名称：
解决的需求：
为什么需要：
替代方案：
当前设计理由：
代价：
评价方式：
```

如果某模块无法填入这些字段，可能只是普通实现，不应被当作贡献。

## 13.7 Agent 诊断 Evaluation

Agent 应优先检查：

```text
1. 是否有 experimental setup？
2. 是否有 baseline？
3. 是否有公平比较条件？
4. 是否有消融？
5. 是否有压力测试？
6. 是否有资源开销？
7. 是否有部署或案例？
8. 是否有失败分析？
9. 是否每个设计目标都有评价？
```

## 13.8 Agent 降级过度声称

高风险词包括：

```text
有效
可靠
可扩展
实时
低成本
轻量
易用
可部署
通用
显著优于
完全解决
工业级
生产可用
```

改写规则：

| 过强表达 | 更稳妥表达 |
|---|---|
| 系统完全解决了 X 问题 | 系统在本文测试条件下缓解了 X 问题 |
| 系统具有良好可扩展性 | 系统在最多 N 个节点的实验中保持性能稳定 |
| 系统适合真实部署 | 系统在 A 场景的初步部署中表现可用 |
| 用户体验显著提升 | 用户研究显示任务完成时间降低，主观评分提高 |
| 全面优于已有方案 | 在延迟和吞吐指标上优于所选 baseline，但资源开销增加 |

## 14. 检查清单

## 14.1 全文闭环检查清单

```markdown
## 问题与动机
- [ ] 论文是否说明真实工程问题？
- [ ] 是否有场景、用户、日志、数据或部署证据？
- [ ] 是否说明已有方案不足？
```
