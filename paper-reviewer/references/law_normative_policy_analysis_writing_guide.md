# 法学 / 规范 / 政策分析型论文写作指南

## 1. 适用范围

本指南适用于以**规则、制度、价值冲突、法律解释、规范推理、政策设计、治理方案和制度后果**为核心研究对象的论文。其目标是帮助 Agent 在辅助用户撰写、重构、审稿和润色此类论文时，判断论文是否建立了清晰、可辩护、可反驳且有实践意义的规范论证链。

适用领域包括但不限于：

- 法学；
- 法理学；
- 宪法学；
- 行政法学；
- 民商法学；
- 刑法学；
- 国际法；
- 比较法；
- 科技法；
- 数据治理与平台治理；
- 伦理学；
- 应用伦理；
- 科技伦理；
- 生命伦理；
- 公共政策；
- 公共治理；
- 制度分析；
- 政策评估；
- 国际关系规范分析；
- 环境治理；
- 教育政策；
- 卫生政策；
- 劳动政策；
- 社会保障政策；
- 城市治理；
- 数字治理；
- 人工智能治理；
- 风险规制；
- 监管政策；
- 企业合规与治理研究。

本指南适用于以下论文类型：

- 法律解释论文；
- 判例评析论文；
- 制度分析论文；
- 规范理论论文；
- 政策分析论文；
- 政策建议型论文；
- 治理方案设计论文；
- 比较制度论文；
- 伦理分析论文；
- 价值冲突分析论文；
- 法政策学论文；
- 法经济学、法社会学中的规范分析部分；
- 国际规范、条约、规则和制度分析论文；
- 科技治理、平台治理、数据治理中的规范和政策论文。

这类论文的核心不是“表达立场”，也不是“提出若干建议”，而是：

> 在事实、规则、价值和后果之间建立可辩护的推理链。

也就是说，论文必须清楚回答：

```text
现实问题是什么？
现行规则或制度为何不足？
该问题涉及哪些规范价值或原则冲突？
作者采用什么法律、伦理或政策框架？
如何从事实、规则和价值推出判断？
可能的反方观点是什么？
作者如何回应？
提出的制度或政策方案是否可行、合法、合比例、可执行？
该方案会产生什么影响和边界？
```

## 2. 不适用范围

本指南不应机械套用于以下文本或研究。

### 2.1 单纯经验社会科学论文

如果论文主要通过问卷、实验、访谈、计量模型、统计识别或田野材料解释社会现象，应使用定量或定性社会科学论文指南。

经验社会科学论文主要问：

```text
发生了什么？
为什么发生？
某变量是否影响另一变量？
某政策是否产生效果？
```

规范 / 政策分析论文主要问：

```text
应当如何评价？
现行规则是否正当？
制度应如何设计？
在价值冲突中应如何权衡？
什么方案更合法、合理、可行？
```

当然，规范 / 政策论文可以使用经验事实，但经验事实必须服务于规范判断或政策论证。

### 2.2 新闻评论、社论或公共议论文

评论文章可以表达明确立场，但学术规范 / 政策论文需要：

- 明确问题；
- 精确定义概念；
- 呈现规则和制度背景；
- 给出规范前提；
- 展开推理；
- 回应反方；
- 说明方案可行性；
- 承认边界和代价。

### 2.3 法律实务文书

法律意见书、诉状、判决摘要、合规备忘录、政策简报等实务文书与学术论文不同。实务文书通常面向具体案件、客户或决策场景；学术论文需要更清楚地进入学术争论、理论框架和制度贡献。

### 2.4 纯法条汇编或政策资料整理

只整理法律条文、政策文件或制度沿革，不构成完整论文。资料整理必须进一步回答：

```text
这些规则之间有什么冲突？
它们体现什么规范取向？
它们导致什么制度后果？
它们为什么需要重新解释或改革？
```

### 2.5 单纯立场宣言

如果文本只是宣称“应加强监管”“应保护权利”“应维护公共利益”，而没有说明规则依据、规范前提、价值权衡和可执行方案，则不构成强学术论文。

### 2.6 没有规则、制度或规范问题的经验报告

例如只描述某地政策执行效果、公众态度或组织行为，如果没有进一步分析制度含义、规范评价或政策设计，不应套用本指南作为主范式。

### 2.7 直接替代法律意见的文本

学术论文可以分析法律问题，但不能把一般性学术讨论写成对具体个案的法律结论。涉及现实法律适用时，应说明司法辖区、时间、资料来源和不确定性。

## 3. 知识生产逻辑

法学 / 规范 / 政策分析型论文通过以下方式生产可信知识：

```text
现实争议 / 制度问题
→ 事实背景与制度语境
→ 规则、原则或政策框架
→ 概念界定
→ 规范冲突识别
→ 法律解释 / 制度分析 / 价值权衡
→ 反方观点
→ 回应与修正
→ 制度或政策方案
→ 可行性、后果和边界
```

### 3.1 这类论文依靠什么产生可信知识

| 可信性来源 | 写作中应体现什么 | Agent 判断重点 |
|---|---|---|
| 问题真实性 | 现实争议、制度困境、司法分歧、政策失败或价值冲突 | 问题是否具体 |
| 规则准确性 | 法律、政策、判例、制度文本、规范原则准确呈现 | 是否误读规则 |
| 概念清晰性 | 核心概念定义明确，如权利、风险、责任、公共利益 | 是否概念混用 |
| 规范前提透明 | 作者依据什么价值、原则或理论判断 | 是否隐藏价值立场 |
| 推理严密性 | 从事实、规则、价值到结论的链条清楚 | 是否逻辑跳跃 |
| 反方处理 | 公平呈现并回应强反对意见 | 是否稻草人化 |
| 比较与语境 | 比较法、比较政策或制度移植考虑语境差异 | 是否机械移植 |
| 可行性 | 建议能否执行、由谁执行、成本如何、风险如何 | 是否空泛建议 |
| 后果意识 | 方案可能的收益、副作用、分配影响和执行后果 | 是否只讲理想 |
| 边界限定 | 说明适用范围和未解决问题 | 是否过度推广 |

### 3.2 规范 / 政策论文与经验社会科学论文的区别

| 维度 | 规范 / 政策分析论文 | 经验社会科学论文 |
|---|---|---|
| 核心问题 | 应当如何评价、解释或设计制度 | 发生了什么、为何发生、有多大效应 |
| 证据类型 | 法条、判例、政策文本、原则、价值、制度后果 | 数据、访谈、观察、统计模型 |
| 推理方式 | 规范推理、规则解释、价值权衡、制度分析 | 描述、解释、因果识别、机制分析 |
| 结论形式 | 应然判断、解释立场、制度建议 | 经验发现、因果估计、机制解释 |
| 可信性来源 | 规则准确、价值前提清楚、推理严密、方案可行 | 测量、样本、识别、材料厚度 |
| 常见风险 | 立场先行、反驳不足、建议空泛 | 识别不足、测量差、理论弱 |

经验研究可以告诉我们：

```text
某政策是否有效；
公众如何看待某制度；
某规则执行中出现什么偏差；
某群体受到什么影响。
```

规范 / 政策分析进一步追问：

```text
即使有效，该政策是否正当？
即使多数人支持，该制度是否侵犯少数权利？
即使规则可执行，它是否合比例？
如果价值冲突不可避免，应如何排序和权衡？
```

### 3.3 核心推理链：事实—规则—价值—后果

强规范 / 政策论文应同时处理四个层面：

```text
事实：现实中发生了什么问题？
规则：现有法律、政策或制度如何规定？
价值：这些规则体现或冲突了哪些价值？
后果：不同方案将产生什么制度和社会影响？
```

示例：

```text
事实层：平台通过算法决定劳动者接单机会和收入。
规则层：现行劳动法对劳动关系认定仍依赖传统控制标准。
价值层：问题涉及劳动者保护、平台创新、市场灵活性和责任分配。
后果层：若直接认定全部劳动关系，可能提高保护但降低灵活性；若完全排除劳动法保护，则可能加剧风险外部化。
论文主张：应建立分层劳动保护制度，根据控制强度和经济依赖程度配置不同保护义务。
```

### 3.4 规范推理中的 Claim-Evidence-Warrant

| 元素 | 规范 / 政策论文中的对应 |
|---|---|
| Claim | 应然主张、解释结论、制度建议 |
| Evidence | 法条、判例、政策文件、制度事实、案例、比较材料 |
| Warrant | 规范原则、解释方法、价值权衡、制度逻辑 |
| Counterargument | 可能反对意见 |
| Boundary | 适用范围、执行条件、未解决问题 |

示例：

```text
Claim：高风险人工智能系统应承担更高的透明度义务。
Evidence：此类系统可能影响就业、信贷、教育和公共服务机会，且受影响者通常难以发现和质疑自动化决策。
Warrant：程序正义要求受影响者能够理解决策依据并获得有效救济；风险越高，透明度义务越强。
Counterargument：过高透明度可能泄露商业秘密或导致系统被规避。
Response：透明度义务可以分层设计，对监管机构、受影响个人和公众设置不同披露范围。
Boundary：该主张主要适用于对个人权利和重要利益有实质影响的系统，而非所有低风险自动化工具。
```

Agent 检查规则：

```text
每当作者写“应当”“必须”“有必要”“不合理”“不正当”“应建立”时，必须检查：
1. 事实问题是什么？
2. 现有规则或制度依据是什么？
3. 规范前提是什么？
4. 是否有价值冲突？
5. 是否回应反方观点？
6. 方案是否可执行？
7. 是否说明后果和边界？
```

## 4. 典型论文结构

法学 / 规范 / 政策分析型论文通常不完全采用 IMRaD，而更适合问题—框架—分析—反驳—方案的结构。

典型结构如下：

```text
Title
Abstract
Keywords

1. Introduction
2. Problem
3. Background / Institutional Context
4. Legal / Normative / Policy Framework
5. Analysis
6. Counterarguments
7. Proposal
8. Implications
9. Conclusion

References
Appendices, if needed
```

根据学科和研究类型，也可采用以下变体：

### 4.1 法学解释型结构

```text
1. Introduction
2. Legal Problem and Doctrinal Background
3. Competing Interpretations
4. Proposed Interpretation
5. Case / Statutory / Comparative Analysis
6. Objections and Responses
7. Implications
8. Conclusion
```

### 4.2 伦理分析型结构

```text
1. Introduction
2. Ethical Problem
3. Conceptual Clarification
4. Normative Framework
5. Analysis of Value Conflicts
6. Objections
7. Practical Recommendations
8. Conclusion
```

### 4.3 公共政策分析型结构

```text
1. Introduction
2. Policy Problem
3. Current Policy and Institutional Context
4. Evaluation Criteria
5. Policy Options
6. Comparative Assessment
7. Recommended Proposal
8. Implementation and Risks
9. Conclusion
```

### 4.4 比较制度型结构

```text
1. Introduction
2. Domestic Problem
3. Comparative Cases / Jurisdictions
4. Analytical Framework
5. Similarities and Differences
6. Lessons and Limits of Transfer
7. Institutional Proposal
8. Conclusion
```

### 4.5 科技治理型结构

```text
1. Introduction
2. Technology and Governance Problem
3. Existing Regulatory Landscape
4. Normative Values and Risk Framework
5. Analysis of Regulatory Gaps
6. Counterarguments
7. Governance Proposal
8. Implementation and Oversight
9. Conclusion
```

### 4.6 章节功能总览

| 章节 | 核心功能 | 不应写成 |
|---|---|---|
| Introduction | 提出现实争议、制度缺口、规范冲突和主张 | 泛泛时事评论 |
| Problem | 精确定义问题和争议对象 | 情绪化控诉 |
| Background / Context | 说明制度、法律、政策和事实背景 | 材料堆砌 |
| Framework | 给出法律、伦理、政策或制度分析标准 | 空泛原则罗列 |
| Analysis | 展开规则解释、价值权衡和制度分析 | 立场宣言 |
| Counterarguments | 公平呈现反方并回应 | 稻草人反驳 |
| Proposal | 提出可执行制度或政策方案 | 口号式建议 |
| Implications | 分析后果、适用范围和风险 | 重复建议 |
| Conclusion | 回到问题并压缩贡献 | 泛泛总结 |

## 5. 各章节写作指南

## 5.1 标题 Title

### 写作目标

标题应准确呈现论文的问题对象、规范冲突、制度领域或政策主张。

### 常见标题类型

| 类型 | 结构 | 示例形式 |
|---|---|---|
| 问题型 | X 的法律规制问题 | 算法决策中的程序正义问题 |
| 规范冲突型 | 在 X 与 Y 之间 | 隐私保护与公共安全之间 |
| 制度方案型 | 构建 X 制度 | 构建分层数据访问治理制度 |
| 解释型 | 重新理解 X | 重新理解平台责任 |
| 比较型 | X 制度的比较法启示 | 欧盟与美国数据治理模式比较 |
| 主副标题型 | 概念性主标题 + 说明性副标题 | “可解释的权力”：自动化决策中的透明度义务 |

### 常见错误

- 标题过大，如“人工智能法律问题研究”；
- 只写政策领域，不写具体问题；
- 标题像评论口号；
- 标题承诺解决全部问题；
- 标题使用“完善”“加强”“优化”但不说明对象；
- 标题不体现规范冲突或主张。

### Agent 质量判断

```text
1. 标题是否说明具体制度或规范问题？
2. 是否体现分析对象和范围？
3. 是否避免空泛政策词？
4. 是否与论文主张一致？
5. 是否能被目标学术共同体识别？
```

---

## 5.2 摘要 Abstract

### 写作目标

摘要应压缩规范 / 政策论证链：

```text
现实问题
→ 制度或规范缺口
→ 分析框架
→ 核心论证
→ 方案或结论
→ 贡献和边界
```

### 推荐结构

```text
1. 问题句：指出现实争议或制度困境。
2. 缺口句：说明现有规则、政策或理论解释不足。
3. 框架句：说明本文采用的法律、伦理或政策分析框架。
4. 论证句：概括核心分析和价值权衡。
5. 方案句：提出主要制度或政策建议。
6. 贡献句：说明本文推进了什么讨论。
7. 边界句：说明适用范围或限制。
```

### 示例骨架

```text
自动化决策正在影响信贷、就业和公共服务分配，但现有规则对受影响者的解释权和救济路径规定不足。既有研究多在透明度与商业秘密之间展开二元对立，较少区分不同风险场景和不同披露对象。本文基于程序正义和风险规制框架，论证高风险自动化决策应承担分层透明度义务。文章进一步区分监管透明、个人解释和公共问责三类机制，并回应商业秘密、合规成本和技术不可解释性三项反对意见。本文主张建立以风险等级、影响程度和救济需求为基础的分层披露制度。该方案主要适用于对个人重大权益产生实质影响的自动化决策场景。
```

### 常见错误

- 摘要只描述背景；
- 不说明论文主张；
- 不说明分析框架；
- 只说“提出建议”但不说明建议内容；
- 不说明规范冲突；
- 贡献写成“具有重要意义”；
- 把政策立场写成结论但无论证。

### Agent 质量判断

```text
1. 摘要是否提出具体问题？
2. 是否说明现有制度或理论缺口？
3. 是否有明确分析框架？
4. 是否概括核心论证？
5. 是否提出具体方案或结论？
6. 是否避免空泛立场表达？
```

## 5.3 Introduction 引言

### 写作目标

Introduction 应提出：

```text
现实争议
制度缺口
规范冲突
论文主张
论证路径
```

### 推荐组织方式

```text
1. 现实争议或典型案例；
2. 该争议暴露的制度问题；
3. 现有法律、政策或理论不足；
4. 核心规范冲突；
5. 本文主张；
6. 分析框架和材料；
7. 贡献；
8. 结构安排。
```

### 好的引言写法

弱写法：

```text
随着人工智能的发展，算法治理越来越重要，亟需加强监管。
```

强写法：

```text
自动化决策系统已被用于信贷、招聘和公共服务分配。问题并不只是算法是否准确，而是当系统作出不利决定时，受影响者是否能够理解决定依据、提出异议并获得有效救济。现有规则虽然强调透明度，但往往没有区分监管机构、受影响个人和公众所需的不同透明度类型，导致透明度义务在实践中要么过窄、要么过宽。
```

### 规范冲突写法

```text
该问题的难点在于，过低的披露要求会削弱程序正义和问责能力，而过高的披露要求又可能损害商业秘密、系统安全和行政效率。因此，问题不是是否需要透明度，而是如何设计与风险程度相匹配的透明度义务。
```

### Thesis Statement 写法

弱写法：

```text
本文认为应完善相关制度。
```

强写法：

```text
本文主张，应将算法透明度义务从单一的信息公开要求重构为分层问责制度：对监管机构强调可审计性，对受影响个人强调可理解解释和救济，对公众强调系统性风险披露。
```

### 常见错误

- 引言像新闻评论；
- 问题过大；
- 只说“制度不完善”；
- 没有规范冲突；
- 没有中心主张；
- 只提出政策建议，没有论证路径；
- 不说明本文与已有研究不同；
- 把事实判断、价值判断和政策建议混在一起。

### Agent 质量判断

```text
1. 是否提出具体争议？
2. 是否说明制度缺口？
3. 是否识别规范冲突？
4. 是否有清楚主张？
5. 是否说明分析框架？
6. 是否说明贡献？
7. 是否避免时评化表达？
```

## 5.4 Problem 问题界定

### 写作目标

Problem 部分必须精确定义论文要解决的问题。规范 / 政策论文常失败于问题过大或问题错位。

它应回答：

```text
问题到底是什么？
是规则缺失、规则冲突、解释分歧、执行失败、价值冲突还是制度后果不佳？
哪些问题本文不处理？
为什么这个问题值得规范分析？
```

### 问题类型

| 类型 | 含义 | 示例 |
|---|---|---|
| 规则缺失 | 现行规则没有覆盖新情境 | AI 生成内容责任 |
| 规则冲突 | 不同规则或原则相互冲突 | 隐私权与公共健康监测 |
| 解释分歧 | 同一规则有多种解释 | 平台是否构成雇主 |
| 执行困境 | 规则存在但难执行 | 数据主体权利行使成本高 |
| 制度错配 | 旧制度不能适应新实践 | 传统劳动法与平台劳动 |
| 价值冲突 | 多种价值难以同时满足 | 安全、自由、公平、效率 |
| 政策失灵 | 政策目标与实际后果不一致 | 过度监管导致地下化 |
| 责任分配问题 | 多方主体责任不清 | 自动驾驶事故责任 |
| 风险治理问题 | 不确定风险如何治理 | 生物技术、AI、安全风险 |

### 问题界定写法

弱写法：

```text
现有制度存在不足。
```

强写法：

```text
现有制度的不足并非完全缺少规则，而是规则将责任建立在可识别的人类决策者之上。当自动化系统在多主体协作中生成决策时，传统责任结构难以确定谁应承担解释、预防和救济义务。
```

### 常见错误

- 问题过大，如“数字经济监管问题”；
- 把多个问题混在一起；
- 只说“不完善”；
- 问题和后文方案不对应；
- 没有区分法律问题、伦理问题和政策问题；
- 问题定义中已经预设结论；
- 忽略受影响主体。

### Agent 质量判断

```text
1. 问题是否具体？
2. 是否区分事实问题、规则问题和价值问题？
3. 是否说明问题为何重要？
4. 是否限定范围？
5. 后文分析和方案是否回应该问题？
```

## 5.5 Background / Institutional Context 背景与制度语境

### 写作目标

Background / Institutional Context 说明问题产生的制度、法律、技术、社会或政策背景。背景不是信息堆砌，而是为规范分析提供必要事实和制度语境。

### 应包含内容

- 现行法律或政策框架；
- 制度运行方式；
- 相关机构和主体；
- 典型案例；
- 技术或社会实践；
- 争议发生场景；
- 数据或事实背景；
- 历史沿革；
- 执行机制；
- 制度后果。

### 背景写法

弱写法：

```text
近年来我国出台了许多人工智能政策。
```

强写法：

```text
现有人工智能治理文件主要通过风险分类、备案、算法透明和安全评估来配置监管义务。然而，这些规则更多针对系统提供者的合规义务，而较少处理受影响个人在具体不利决定中如何获得解释和救济的问题。
```

### 制度语境应回答

```text
1. 现有制度如何运作？
2. 哪些主体参与？
3. 哪些规则适用？
4. 实践中出现什么缺口？
5. 该缺口如何引发规范冲突？
```

### 常见错误

- 背景太长，像资料汇编；
- 背景与主张无关；
- 只列政策文件，不分析制度逻辑；
- 不区分法律效力层级；
- 不说明规则适用范围；
- 不说明实践运行；
- 忽略机构能力和执行条件。

### Agent 质量判断

```text
1. 背景是否服务问题界定？
2. 是否说明现行制度如何运行？
3. 是否准确呈现规则和主体？
4. 是否说明制度缺口如何产生？
5. 是否避免资料堆砌？
```

---

## 5.6 Legal / Normative / Policy Framework 法律、规范或政策框架

### 写作目标

Framework 部分给出分析标准。它应说明作者依据什么规则、原则、价值或政策标准进行判断。

没有框架，论文容易变成立场表达。

### 框架类型

| 框架类型 | 核心内容 | 适用 |
|---|---|---|
| 法律解释框架 | 文义、体系、目的、历史、合宪性解释等 | 法条和判例分析 |
| 权利框架 | 权利范围、限制条件、救济方式 | 宪法、隐私、平等 |
| 比例原则 | 正当目的、适当性、必要性、均衡性 | 权利限制和监管 |
| 程序正义 | 通知、参与、解释、听证、救济 | 行政、算法治理 |
| 风险规制 | 风险等级、预防原则、责任配置 | 科技治理、环境 |
| 成本收益分析 | 成本、收益、替代方案、分配效应 | 公共政策 |
| 公平 / 正义理论 | 分配正义、承认正义、程序正义 | 社会政策、伦理 |
| 功利主义 | 后果最大化、总福利 | 政策选择 |
| 义务论 | 权利、尊严、不可工具化 | 伦理和人权 |
| 德性 / 照护伦理 | 关系、责任、脆弱性 | 医疗、照护、AI伦理 |
| 制度分析 | 激励、执行、责任、合规成本 | 治理方案设计 |
| 比较法框架 | 功能比较、制度语境、移植限制 | 比较制度 |

### 框架写法

弱写法：

```text
本文从公平、效率和安全角度分析。
```

强写法：

```text
本文采用程序正义框架分析自动化决策透明度。该框架包含三项要求：第一，受影响者应知道决定已由自动化系统作出；第二，受影响者应获得足以理解不利决定依据的解释；第三，受影响者应能通过有效程序提出异议。该框架不同于一般信息公开要求，因为其重点不是最大化披露量，而是保障受影响者参与和救济能力。
```

### 框架应避免

- 空泛列举价值；
- 同时使用多个理论但不说明关系；
- 用价值词替代分析标准；
- 框架与后文分析脱节；
- 只说“本文采用法经济学视角”但不说明评价标准；
- 只引用理论名词，不说明如何适用。

### Agent 质量判断

```text
1. 框架是否明确？
2. 框架是否适合问题？
3. 关键概念是否定义？
4. 分析标准是否可操作？
5. 框架是否会在后文实际使用？
6. 是否避免原则罗列？
```

## 5.7 概念界定、规则解释、案例分析、比较分析和价值权衡

### 5.7.1 概念界定

概念界定用于避免论证混乱。常见需要界定的概念包括：

- 权利；
- 义务；
- 责任；
- 公共利益；
- 风险；
- 安全；
- 公平；
- 歧视；
- 自主；
- 尊严；
- 合理期待；
- 透明度；
- 问责；
- 合规；
- 平台；
- 算法决策；
- 个人信息；
- 公共数据；
- 弱势群体；
- 比例原则；
- 必要性；
- 正当程序。

概念界定写法：

```text
本文所称“透明度”并非泛指所有信息公开，而是指受影响主体、监管机构或公众能够理解、审查或质疑决策过程的制度能力。因此，透明度至少包含通知、解释和可审计性三个层次。
```

### 5.7.2 规则解释

法律论文常用解释方法：

| 方法 | 问题 |
|---|---|
| 文义解释 | 条文通常含义是什么 |
| 体系解释 | 该规则在法律体系中如何定位 |
| 目的解释 | 规则要实现什么目的 |
| 历史解释 | 立法背景和沿革是什么 |
| 合宪性解释 | 如何避免与上位法冲突 |
| 功能解释 | 规则在制度中发挥什么功能 |
| 比较解释 | 其他法域如何处理类似问题 |

规则解释应说明：

```text
1. 被解释的规则是什么；
2. 争议点在哪里；
3. 可能解释有哪些；
4. 作者为何选择某解释；
5. 该解释的制度后果是什么。
```

### 5.7.3 判例 / 案例分析

判例或案例不能只是列举。应分析：

- 案件事实；
- 争点；
- 裁判理由；
- 规则提炼；
- 与本文问题的关系；
- 判例之间是否一致；
- 案例的适用边界；
- 是否有反例或不同裁判路径。

弱写法：

```text
某案中法院支持了原告，因此说明应保护该权利。
```

强写法：

```text
某案的关键并不在于法院支持原告，而在于法院将平台的推荐排序视为影响交易机会的实质性控制。这一裁判理由使传统合同关系分析转向控制能力分析，为平台责任配置提供了新的解释路径。
```

### 5.7.4 比较分析

比较法或比较政策应避免机械移植。

比较分析应说明：

```text
1. 为什么选择这些比较对象？
2. 它们面对的问题是否相同？
3. 制度背景有何差异？
4. 哪些规则可借鉴？
5. 哪些条件不可移植？
6. 对本文问题有什么启示？
```

弱写法：

```text
我国应借鉴欧盟经验。
```

强写法：

```text
欧盟的高风险分类框架可为我国提供风险分层思路，但其执行依赖较强的合规评估和监管机构能力。若直接移植，可能产生形式合规而非实质问责。因此，本文主张借鉴风险分层逻辑，而非复制完整合规程序。
```

### 5.7.5 价值权衡

规范 / 政策论文常涉及价值冲突：

- 自由与安全；
- 隐私与公共利益；
- 公平与效率；
- 创新与风险；
- 个体权利与集体福利；
- 商业秘密与透明问责；
- 国家安全与开放流动；
- 专业自治与公众监督；
- 平等保护与差异化治理。

价值权衡应避免简单宣称某价值优先，而应说明：

```text
1. 价值冲突是什么；
2. 为什么无法同时完全实现；
3. 在何种情境下哪个价值权重更高；
4. 是否有折中或分层方案；
5. 权衡标准是什么；
6. 该权衡会带来什么代价。
```

示例：

```text
在低风险自动化推荐场景中，过高解释义务可能带来不成比例的合规成本。但在影响就业、信贷或公共服务的高风险场景中，解释义务直接关系到个人获得救济的能力，因此程序正义的权重应高于一般商业保密利益。由此，透明度义务应采用风险分层而非一刀切模式。
```

## 5.8 Analysis 分析主体

### 写作目标

Analysis 是规范 / 政策论文的核心。它应将事实、规则、价值和后果连接起来，逐步支持中心主张。

### 推荐组织方式

#### 方式一：按规范冲突组织

```text
5.1 权利保护不足
5.2 公共利益和监管需求
5.3 商业秘密与执行成本
5.4 分层权衡方案
```

#### 方式二：按规则解释组织

```text
5.1 文义解释的限制
5.2 目的解释的支持
5.3 体系解释中的冲突
5.4 本文解释的制度后果
```

#### 方式三：按制度功能组织

```text
5.1 谁应承担义务
5.2 义务内容是什么
5.3 如何执行和监督
5.4 救济如何配置
```

#### 方式四：按政策选项组织

```text
5.1 选项一：维持现状
5.2 选项二：全面监管
5.3 选项三：风险分层监管
5.4 比较与选择
```

#### 方式五：按案例 / 法域比较组织

```text
5.1 国内制度现状
5.2 欧盟方案
5.3 美国方案
5.4 比较启示和移植边界
```

### 分析段落结构

推荐段落结构：

```text
1. 子主张；
2. 规则 / 案例 / 政策事实；
3. 规范解释或制度分析；
4. 后果评估；
5. 回到中心主张。
```

示例：

```text
高风险自动化决策中的解释义务应优先保障受影响者的救济能力，而不是追求完全公开算法细节。原因在于，程序正义关注的是个人能否理解并质疑对其不利的决定。即便完整模型参数被公开，普通个人也未必能够据此形成有效异议。相反，围绕关键因素、错误纠正路径和人工复核机制设计解释义务，更能实现程序正义目标。
```

### 常见错误

- 只表达立场，没有推理；
- 只列法条，没有解释；
- 只讲原则，没有制度后果；
- 分析中事实、规范和建议混杂；
- 每节都重复“应加强监管”；
- 不回应价值冲突；
- 不说明规则适用条件；
- 方案突然出现，前文没有铺垫。

### Agent 质量判断

```text
1. 每节是否有子主张？
2. 子主张是否支持中心主张？
3. 是否使用规则、案例或制度材料？
4. 是否解释材料如何支持规范判断？
5. 是否考虑后果和反方？
6. 分析是否推进而非重复？
```

## 5.9 Counterarguments 反方观点与回应

### 写作目标

规范 / 政策论文必须认真处理反方。反方不是形式化段落，而是检验作者论证强度的关键。

### 常见反方类型

| 反方类型 | 示例 |
|---|---|
| 权利反方 | 方案侵犯隐私、表达自由、财产权等 |
| 效率反方 | 方案成本过高、降低创新 |
| 可行性反方 | 监管机构能力不足，执行困难 |
| 法律权限反方 | 超越现行法律授权或违背上位法 |
| 证据反方 | 事实基础不足，不足以支持改革 |
| 替代方案反方 | 有更温和或更有效方案 |
| 滑坡反方 | 方案会扩张为过度监管 |
| 平等反方 | 方案可能产生不公平分配影响 |
| 制度移植反方 | 外国经验不适合本土语境 |
| 技术反方 | 技术上不可解释、不可执行 |

### 反方回应结构

```text
1. 公平呈现反方观点；
2. 承认其合理部分；
3. 指出其忽略或夸大的地方；
4. 用规则、价值或后果分析回应；
5. 必要时修正方案。
```

### 好的反方写法

弱写法：

```text
有人认为监管会阻碍创新，但这种观点是不正确的。
```

强写法：

```text
反对者可能认为，强透明度义务会增加企业合规成本并削弱技术创新。这一担忧并非没有根据，尤其是在低风险应用中，一刀切披露可能造成不成比例负担。本文的回应不是否认成本，而是主张通过风险分层降低过度监管风险：低风险系统只需承担基本通知义务，高风险系统才承担更高解释和审计义务。
```

### 常见错误

- 把反方写得很弱；
- 不回应最强反方；
- 只用价值口号反驳；
- 不承认反方合理性；
- 回应后不修正方案；
- 忽略执行成本和制度能力；
- 将不同立场道德化。

### Agent 质量判断

```text
1. 是否识别最强反方？
2. 是否公平呈现？
3. 是否回应其核心担忧？
4. 是否使用规则、价值或后果分析回应？
5. 是否必要时调整主张或方案？
```

---

## 5.10 Proposal 制度、法律或政策方案

### 写作目标

Proposal 部分应提出可执行、可评估、与前文分析一致的方案。它不是口号式建议。

好的方案应回答：

```text
方案目标是什么？
适用于哪些对象？
由谁执行？
包含哪些义务或权利？
程序如何运行？
如何监督和救济？
成本和风险是什么？
如何分阶段实施？
```

### 方案类型

| 类型 | 示例 |
|---|---|
| 规则解释方案 | 对现有法条作特定解释 |
| 立法方案 | 新增或修改规则 |
| 监管方案 | 建立备案、审计、许可、评估机制 |
| 程序方案 | 通知、听证、申诉、复核、救济 |
| 责任配置方案 | 分配平台、开发者、使用者、监管者责任 |
| 分层治理方案 | 按风险、规模、影响程度配置义务 |
| 协同治理方案 | 政府、行业、平台、公众共同治理 |
| 软法方案 | 指引、标准、伦理准则、行业规范 |
| 机构改革方案 | 新设机构、明确权限、协调机制 |
| 政策工具组合 | 补贴、税收、信息披露、监管沙盒等 |

### 可行方案写法

弱写法：

```text
应完善相关法律法规，加强监管。
```

强写法：

```text
本文建议建立分层解释义务制度。第一，对所有自动化决策系统，应要求基本通知，即告知个人其受到自动化处理影响。第二，对高风险系统，应要求个体化解释，包括主要影响因素、数据来源类型、错误纠正路径和人工复核渠道。第三，对系统性影响较大的平台，应向监管机构提交可审计文档，包括模型目的、训练数据类别、风险评估和纠偏措施。该制度由行业监管机构负责监督，并通过行政处罚、民事救济和合规审计共同执行。
```

### 方案可行性维度

| 维度 | 检查问题 |
|---|---|
| 合法性 | 是否与上位法、权利保护和权限结构一致 |
| 正当性 | 是否符合规范原则和价值目标 |
| 比例性 | 手段是否过度，是否有更温和替代 |
| 可执行性 | 由谁执行，能力是否足够 |
| 可监督性 | 如何检查执行 |
| 成本 | 合规成本、行政成本、社会成本 |
| 分配影响 | 谁受益，谁承担成本 |
| 激励相容 | 相关主体是否有执行动机 |
| 风险 | 是否产生规避、形式合规或副作用 |
| 可扩展性 | 是否适用于不同场景 |
| 渐进性 | 是否可分阶段实施 |

### 常见错误

- 建议太空泛；
- 没有责任主体；
- 没有实施路径；
- 没有考虑成本；
- 不回应反方；
- 与前文分析不一致；
- 方案超出法律权限；
- 直接照搬域外制度；
- 方案只表达价值，不解决制度问题。

### Agent 质量判断

```text
1. 方案是否具体？
2. 是否回应前文问题？
3. 是否说明适用对象和责任主体？
4. 是否说明执行程序？
5. 是否考虑成本和副作用？
6. 是否回应反方担忧？
7. 是否有边界和实施条件？
```

## 5.11 Implications 影响与边界

### 写作目标

Implications 部分说明方案或论点对法律、政策、治理、伦理、实践和未来研究的影响。

它应回答：

```text
如果接受本文主张，现有制度需要如何改变？
该方案会影响哪些主体？
可能产生什么风险？
哪些问题仍未解决？
```

### 可讨论内容

- 对立法的影响；
- 对司法解释的影响；
- 对监管机构的影响；
- 对企业合规的影响；
- 对个人权利和救济的影响；
- 对行业发展的影响；
- 对公共利益的影响；
- 对弱势群体的影响；
- 对国际规则协调的影响；
- 对未来研究的影响。

### 影响写法

弱写法：

```text
本文对实践具有重要启示。
```

强写法：

```text
该方案将平台透明度义务从单一披露转向程序性问责。这意味着监管重点不应只检查平台是否公开算法规则，而应检查受影响个人是否能够识别不利决定、获得可理解解释并启动有效复核程序。
```

### 边界写法

```text
本文方案主要面向对个人重要权益产生实质影响的自动化决策。对于低风险内容推荐或个性化展示场景，完整解释义务可能不具备比例性，基本通知和退出机制可能更适当。
```

### 常见错误

- 影响分析重复建议；
- 不讨论副作用；
- 不说明适用范围；
- 不承认制度能力限制；
- 不说明未来研究；
- 将理论方案写成已经可直接实施的政策。

### Agent 质量判断

```text
1. 是否说明方案影响？
2. 是否讨论风险和副作用？
3. 是否说明适用边界？
4. 是否提出合理未来问题？
5. 是否避免夸大政策效果？
```

## 5.12 Conclusion 结论

### 写作目标

Conclusion 应回到问题，压缩论证，明确主张和边界。

### 推荐结构

```text
1. 回到现实争议；
2. 重申制度或规范问题；
3. 概括核心分析；
4. 重申方案；
5. 说明贡献；
6. 限定边界。
```

### 示例

```text
本文围绕高风险自动化决策中的透明度义务展开。文章指出，现有讨论将透明度简单理解为信息公开，忽略了受影响个人、监管机构和公众在问责机制中的不同需求。基于程序正义和风险规制框架，本文论证透明度义务应当分层配置：基本通知保障知情，个体化解释保障救济，监管审计保障系统性问责。该方案试图在权利保护、商业秘密和监管成本之间建立可辩护平衡。本文讨论主要适用于对个人重大权益产生实质影响的自动化决策场景，对低风险系统仍需适用更轻量的治理工具。
```

### 常见错误

- 只重复“应完善制度”；
- 不回到规范冲突；
- 不总结推理链；
- 不说明方案；
- 过度声称解决所有问题；
- 引入新案例或新理论；
- 以口号结尾。

### Agent 质量判断

```text
1. 是否回答论文问题？
2. 是否重申中心主张？
3. 是否概括事实—规则—价值—后果链条？
4. 是否说明方案和贡献？
5. 是否限定适用范围？
```

## 6. 常见贡献类型

### 6.1 法律解释贡献

对现有法条、判例、原则或制度作出更清晰或更有说服力的解释。

示例：

```text
本文主张将平台控制能力纳入劳动关系认定中的从属性分析，从而弥补传统人身控制标准在算法管理场景中的不足。
```

质量要求：

- 解释对象明确；
- 解释方法清楚；
- 与既有解释比较；
- 说明制度后果；
- 处理反对解释。

### 6.2 规范理论贡献

提出或修正某个规范原则、价值排序或伦理框架。

示例：

```text
本文将算法透明度从信息公开原则重构为程序正义原则下的可救济性要求。
```

质量要求：

- 规范前提透明；
- 价值冲突处理清楚；
- 能解释具体制度问题；
- 不只是抽象道德宣言。

### 6.3 制度分析贡献

揭示某制度结构为何导致特定问题，或解释制度功能如何失衡。

示例：

```text
本文说明现行数据授权制度的问题不在于同意形式缺失，而在于个体同意无法应对数据的群体性外部效应。
```

质量要求：

- 制度运行机制清楚；
- 说明因果或功能逻辑；
- 结合规则和实践；
- 提出制度修正方向。

### 6.4 政策设计贡献

提出具体、可执行、可评估的政策方案。

示例：

```text
本文提出以风险等级、影响范围和可逆性为基础的分层监管方案。
```

质量要求：

- 目标明确；
- 工具具体；
- 责任主体清楚；
- 成本和风险讨论充分；
- 可实施路径清楚。

### 6.5 比较法 / 比较政策贡献

通过比较不同法域或制度，提炼可借鉴但不机械移植的经验。

质量要求：

- 比较对象选择有理由；
- 比较维度明确；
- 考虑制度语境差异；
- 说明移植边界；
- 提炼功能性启示。

### 6.6 批判贡献

揭示某规则、政策或制度背后的权力、排除、偏见或不平等结构。

质量要求：

- 批判对象清楚；
- 证据充分；
- 不只是价值表态；
- 提出规范标准；
- 说明替代方向。

### 6.7 责任配置贡献

重新分配不同主体的法律、伦理或治理责任。

示例：

```text
本文主张自动化决策责任不应只归于最终使用者，而应根据开发、部署、数据控制和风险预防能力在多主体之间分配。
```

质量要求：

- 主体角色清楚；
- 责任类型区分；
- 分配标准合理；
- 执行机制明确。

### 6.8 价值权衡贡献

在冲突价值之间提出更细致的排序或分层方案。

示例：

```text
本文主张隐私与公共健康之间不应作抽象二分，而应根据数据敏感性、识别风险、公共利益强度和退出机制进行分层权衡。
```

质量要求：

- 冲突价值清楚；
- 权衡标准可操作；
- 处理反方；
- 说明适用边界。

## 7. 证据与论证标准

## 7.1 强规范 / 政策论证的特征

```text
1. 问题具体；
2. 事实背景准确；
3. 规则和制度描述准确；
4. 概念界定清楚；
5. 规范前提透明；
6. 价值冲突被正面处理；
7. 推理链条可追踪；
8. 反方观点被公平回应；
9. 方案具体且可执行；
10. 成本、副作用和边界得到讨论。
```

### 7.2 弱规范 / 政策论证的特征

弱论证常表现为：

- 只有立场，没有推理；
- 只说“应加强监管”；
- 不说明现有规则；
- 不说明制度缺口；
- 概念混乱；
- 把事实判断直接跳到政策建议；
- 不处理反方；
- 价值排序不透明；
- 建议不可执行；
- 只借鉴域外制度，不讨论本土语境；
- 过度依赖个案；
- 忽略成本和副作用；
- 没有边界。

### 7.3 描述性、规范性和政策性主张的区分

| 主张类型 | 问题 | 示例 | 需要的支持 |
|---|---|---|---|
| 描述性 | 是什么 | 现行规则缺少解释权规定 | 法条、政策、案例 |
| 解释性 | 为什么 | 缺口源于传统责任结构 | 制度分析 |
| 规范性 | 应当如何评价 | 该缺口损害程序正义 | 价值和原则 |
| 政策性 | 应当如何行动 | 应建立分层解释义务 | 可行方案和后果 |
| 预测性 | 可能会怎样 | 过度披露可能增加合规成本 | 经验、逻辑、比较 |
| 比较性 | 何者更好 | 风险分层优于一刀切 | 标准和权衡 |

Agent 判断规则：

```text
不同类型主张需要不同类型支持。
不能用一个典型案例直接推出制度方案。
不能用价值判断替代事实证明。
不能用政策偏好替代规范推理。
```

### 7.4 价值权衡强度判断

强价值权衡应说明：

```text
1. 哪些价值冲突；
2. 冲突为何无法完全避免；
3. 具体场景中价值权重如何变化；
4. 选择某方案的理由；
5. 该方案牺牲了什么；
6. 为什么牺牲是可接受或合比例的；
7. 是否有更温和替代方案。
```

## 8. 文献综述写法

### 8.1 文献综述的功能

法学 / 规范 / 政策论文的文献综述应建立“争论地图”，而不是研究史罗列。

它应回答：

```text
已有研究如何界定问题？
有哪些主要立场？
争议焦点是什么？
已有方案为何不足？
本文如何介入？
```

### 8.2 组织方式

#### 方式一：立场式

```text
强监管立场
市场自治立场
协同治理立场
本文分层治理立场
```

#### 方式二：价值冲突式

```text
隐私保护研究
公共利益研究
平衡机制研究
本文提出新的权衡标准
```

#### 方式三：制度工具式

```text
许可制
备案制
透明度义务
审计制度
本文组合方案
```

#### 方式四：解释路径式

```text
文义解释
目的解释
功能解释
本文重构解释
```

### 8.3 文献综述段落模板

```text
关于算法透明度，既有研究主要形成两种路径。第一种路径强调信息公开，认为披露模型逻辑能够提升问责。第二种路径则强调商业秘密和技术复杂性，认为过度透明不现实。两种路径都抓住了问题的一部分，但都将透明度理解为披露量问题。本文认为，应将透明度重新理解为程序性救济能力问题，由此区分通知、解释和审计三类义务。
```

### 8.4 常见错误

- 逐篇罗列；
- 不区分立场；
- 只引用支持自己观点的文献；
- 不说明争议焦点；
- 不处理强反方；
- 文献综述与主张脱节；
- 只说“研究不足”，不说明不足在哪里；
- 不说明本文位置。

### Agent 质量判断

```text
1. 文献综述是否呈现主要立场？
2. 是否说明争议焦点？
3. 是否处理反方文献？
4. 是否自然推出本文主张？
5. 是否避免研究史堆砌？
```

## 9. 结果、分析或论证主体的组织方式

规范 / 政策论文主体应按照推理链组织，而不是按资料来源堆砌。

### 9.1 推荐主体结构

```text
5.1 现行规则的解释限度
5.2 规范价值冲突的真正结构
5.3 既有方案为何不足
5.4 分层制度方案的必要性
5.5 反方观点与回应
```

### 9.2 段落论证结构

每段应包含：

```text
主张
→ 规则 / 事实 / 案例
→ 解释或规范推理
→ 后果或边界
→ 与主论点连接
```

### 9.3 论证链检查

Agent 应尝试将论文压缩为：

```text
由于现行制度在……方面存在缺口，
而该缺口导致……规范冲突，
本文基于……框架认为……
因此应当……
该方案相较于……更能……
同时通过……回应……反对意见。
```

如果无法压缩，说明论文可能存在：

- 问题不清；
- 主张不清；
- 框架不清；
- 分析散乱；
- 建议与分析脱节；
- 反方缺失。

### 9.4 常见主体失败

| 失败模式 | 表现 | 修正 |
|---|---|---|
| 法条堆砌 | 列很多规则但不分析 | 提炼制度逻辑和冲突 |
| 立场宣言 | 反复说应保护 / 应监管 | 加入规范前提和推理 |
| 方案突兀 | 最后突然提出建议 | 让分析逐步推出方案 |
| 反方缺失 | 只讲自己立场 | 增加强反方 |
| 比较机械 | 直接照搬外国制度 | 分析语境和移植条件 |
| 建议空泛 | 完善法律、加强监管 | 具体化主体、程序、责任 |

## 10. 讨论与结论写法

### 10.1 Discussion 的功能

Discussion 应说明本文主张对法律解释、制度设计、政策实践或理论争论有什么影响。

它应回答：

```text
如果接受本文论证，现有制度理解要如何改变？
政策设计要如何调整？
该方案有什么风险和边界？
它对未来研究有什么启示？
```

### 10.2 从分析到贡献的提升

规则层：

```text
现行规则没有明确个体化解释义务。
```

规范层：

```text
这削弱了受影响者获得救济的程序能力。
```

制度层：

```text
透明度应从信息披露转向程序性问责。
```

方案层：

```text
应建立分层解释义务制度。
```

贡献层：

```text
本文将算法透明度讨论从“公开多少信息”推进到“如何配置问责能力”的制度问题。
```

### 10.3 Conclusion 写法

结论应回到：

```text
问题
规范冲突
分析框架
核心主张
方案
边界
```

不要只写：

```text
综上，应完善相关制度。
```

应写：

```text
综上，本文认为……。这一结论基于……。本文提出……。该方案的优势在于……，但其适用范围限于……。
```

## 11. 审稿人关注点

### 11.1 问题是否清楚

审稿人会问：

```text
1. 论文到底解决什么法律、伦理或政策问题？
2. 问题是否过大？
3. 是否只是时事评论？
4. 是否有明确制度缺口？
```

### 11.2 规范框架是否准确

审稿人会问：

```text
1. 作者使用的法律或伦理框架是否准确？
2. 概念是否清楚？
3. 是否混淆不同原则？
4. 是否误读法律规则或政策文件？
```

### 11.3 推理是否跳跃

审稿人会问：

```text
1. 是否从事实直接跳到建议？
2. 是否说明价值前提？
3. 是否处理规则适用条件？
4. 是否有逻辑断裂？
```

### 11.4 反驳是否充分

审稿人会问：

```text
1. 是否处理最强反方？
2. 是否公平呈现反对意见？
3. 是否回应成本、权限、可行性？
4. 是否只是稻草人反驳？
```

### 11.5 建议是否可行

审稿人会问：

```text
1. 建议是否具体？
2. 谁执行？
3. 成本如何？
4. 是否有法律授权？
5. 是否有副作用？
6. 是否能在现有制度中落地？
```

### 11.6 贡献是否明确

审稿人会问：

```text
1. 论文相对已有研究新在哪里？
2. 是新解释、新框架、新方案还是新批判？
3. 是否只是常见政策建议？
4. 是否进入学术争论？
```

## 12. 常见失败模式

### 12.1 写成评论文章

表现：

- 语言情绪化；
- 立场鲜明但论证薄弱；
- 不处理反方；
- 没有规则和制度分析。

修正：

```text
1. 明确问题类型；
2. 补充法律或政策框架；
3. 将价值判断转化为可论证命题；
4. 增加强反方回应。
```

### 12.2 写成立场宣言

表现：

```text
必须加强监管。
必须保护权利。
必须坚持公平原则。
```

修正：

```text
1. 说明为什么必须；
2. 说明依据什么原则；
3. 说明与其他价值如何权衡；
4. 说明具体制度方案。
```

### 12.3 政策建议泛泛

表现：

```text
完善法律法规，强化监管责任，提升公众意识。
```

修正：

```text
1. 指定修改哪类规则；
2. 指定责任主体；
3. 指定程序和工具；
4. 指定监督和救济；
5. 说明成本和风险。
```

### 12.4 事实、规范和建议混乱

修正：

```text
1. 标注哪些是事实判断；
2. 哪些是规则解释；
3. 哪些是价值判断；
4. 哪些是政策建议；
5. 重新排列推理链。
```

### 12.5 规范框架空泛

表现：

- 公平、效率、安全、自由都提到；
- 不说明如何排序；
- 不说明适用标准。

修正：

```text
1. 选择主框架；
2. 定义核心原则；
3. 说明权衡标准；
4. 在分析中实际使用。
```

### 12.6 比较法机械移植

修正：

```text
1. 说明为什么选择该法域；
2. 比较制度背景；
3. 只提炼功能性启示；
4. 说明不可移植条件。
```

### 12.7 不回应反方

修正：

```text
1. 找出最强反方；
2. 公平呈现；
3. 承认合理部分；
4. 修改方案以回应担忧。
```

### 12.8 结论过度

表现：

- 一个制度方案被说成彻底解决问题；
- 忽略执行困难；
- 不说明边界。

修正：

```text
1. 限定适用范围；
2. 承认方案代价；
3. 说明需要配套条件；
4. 提出渐进实施。
```

## 13. Agent 写作辅助策略

### 13.1 Agent 总体流程

Agent 辅助法学 / 规范 / 政策论文时，应按以下流程诊断：

```text
1. 识别问题类型；
2. 区分事实、规则、价值和政策建议；
3. 提取中心主张；
4. 检查法律 / 伦理 / 政策框架；
5. 检查概念定义；
6. 检查规则解释和制度分析；
7. 检查价值冲突和权衡；
8. 检查反方观点；
9. 检查政策方案可行性；
10. 检查结论是否过度；
11. 重构推理链。
```

### 13.2 Agent 如何检查论证链

生成“事实—规则—价值—后果—方案”矩阵：

| 环节 | 当前文本内容 | 是否充分 | 问题 | 修改建议 |
|---|---|---|---|---|
| 事实 |  |  |  |  |
| 规则 |  |  |  |  |
| 价值 |  |  |  |  |
| 后果 |  |  |  |  |
| 方案 |  |  |  |  |
| 反方 |  |  |  |  |

### 13.3 Agent 如何检查规范前提

Agent 应追问：

```text
1. 作者依据什么价值作出判断？
2. 该价值是否被定义？
3. 是否与其他价值冲突？
4. 作者如何排序或权衡？
5. 这种排序是否需要辩护？
6. 是否存在不同理论框架会得出不同结论？
```

### 13.4 Agent 如何检查政策可行性

Agent 应检查方案是否回答：

```text
1. 谁执行？
2. 执行什么？
3. 依据什么权限？
4. 采用什么程序？
5. 如何监督？
6. 如何救济？
7. 成本由谁承担？
8. 可能被如何规避？
9. 是否有副作用？
10. 是否需要分阶段实施？
```

### 13.5 Agent 如何区分描述性分析与规范性主张

Agent 应将句子分类：

| 原句 | 类型 | 是否需要支持 |
|---|---|---|
| 现行规则未规定 X | 描述性 | 法条 / 政策依据 |
| 该缺失削弱程序正义 | 规范性 | 程序正义框架 |
| 应建立 X 制度 | 政策性 | 可行性和权衡 |
| X 可能增加成本 | 预测性 | 经验或制度逻辑 |

### 13.6 Agent 如何降级过度政策主张

| 过强表达 | 更稳妥表达 |
|---|---|
| 必须全面禁止 | 在高风险场景中应考虑限制或严格许可 |
| 应立即建立完整制度 | 可先在高风险领域试点分层制度 |
| 该方案能解决问题 | 该方案可缓解……，但仍需……配套 |
| 国外经验可直接借鉴 | 可借鉴其功能逻辑，但需结合本土制度条件 |
| 监管越强越好 | 监管强度应与风险和权利影响程度相匹配 |

## 14. 检查清单

## 14.1 全文总检清单

```markdown
## 问题
- [ ] 是否提出具体现实争议？
- [ ] 是否说明制度缺口？
- [ ] 是否区分事实问题、规则问题、价值问题和政策问题？
- [ ] 是否限定研究范围？
- [ ] 是否避免泛泛“应完善制度”？

## 框架
- [ ] 是否有法律 / 伦理 / 政策分析框架？
- [ ] 核心概念是否界定？
- [ ] 规范前提是否透明？
- [ ] 价值冲突是否明确？
- [ ] 框架是否在后文实际使用？

## 规则与制度
- [ ] 现行规则是否准确呈现？
- [ ] 法律效力层级是否清楚？
- [ ] 判例或案例是否被分析而非罗列？
- [ ] 制度运行机制是否说明？
- [ ] 比较制度是否考虑语境差异？

## 论证
- [ ] 是否建立事实—规则—价值—后果推理链？
- [ ] 是否存在逻辑跳跃？
- [ ] 是否处理替代解释或反方观点？
- [ ] 是否解释价值权衡？
- [ ] 是否说明结论边界？

## 方案
- [ ] 政策或制度建议是否具体？
- [ ] 是否说明责任主体？
- [ ] 是否说明执行程序？
- [ ] 是否说明监督和救济？
- [ ] 是否评估成本和副作用？
- [ ] 是否说明可行性和实施条件？

## 写作
- [ ] 是否避免评论化语言？
- [ ] 是否避免口号式建议？
- [ ] 是否避免立场先行？
- [ ] 是否区分描述性和规范性表达？
- [ ] 是否回到学术贡献？
```
