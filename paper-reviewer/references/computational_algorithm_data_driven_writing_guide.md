# 计算 / 算法 / 数据驱动型论文写作指南

## 1. 适用范围

本指南适用于以**算法、模型、数据集、benchmark、计算流程、实验协议或数据驱动分析方法**为核心贡献的科研论文。其目标是帮助 Agent 在辅助用户撰写、重构、评审和润色此类论文时，判断论文是否形成了清晰的计算研究论证链。

适用领域包括但不限于：

- 计算机科学；
- 人工智能；
- 机器学习；
- 深度学习；
- 数据科学；
- 数据挖掘；
- 信息检索；
- 自然语言处理；
- 计算语言学；
- 计算机视觉；
- 语音处理；
- 推荐系统；
- 图学习；
- 生物信息学；
- 计算社会科学；
- 计算物理；
- 计算化学；
- 计算材料；
- 计算神经科学；
- 数字人文中的计算方法研究；
- 面向特定领域的计算方法应用论文。

本指南适用于以下论文类型：

- 提出新算法的论文；
- 提出新模型结构的论文；
- 提出新训练策略、推理策略或优化方法的论文；
- 构建新数据集的论文；
- 构建 benchmark 或评价协议的论文；
- 提出新任务定义的论文；
- 提出计算分析框架的论文；
- 将计算方法应用于特定科学或社会问题的论文；
- 对已有模型进行系统评测、诊断或解释的论文；
- 提出高效、鲁棒、公平、可解释、可迁移或可复现计算方法的论文。

此类论文的核心不是简单地宣称“我们的方法达到 SOTA”，而是形成以下闭环：

> 方法主张 — 评价协议 — 实验结果 — 解释分析 的一致性。

也就是说，论文必须说明：

```text
本文提出了什么计算主张？
该主张需要在什么任务和假设下成立？
采用什么数据、baseline、指标和实验协议检验？
实验结果是否真正支持该主张？
分析是否解释了为什么方法有效、何时失效、边界在哪里？
```

## 2. 不适用范围

本指南不应机械套用于以下情况。

### 2.1 纯理论数学论文

如果论文主要贡献是定理、证明、收敛性分析、复杂度下界、泛化界或形式化性质，而实验只是辅助说明，则应使用理论 / 数学论文范式。

本指南可用于其中的实验部分，但不应替代理论证明逻辑。

### 2.2 纯工程系统论文

如果论文主要贡献是一个系统、平台、工具链、软硬件架构或部署方案，而算法只是系统中的一个模块，则更适合工程 / 系统 / 设计型论文范式。

计算类论文关注：

```text
方法是否在明确任务和评价协议下有效。
```

工程系统论文关注：

```text
系统设计选择是否在真实约束下被评价证据证明有效。
```

### 2.3 纯自然科学实验论文

如果论文主要通过实验、观测、测量和表征解释自然现象，而计算只是辅助分析，则应以实验 / 观察型自然科学论文范式为主。

### 2.4 统计报告或数据分析报告

如果文本只是用现成方法分析数据，没有明确方法贡献、任务贡献、评价协议贡献或科学解释贡献，则不应包装成计算方法论文。

### 2.5 软件使用教程或工具说明书

如果文本主要说明如何使用某工具、库或软件包，而没有研究问题、方法设计、评价协议和实验分析，则不构成完整计算研究论文。

### 2.6 单纯排行榜式结果汇报

只报告“我们的模型在某榜单上得分最高”，但没有清晰方法贡献、实验公平性、消融分析、误差分析和机制解释，不构成强计算论文。

### 2.7 只换数据集或只换应用场景的弱应用论文

如果只是把已有模型应用到新数据上，而没有新任务、新问题、新数据质量、新评价协议、新领域洞见或系统性分析，贡献通常不足。

## 3. 知识生产逻辑

计算 / 算法 / 数据驱动型论文通过以下方式生产可信知识：

```text
任务定义
→ 问题形式化
→ 方法主张
→ 数据与实验协议
→ baseline 与评价指标
→ 主结果
→ 消融与分析
→ 鲁棒性 / 泛化 / 误差 / 失败案例
→ 解释方法为何有效
→ 限定适用范围
→ 声明贡献
```

### 3.1 可信知识从哪里来

这类论文的可信性主要来自：

| 可信性来源 | 写作中应体现什么 | Agent 判断重点 |
|---|---|---|
| 清晰任务定义 | 输入、输出、目标、假设、约束 | 是否知道模型解决什么问题 |
| 合理问题形式化 | 符号、变量、目标函数、优化目标 | 是否能支撑方法设计 |
| 明确方法主张 | 新模型、新策略、新数据、新协议或新解释 | 主张是否具体 |
| 公平实验协议 | 数据划分、baseline、指标、训练设置一致 | 是否可比较 |
| 强 baseline | 与领域强方法、简单强基线、合理变体比较 | 是否遗漏关键对手 |
| 充分实验 | 主结果、消融、鲁棒性、泛化、误差分析 | 是否覆盖主张 |
| 可复现性 | 代码、数据、随机种子、超参数、计算资源 | 是否能复查 |
| 解释分析 | 为什么有效、什么时候失效 | 是否超过“分数更高” |
| 边界限定 | 适用条件、数据限制、计算成本、伦理风险 | 是否避免过度推广 |

### 3.2 核心闭环：方法主张—评价协议—实验结果—解释分析

计算类论文最常见的问题是四者不一致。

强论文应形成闭环：

```text
方法主张：
本文提出一种层次化图神经网络，用于捕捉长距离依赖。

评价协议：
选择需要长距离推理的数据集，加入普通 GNN、Transformer 和无层次结构变体作为 baseline。

实验结果：
主结果显示本文方法在长距离依赖任务上提升明显，但在局部任务上收益较小。

解释分析：
消融实验表明层次化模块贡献最大；距离分组分析显示提升集中在远距离样本；失败案例表明噪声图结构会削弱效果。

结论：
本文方法适合长距离依赖显著且图结构较可靠的任务。
```

弱论文常见断裂：

```text
方法声称提升泛化能力，
但只在同分布测试集上评价。

方法声称更鲁棒，
但没有噪声、扰动或分布偏移实验。

方法声称更高效，
但只报告准确率，不报告训练 / 推理成本。

方法声称解决真实应用问题，
但只在标准 benchmark 上跑分。

方法声称可解释，
但没有解释性分析或人类可理解证据。
```

Agent 检查时应强制建立矩阵：

| 方法主张 | 需要的评价协议 | 当前实验是否支持 | 缺失分析 |
|---|---|---|---|
| 提升准确率 | 与强 baseline 比较 | 是 / 否 | 是否显著 |
| 提升鲁棒性 | 噪声 / 攻击 / 分布偏移 | 是 / 否 | 鲁棒曲线 |
| 降低计算成本 | 时间、显存、参数、FLOPs | 是 / 否 | 硬件说明 |
| 提升泛化 | 跨域、跨数据集、低资源 | 是 / 否 | 迁移实验 |
| 提升可解释性 | 案例、可视化、人评、忠实度 | 是 / 否 | 解释质量 |
| 新数据集有效 | 数据质量、标注一致性、任务难度 | 是 / 否 | 数据分析 |

### 3.3 计算论文中的 Claim-Evidence-Warrant

计算论文中的每个核心主张都应包含：

```text
Claim：方法主张是什么？
Evidence：哪些实验结果支持？
Warrant：为什么这些实验能检验该主张？
Limitation：该主张在哪些条件下成立？
```

示例：

```text
Claim：本文模型更擅长处理长文本分类任务。
Evidence：在三个长文本数据集上，本文模型相比 Transformer baseline 提升 3.2—5.1 个 F1 点；按文本长度分组后，提升主要出现在最长 25% 的样本上。
Warrant：由于长度分组分析直接对应长距离依赖能力，且消融实验显示去掉记忆模块后性能下降，该结果支持模型通过记忆模块改善长文本处理。
Limitation：该结论目前只在文本长度超过 2,000 token 的分类任务中验证，不能直接推广到生成任务。
```

Agent 判断规则：

```text
每当论文写“更好”“更鲁棒”“更泛化”“更高效”“更可解释”“解决了……”时，必须检查：
1. 评价指标是什么？
2. baseline 是谁？
3. 数据划分是否公平？
4. 是否有消融证明关键设计？
5. 是否有统计稳定性或多随机种子？
6. 是否有误差或失败案例分析？
7. 结论是否限定在实验覆盖范围内？
```

## 4. 典型论文结构

计算 / 算法 / 数据驱动型论文常见结构如下：

```text
Title
Abstract
Keywords

1. Introduction
2. Related Work
3. Problem Formulation / Task Definition
4. Method / Model / Algorithm
5. Experiments
6. Results
7. Analysis
8. Limitations
9. Conclusion

References
Appendix / Supplementary Materials
Code and Data Availability
```

不同论文类型可能有不同变体。

### 4.1 算法论文结构

```text
1. Introduction
2. Related Work
3. Problem Formulation
4. Proposed Algorithm
5. Theoretical or Complexity Analysis
6. Experiments
7. Results and Analysis
8. Limitations
9. Conclusion
```

重点：

- 算法流程；
- 目标函数；
- 收敛性或复杂度；
- 与已有算法的关系；
- 实验验证算法主张。

### 4.2 模型论文结构

```text
1. Introduction
2. Related Work
3. Task Definition
4. Model Architecture
5. Training Objective
6. Experiments
7. Main Results
8. Ablation and Analysis
9. Limitations
10. Conclusion
```

重点：

- 模型结构；
- 模块功能；
- 训练策略；
- 消融实验；
- 可解释分析；
- 计算成本。

### 4.3 数据集论文结构

```text
1. Introduction
2. Related Datasets and Tasks
3. Dataset Construction
4. Dataset Statistics and Quality Control
5. Task / Benchmark Setup
6. Baseline Experiments
7. Dataset Analysis
8. Ethical Considerations and Limitations
9. Conclusion
```

重点：

- 数据来源；
- 收集标准；
- 标注协议；
- 质量控制；
- 数据分布；
- 偏差与伦理；
- benchmark 任务；
- baseline 结果。

### 4.4 Benchmark 论文结构

```text
1. Introduction
2. Motivation for Benchmark
3. Task Suite and Evaluation Protocol
4. Dataset / Environment
5. Baseline Systems
6. Evaluation Results
7. Diagnostic Analysis
8. Limitations and Maintenance
9. Conclusion
```

重点：

- 为什么需要新 benchmark；
- benchmark 测什么能力；
- 指标是否合理；
- baseline 是否覆盖；
- 是否防止数据泄漏；
- 是否可持续维护。

### 4.5 应用型计算论文结构

```text
1. Introduction
2. Domain Problem and Related Work
3. Task Formulation
4. Data and Methods
5. Experiments
6. Results
7. Domain Interpretation / Error Analysis
8. Limitations
9. Conclusion
```

重点：

- 领域问题真实；
- 计算任务定义合理；
- 数据质量和偏差；
- 方法与领域问题匹配；
- 结果能否产生领域洞见，而不是只报告模型分数。

### 4.6 章节功能总览

| 章节 | 核心功能 | 不应写成 |
|---|---|---|
| Introduction | 提出任务、缺口、方法主张和贡献 | 泛泛 AI 背景 |
| Related Work | 定位已有方法、任务、数据和评价不足 | 文献列表 |
| Problem Formulation | 明确输入、输出、目标、假设 | 模糊任务描述 |
| Method / Model | 说明方法如何工作及为何这样设计 | 模块堆砌 |
| Experiments | 说明实验协议，使结果可比较、可复现 | 只列数据集和指标 |
| Results | 呈现主结果和对比 | 只说 SOTA |
| Analysis | 解释为什么有效、何时失效 | 重复主结果 |
| Limitations | 说明边界、风险、成本和失败场景 | 形式化套话 |
| Conclusion | 回到任务和贡献 | 夸大总结 |

## 5. 各章节写作指南

## 5.1 标题 Title

### 写作目标

标题应准确传达论文的任务、方法、数据、模型或评价贡献，使目标读者能够快速判断论文类型和核心 novelty。

### 常见标题类型

| 类型 | 结构 | 适用场景 |
|---|---|---|
| 方法名 + 任务 | X: A Method for Y | 新模型 / 新算法 |
| 能力 + 方法 | Improving X via Y | 改进某能力 |
| 任务定义型 | Towards X for Y | 新任务或探索性问题 |
| 数据集型 | XBench / XData: A Dataset for Y | 数据集或 benchmark |
| 诊断型 | Evaluating X under Y | 评测或分析论文 |
| 解释型 | Understanding X in Y Models | 机制解释论文 |
| 应用型 | Predicting X from Y using Z | 领域应用 |

### 常见错误

- 标题只写模型名，读者不知道解决什么任务；
- 标题使用 “SOTA”“powerful”“efficient” 等宣传词；
- 标题声称 general / robust / universal，但实验范围有限；
- 标题过度强调应用，实际只是 benchmark 跑分；
- 标题暗示理论贡献，正文只有实验；
- 标题过长，堆叠模型、数据、任务和场景。

### Agent 质量判断

```text
1. 标题是否体现任务或问题？
2. 是否体现方法或贡献类型？
3. 是否避免过度声称？
4. 是否与实验范围一致？
5. 是否能被目标领域读者检索？
```

---

## 5.2 摘要 Abstract

### 写作目标

摘要应压缩完整计算研究链条：

```text
任务 / 问题
→ 现有方法不足
→ 本文方法 / 数据 / benchmark
→ 实验协议
→ 关键结果
→ 分析或贡献
→ 边界
```

### 推荐结构

```text
1. 问题句：指出任务或计算问题的重要性。
2. 缺口句：说明现有方法、数据或评价协议的不足。
3. 方法句：概括本文提出的模型、算法、数据集或 benchmark。
4. 实验句：说明在哪些数据、baseline 和指标上评价。
5. 结果句：报告关键结果，不只说 SOTA。
6. 分析句：说明为什么有效或在哪些条件下有效。
7. 贡献句：说明本文推进了什么计算知识。
```

### 示例骨架

```text
X 任务要求模型在 Y 条件下完成 Z，但现有方法通常依赖 A，难以处理 B。本文提出 C，一种基于 D 的模型 / 算法，通过 E 模块和 F 训练目标提升 G 能力。我们在 H 个数据集上与 I 类 baseline 进行比较，并报告 J、K 和 L 指标。实验结果显示，C 在主要指标上平均提升 M，同时在 N 分析中表现出更好的鲁棒性。消融实验表明，E 模块是性能提升的主要来源。本文的贡献在于为 X 任务提供了一种可复现的建模方案，但其计算成本和跨域泛化仍需进一步研究。
```

### 常见错误

- 摘要只写模型结构，不写任务缺口；
- 只写“达到 SOTA”，不写具体结果；
- 不说明数据集和 baseline；
- 不说明方法为什么有效；
- 贡献写成“提出了一种新方法”，但不说明新在哪里；
- 摘要中过度声称泛化、鲁棒、可解释；
- 摘要没有提限制或边界。

### Agent 质量判断

```text
1. 摘要是否明确任务？
2. 是否说明现有方法不足？
3. 是否概括本文方法主张？
4. 是否报告实验协议和关键结果？
5. 是否说明分析发现？
6. 是否避免只写 SOTA？
7. 是否与正文实验一致？
```

## 5.3 Introduction 引言

### 写作目标

Introduction 应建立任务重要性、现有方法不足、本文方法主张和贡献。

它不是“AI 很重要”的背景介绍，而应回答：

```text
这个任务为什么重要？
现有方法为什么不足？
不足来自模型、数据、任务定义、评价协议还是理论解释？
本文提出什么主张？
实验如何验证？
贡献是什么？
```

### 推荐组织方式

```text
1. 任务背景：说明研究任务和应用 / 科学意义。
2. 挑战：说明该任务的计算难点。
3. 现有方法：概括已有路线。
4. 缺口：指出现有方法的具体不足。
5. 本文方法：说明核心思想。
6. 实验设计：概括评价数据、baseline、指标和分析。
7. 贡献列表：明确 novelty。
```

### 好的任务挑战写法

弱写法：

```text
近年来，深度学习发展迅速，在很多任务上取得了好成绩。
```

强写法：

```text
长文档事件抽取要求模型同时识别跨段落实体、事件触发词和远距离论元关系。现有句子级模型难以建模跨段落依赖，而直接使用长上下文 Transformer 又带来显著计算成本。
```

弱写法：

```text
现有方法存在泛化能力不足的问题。
```

强写法：

```text
现有方法主要在同分布测试集上优化准确率，但在跨领域数据上性能下降明显。这表明模型可能学习了领域特定词汇模式，而不是任务所需的稳定语义关系。
```

### 缺口类型

| 缺口类型 | 说明 | 示例 |
|---|---|---|
| 方法缺口 | 现有模型不能处理某结构、约束或能力 | 难以建模长距离依赖 |
| 数据缺口 | 缺少高质量、多样性或特定场景数据 | 缺少跨语言标注数据 |
| 任务缺口 | 现有任务定义不能覆盖真实需求 | 只评估分类，不评估解释 |
| 评价缺口 | benchmark 或指标不能测出目标能力 | 指标忽略校准性 |
| 效率缺口 | 方法性能好但成本高 | 推理延迟过高 |
| 鲁棒性缺口 | 对噪声、攻击、分布偏移脆弱 | 轻微扰动导致性能下降 |
| 可解释性缺口 | 不知道模型为什么有效 | 缺少机制分析 |
| 公平性 / 安全缺口 | 数据和模型存在偏差或风险 | 对低资源群体效果差 |

### 贡献写法

弱贡献：

```text
本文提出一种新模型，并在多个数据集上取得 SOTA。
```

强贡献：

```text
本文贡献如下：
1. 提出一种面向长文档事件抽取的层次化记忆网络，将段落级状态与文档级全局记忆结合，以建模跨段落论元关系。
2. 设计了基于论元距离的诊断评价，验证模型提升主要来自远距离关系建模。
3. 在三个公开数据集和一个跨领域测试集上进行实验，并通过消融、长度分组和错误分析说明模型的有效性与边界。
```

### 常见错误

- 引言过度泛泛谈 AI、大数据或深度学习；
- 不清楚任务是什么；
- 缺口只说“现有方法效果不好”，没有分析原因；
- 贡献只写 SOTA；
- 没有说明方法核心思想；
- 没有预告实验如何验证主张；
- 引言声称鲁棒 / 泛化 / 高效，但后文实验没有对应；
- Related Work 和 Introduction 重复。

### Agent 质量判断

```text
1. 是否在前几段明确任务？
2. 是否说明任务难点？
3. 是否指出具体缺口？
4. 是否说明本文方法的核心思想？
5. 是否明确贡献类型？
6. 是否预告评价协议和分析？
7. 是否避免过度声称？
```

## 5.4 Related Work 相关研究

### 写作目标

Related Work 的功能是定位本文在已有方法、任务、数据和评价协议中的位置。它不是文献摘要列表，而是说明：

```text
已有方法有哪些路线？
每条路线解决了什么？
仍有什么不足？
本文与它们的关系是什么？
为什么本文的设计是必要的？
```

### 推荐组织方式

#### 方式一：按方法路线组织

```text
1. 基于规则 / 特征的方法
2. 神经网络方法
3. 预训练模型方法
4. 检索增强 / 图结构 / 多模态方法
5. 本文方法定位
```

#### 方式二：按能力问题组织

```text
1. 长距离依赖建模
2. 鲁棒性提升
3. 参数高效训练
4. 可解释分析
5. 本文如何连接这些问题
```

#### 方式三：按任务 / 数据组织

```text
1. 现有任务定义
2. 现有数据集
3. 现有 benchmark
4. 本文新任务 / 新数据 / 新协议
```

#### 方式四：按评价不足组织

```text
1. 现有方法主要比较准确率
2. 少量研究关注泛化
3. 缺少系统误差分析
4. 本文补充诊断性评价
```

### 好的 Related Work 段落结构

```text
主题句：指出一类方法或研究方向。
综合句：概括该方向的核心思想。
不足句：说明其在本文任务中的限制。
连接句：说明本文如何继承、改进或区别于该方向。
```

示例：

```text
一类方法通过扩大上下文窗口处理长文档任务。这些方法能够直接利用更多文本信息，但计算复杂度随输入长度快速增加，且在极长文档中仍容易忽略局部事件结构。与这类方法不同，本文采用层次化记忆机制，将局部段落表示与全局文档状态分离，从而在控制计算成本的同时保留跨段落信息。
```

### Related Work 与 baseline 的关系

Related Work 中讨论的重要方法，通常应在 Experiments 中作为 baseline 或说明为什么不能比较。

Agent 检查：

```text
1. Related Work 中批评的主要方法，Evaluation 是否比较了？
2. 没有比较时是否说明原因？
3. baseline 是否覆盖主要研究路线？
4. 是否遗漏公认强方法？
```

### 常见错误

- 按作者逐篇罗列；
- 不说明已有方法的假设和限制；
- 把 Related Work 写成 Introduction 的重复；
- 只引用支持本文的文献；
- 批评已有方法不准确；
- 相关工作与实验 baseline 脱节；
- 忽略近期强 baseline；
- 不区分算法、模型、数据集、benchmark 和应用论文。

### Agent 质量判断

```text
1. Related Work 是否按问题或方法路线组织？
2. 是否说明已有方法为何不足？
3. 是否准确定位本文方法？
4. 是否为 baseline 选择提供依据？
5. 是否避免文献罗列？
6. 是否处理最接近本文的工作？
```

## 5.5 Problem Formulation / Task Definition 问题形式化 / 任务定义

### 写作目标

Problem Formulation 是计算论文的地基。它必须使读者清楚：

```text
输入是什么？
输出是什么？
模型要学习什么函数？
有哪些假设？
训练和测试如何定义？
评价目标是什么？
本文不解决什么？
```

### 应包含内容

- 输入空间；
- 输出空间；
- 数据实例；
- 标签或目标；
- 任务约束；
- 学习设置；
- 训练 / 验证 / 测试划分；
- 评价目标；
- 符号定义；
- 假设条件；
- 与已有任务定义的区别。

### 常见学习设置

| 设置 | 需要说明 |
|---|---|
| 监督学习 | 输入、标签、损失函数、数据划分 |
| 半监督学习 | 有标签 / 无标签数据比例和使用方式 |
| 自监督学习 | 预训练任务、正负样本、目标函数 |
| 强化学习 | 状态、动作、奖励、环境、策略 |
| 生成建模 | 条件、目标分布、采样方式、评价 |
| 检索任务 | 查询、候选集、相关性定义、排序指标 |
| 图学习 | 节点、边、图结构、任务级别 |
| 时序预测 | 历史窗口、预测步长、滚动方式 |
| 多模态学习 | 模态输入、对齐假设、融合目标 |
| 低资源学习 | 数据限制、迁移设定、few-shot 定义 |
| 跨域泛化 | 源域、目标域、训练 / 测试分布 |
| 联邦学习 | 客户端、通信轮次、隐私约束 |
| 隐私 / 安全 | 威胁模型、攻击者能力、防御目标 |

### 任务定义模板

```text
给定数据集 D = {(x_i, y_i)}_{i=1}^N，其中 x_i 表示……，y_i 表示……。
本文任务是学习函数 f_\theta: X → Y，使其在测试集上最大化 / 最小化指标 M。
训练阶段允许使用……，测试阶段只能访问……。
本文假设……，不考虑……。
```

### 输入输出写法

弱写法：

```text
模型输入用户数据，输出预测结果。
```

强写法：

```text
模型输入用户在过去 T 天内的交互序列 x = (a_1, a_2, ..., a_T)，其中每个 a_t 包含物品 ID、时间戳和上下文特征；模型输出下一个时间窗口内用户点击候选物品的概率分布。
```

### 假设与边界

必须说明：

- 是否假设训练和测试同分布；
- 是否假设标签准确；
- 是否假设图结构可用；
- 是否假设完整历史数据；
- 是否假设候选集固定；
- 是否假设无数据泄漏；
- 是否假设测试时可访问外部知识；
- 是否假设模型大小或计算资源限制。

### 常见错误

- 任务定义模糊；
- 输入输出不明确；
- 符号定义不完整；
- 训练和测试条件混淆；
- 未说明数据可见性；
- 未说明任务与已有任务的区别；
- 评价目标和方法目标不一致；
- 后文实验使用了 Problem Formulation 中未定义的信息。

### Agent 质量判断

```text
1. 是否能明确说出输入、输出和目标？
2. 是否定义符号和变量？
3. 是否说明学习设置？
4. 是否说明训练 / 测试可用信息？
5. 是否说明假设和不处理的问题？
6. 是否为 Method 和 Experiments 提供基础？
```

## 5.6 Method / Model / Algorithm 方法、模型或算法

### 写作目标

Method 部分应说明本文方法如何工作，以及为什么这种设计能解决前文任务难点。

它不应只是模块堆砌，而应形成：

```text
设计动机
→ 方法总览
→ 核心模块
→ 训练目标
→ 算法流程
→ 复杂度 / 成本
→ 实现细节
```

### 推荐组织方式

```text
4.1 Overview
4.2 Input Representation
4.3 Core Module 1
4.4 Core Module 2
4.5 Training Objective
4.6 Inference Procedure
4.7 Complexity and Implementation Details
```

### 方法总览

方法总览应回答：

```text
1. 方法整体输入输出是什么？
2. 由哪些核心组件组成？
3. 每个组件解决什么问题？
4. 组件之间如何连接？
5. 与已有方法的关键区别在哪里？
```

### 模型结构写法

弱写法：

```text
模型由编码器、注意力模块和分类器组成。
```

强写法：

```text
模型首先使用局部编码器获得段落级表示，以保留短距离上下文；随后通过全局记忆模块在段落之间传递状态，从而建模跨段落依赖；最后，分类器将局部表示与全局记忆拼接，用于预测事件论元角色。
```

### 算法流程

对于算法论文，应提供：

- 输入；
- 输出；
- 初始化；
- 迭代步骤；
- 更新规则；
- 终止条件；
- 复杂度；
- 与已有算法差异。

推荐写法：

```text
Algorithm 1 summarizes the training procedure. The algorithm alternates between updating the representation module and recalibrating pseudo-labels. This alternating design prevents early noisy pseudo-labels from dominating representation learning.
```

### 目标函数

应说明：

- 优化目标；
- 损失函数每一项的含义；
- 正则项；
- 权重系数；
- 训练信号来源；
- 为什么这样设计；
- 与任务指标的关系；
- 是否存在近似或松弛。

弱写法：

```text
我们使用交叉熵损失训练模型。
```

强写法：

```text
训练目标包含监督交叉熵损失和一致性正则项。前者使模型拟合标注样本，后者约束同一实例在两种扰动视图下的预测保持一致，以提升对输入噪声的鲁棒性。
```

### 复杂度和效率

如果论文声称高效、轻量、可扩展或低成本，应报告：

- 参数量；
- FLOPs；
- 训练时间；
- 推理延迟；
- 显存占用；
- 数据预处理成本；
- 通信成本；
- 时间复杂度；
- 空间复杂度；
- 与 baseline 的比较。

### 实现细节

Method 或 Experiments 中应说明影响复现的细节：

- backbone；
- optimizer；
- learning rate；
- batch size；
- epoch；
- dropout；
- scheduler；
- token length；
- sampling strategy；
- negative sampling；
- early stopping；
- random seed；
- hardware；
- framework；
- library version；
- inference setting；
- prompt template；
- decoding strategy；
- checkpoint selection。

### 常见错误

- 只画模型图，不解释设计理由；
- 每个模块看似复杂，但不知道解决什么问题；
- 数学符号未定义；
- 目标函数与任务指标脱节；
- 没有说明训练和推理差异；
- 不报告复杂度却声称高效；
- 不说明实现细节；
- 与已有方法的区别不清；
- 模块越堆越多，但消融不足。

### Agent 质量判断

```text
1. Method 是否直接回应任务挑战？
2. 是否有整体方法总览？
3. 每个模块是否有 rationale？
4. 符号和公式是否清楚？
5. 目标函数是否解释每一项作用？
6. 是否说明训练和推理流程？
7. 是否报告复杂度和实现细节？
8. 是否能被消融实验验证？
```

```text
在哪些数据上测试？
与哪些 baseline 比较？
用什么指标？
如何训练？
如何调参？
使用什么计算资源？
如何保证公平？
```

```text
5.1 Datasets

5.3 Evaluation Metrics
5.4 Implementation Details

5.6 Training and Inference Protocol
5.7 Computational Resources
5.8 Reproducibility Setup
```

### 数据集写法

应说明：

- 数据来源；
- 数据规模；
- 划分方式；
- 标签定义；
- 类别分布；
- 数据预处理；

- 数据泄漏防控；

- 许可和伦理；
- 与任务的适配性。

| Dataset | Domain | Train | Dev | Test | Labels | Metric | Notes |
|---|---:|---:|---:|---:|---:|---|---|

### baseline 写法

baseline 应覆盖：

- 经典方法；
- 当前强方法；
- 简单强基线；
- 与本文最接近的方法；
- 同类架构；
- 大模型 / 小模型；

```text
3. 说明是否使用官方代码；
4. 说明是否重新调参；
5. 说明所有模型是否使用相同训练数据；
6. 说明计算资源是否可比；
7. 对不能比较的方法解释原因。
```

### 指标写法

指标必须对应任务和方法主张。

| 任务 / 主张 | 常见指标 |
|---|---|
| 分类 | Accuracy, F1, Macro-F1, AUROC |
| 不平衡分类 | Macro-F1, balanced accuracy, PR-AUC |
| 检索 / 排序 | MRR, MAP, Recall@K, NDCG |
| 生成 | BLEU, ROUGE, METEOR, BERTScore, human evaluation |
| 语言模型 | perplexity, exact match, pass@k |
| 检测 | mAP, IoU, precision, recall |
| 分割 | Dice, IoU |
| 校准 | ECE, Brier score |
| 鲁棒性 | accuracy under perturbation, attack success rate |
| 效率 | latency, throughput, memory, FLOPs |
| 公平性 | group-wise performance, disparity metrics |
| 可解释性 | fidelity, plausibility, human agreement |
| 科学预测 | RMSE, MAE, R², correlation, uncertainty calibration |

Agent 应提醒：

```text
不要只选择最有利指标。
指标应与任务目标、数据分布和应用风险匹配。
```

- optimizer；
- learning rate；
- batch size；
- epochs；
- early stopping；

- 训练时间；
- 推理时间；
- 显存；
- 参数量；
- energy / cost，如相关；
- batch size；
- precision；
- distributed training setting。

公平实验需要：

```text

2. 相同预处理；
3. 相同训练数据；
4. 合理调参；
5. 统一指标；
```

### 常见错误

- 不说明数据划分；

```text
3. baseline 是否强且公平？
4. 指标是否对应主张？
5. 训练细节是否完整？

7. 是否说明计算资源？
8. 是否防止数据泄漏？
```

## 5.8 Results 结果

### 写作目标

Results 部分展示实验结果，但不应停留在排行榜。它应说明：

```text
本文方法在哪些任务上有效？
相对于哪些 baseline 有提升？
提升在哪些指标、数据或条件下出现？
是否稳定？
是否有代价？
结果是否支持方法主张？
```

### 主结果写法

推荐结构：

```text
1. 说明表格回答什么评价问题。
2. 描述主要趋势。
3. 指出本文方法相对最强 baseline 的差异。
4. 说明统计稳定性或多随机种子结果。
5. 指出例外情况。
6. 连接到后续分析。
```

示例：

```text
Table 2 compares our method with six baselines across three datasets. Our method achieves the best Macro-F1 on all datasets, with the largest gain on the low-resource dataset. The improvement over the strongest baseline is 2.8 points on average. However, the gain on Dataset C is small, suggesting that the proposed module is most beneficial when training data are limited. We analyze this effect further in Section 6.3.
```

### 表格写作原则

主结果表应清楚：

- 数据集；
- 指标；
- baseline；
- 本文方法；
- 平均值；
- 标准差或置信区间；
- 是否显著；
- 最好和次好结果；
- 模型大小或成本，如相关。

### 常见主结果错误

- 只强调最高分；
- 不说明与最强 baseline 的差距；
- 不报告方差；
- 不解释不同数据集差异；
- 忽略性能下降项；
- 表格太大但无叙事；
- 没有连接方法设计。

### Agent 质量判断

```text
1. Results 是否回答评价问题？
2. 是否与 baseline 公平比较？
3. 是否报告稳定性？
4. 是否解释差异模式？
5. 是否处理负结果？
6. 是否为 Analysis 铺垫？
```

## 5.9 Analysis 分析

### 写作目标

Analysis 是计算论文从“跑分”走向“知识贡献”的关键。它应回答：

```text
为什么方法有效？
哪个模块有效？
在哪些样本上有效？
什么时候失效？
是否真的学到了目标能力？
是否具有泛化、鲁棒、效率或可解释优势？
```

### 常见分析类型

#### 1. 消融实验 Ablation

目的：

```text
证明关键模块、损失项、训练策略或数据组件确实贡献性能。
```

应包括：

- 完整模型；
- 去掉模块；
- 替换模块；
- 去掉损失项；
- 改变超参数；
- 简化架构；
- 只保留某部分；
- 分析性能变化。

消融写法：

```text
Removing the global memory module causes the largest drop, especially on long-document instances. This supports our claim that the module helps capture cross-paragraph dependencies.
```

#### 2. 鲁棒性分析

用于检验：

- 输入噪声；
- 对抗扰动；
- 拼写错误；
- 缺失模态；
- 标签噪声；
- 分布偏移；
- 长尾样本；
- 低资源条件；
- 不同随机种子。

#### 3. 泛化分析

用于检验：

- 跨数据集；
- 跨领域；
- 跨语言；
- 跨时间；
- 跨用户群体；
- 跨分辨率；
- 跨硬件；
- zero-shot / few-shot；
- out-of-distribution。

#### 4. 误差分析

用于解释失败模式。

推荐分类：

```text
错误类型
→ 错误比例
→ 代表案例
→ 可能原因
→ 与方法设计的关系
→ 未来改进方向
```

#### 5. 效率分析

用于支持高效、轻量、可部署主张。

应报告：

- 参数量；
- FLOPs；
- 推理延迟；
- 吞吐；
- 显存；
- 训练时间；
- 能耗或成本；
- 随输入长度 / 数据规模变化的曲线。

#### 6. 可解释性分析

用于支持模型解释或机制理解。

可包括：

- attention / attribution 可视化；
- feature importance；
- counterfactual；
- representation analysis；
- probing；
- clustering；
- human evaluation；
- explanation fidelity；
- case study。

Agent 应提醒：

```text
注意力图不等于可靠解释。
可解释性分析应说明解释指标或解释目的。
```

#### 7. 失败案例

失败案例应说明：

- 模型在哪些输入上失败；
- 失败是否与数据偏差、任务难度或模型假设有关；
- 是否存在安全风险；
- 是否影响主要结论；
- 后续如何改进。

### 常见错误

- 没有消融；
- 消融只去掉小模块，不检验核心贡献；
- 分析与方法主张无关；
- 只展示成功案例；
- 鲁棒性声称没有鲁棒性实验；
- 泛化声称没有跨域实验；
- 可解释声称只有可视化；
- 错误分析只列几个例子；
- 不报告效率却声称高效。

### Agent 质量判断

```text
1. Analysis 是否解释主结果？
2. 消融是否对应方法创新点？
3. 是否分析负结果和失败案例？
4. 是否支持鲁棒、泛化、高效、可解释等主张？
5. 是否将分析结果回扣方法设计？
6. 是否避免只写 SOTA？
```

## 5.10 Limitations 局限性

### 写作目标

Limitations 应清楚说明方法、数据、实验协议和应用边界，避免过度推广。

### 应包含内容

- 数据限制；
- 方法假设；
- 计算成本；
- 评估范围；
- 任务边界；
- baseline 限制；
- 可复现限制；
- 部署风险；
- 偏差、公平性、隐私、安全或伦理问题；
- 失败场景；
- 尚未验证的条件。

### 好的局限性写法

弱写法：

```text
本文仍有一些不足，未来会进一步优化。
```

强写法：

```text
本文方法在跨领域测试中仍出现明显性能下降，说明模型可能依赖源域中的表面模式。由于本文没有引入目标域无标签数据，当前结果不能证明模型具有强域泛化能力。
```

弱写法：

```text
模型计算成本较高。
```

强写法：

```text
本文模型相比最强 baseline 增加约 35% 训练时间，主要来自全局记忆更新模块。因此，该方法更适合离线训练场景，在严格低延迟推理场景中需要进一步压缩。
```

### 常见错误

- 局限性过于笼统；
- 不承认实验协议不足；
- 不讨论数据偏差；
- 声称开源但没有说明数据许可；
- 不讨论计算成本；
- 不讨论失败案例；
- 将未完成工作写成未来计划，而不是边界说明。

### Agent 质量判断

```text
1. 是否说明方法在哪些条件下不适用？
2. 是否说明实验没有覆盖什么？
3. 是否说明数据偏差和伦理风险？
4. 是否说明计算成本？
5. 是否避免自我否定式写法？
6. 是否与 Discussion 和 Conclusion 一致？
```

## 5.11 Conclusion 结论

### 写作目标

Conclusion 应回到任务、方法主张、实验结果和贡献边界。

### 推荐结构

```text
本文研究了 X 任务中的 Y 问题。
为解决该问题，本文提出 Z 方法 / 数据集 / benchmark。
实验结果显示……
分析表明……
本文贡献在于……
局限在于……
未来工作可……
```

### 常见错误

- 只重复摘要；
- 只说 SOTA；
- 不提分析发现；
- 结论比实验更强；
- 不说明边界；
- 引入新实验或新主张；
- 用宣传语结束。

### Agent 质量判断

```text
1. 是否回答 Introduction 中的问题？
2. 是否总结方法贡献？
3. 是否引用关键实验和分析结果？
4. 是否说明边界？
5. 是否避免过度声称？
```

## 6. 常见贡献类型

计算 / 算法 / 数据驱动型论文的贡献可以来自多个方面。Agent 应帮助用户明确贡献类型，而不是只写“提出新方法并取得 SOTA”。

### 6.1 新算法贡献

提出新的计算过程、优化策略、搜索策略、推理算法或训练算法。

质量要求：

- 算法流程清楚；
- 与已有算法区别明确；
- 复杂度或效率分析；
- 实验验证算法主张；
- 消融或变体分析。

贡献句示例：

```text
本文提出一种交替优化算法，将伪标签更新与表示学习解耦，从而缓解早期噪声标签对模型训练的影响。
```

### 6.2 新模型结构贡献

提出新的架构、模块、编码器、解码器、融合机制、记忆机制、图结构或多模态结构。

质量要求：

- 模块有明确任务动机；
- 不是简单堆叠；
- 消融证明模块有效；
- 分析说明在哪些样本或条件下有效。

贡献句示例：

```text
本文设计了层次化记忆模块，用于在长文档中同时保留局部事件结构和全局上下文。
```

### 6.3 新训练策略贡献

提出新的预训练、微调、对比学习、数据增强、蒸馏、正则化、prompting、curriculum 或优化策略。

质量要求：

- 说明训练信号来源；
- 与目标能力对应；
- 与普通训练策略比较；
- 分析超参数和稳定性。

贡献句示例：

```text
本文提出基于语义一致性的扰动训练策略，使模型在输入噪声下保持预测稳定。
```

### 6.4 新数据集贡献

构建新的数据集，解决规模、多样性、标注质量、场景覆盖或任务定义不足。

质量要求：

- 数据来源透明；
- 标注协议清楚；
- 质量控制；
- 数据统计；
- 偏差和伦理；
- benchmark baseline；
- 数据可用性说明。

贡献句示例：

```text
本文构建了一个跨领域长文本事件抽取数据集，并提供细粒度论元标注和跨域测试划分。
```

### 6.5 新 benchmark / 评价协议贡献

提出新的任务集合、评价指标、诊断测试或 benchmark。

质量要求：

- 说明现有 benchmark 不足；
- 指标和任务测量目标能力；
- baseline 覆盖充分；
- 防止数据泄漏；
- 可维护和可复现；
- 诊断结果有洞见。

贡献句示例：

```text
本文提出一个面向跨域鲁棒性的 benchmark，将同分布测试、领域迁移测试和噪声扰动测试统一到同一评价协议中。
```

### 6.6 新任务贡献

形式化一个此前未充分定义的计算任务。

质量要求：

- 任务来自真实需求；
- 输入输出明确；
- 与已有任务差异清楚；
- 数据和评价支持任务定义；
- baseline 证明任务具有挑战性。

贡献句示例：

```text
本文定义了证据约束问答任务，要求模型不仅输出答案，还必须选择支持答案的可验证证据片段。
```

### 6.7 新理论解释或机制分析贡献

解释模型为何有效、何时失效、学到了什么表示或能力。

质量要求：

- 分析方法明确；
- 证据来自实验、诊断、可视化或理论；
- 不把可视化当作证明；
- 能产生可迁移洞见。

贡献句示例：

```text
本文通过距离分组、表示探针和反事实扰动分析表明，模型性能提升主要来自远距离依赖建模能力。
```

### 6.8 应用型计算贡献

将计算方法用于特定领域，并产生领域洞见或解决真实任务。

质量要求：

- 领域问题真实；
- 方法适配领域约束；
- 数据质量可靠；
- 结果不仅是模型分数，还应有领域解释；
- 与领域专家或已知规律对话。

贡献句示例：

```text
本文将图神经网络用于蛋白相互作用预测，并通过结构域级误差分析揭示模型在低同源序列上的主要失败模式。
```

### 6.9 效率 / 资源贡献

降低训练、推理、存储、通信或标注成本。

质量要求：

- 明确成本指标；
- 与强 baseline 比较；
- 报告硬件和设置；
- 说明性能—成本 trade-off；
- 不只报告准确率。

贡献句示例：

```text
本文提出参数高效微调策略，在保持主要任务性能的同时将可训练参数减少到全量微调的 3%。
```

## 7. 证据与论证标准

## 7.1 强证据的特征

计算类论文中的强证据通常具有以下特征：

```text
1. 任务定义清楚；
2. 数据划分合理；
3. baseline 强且公平；
4. 指标与主张对应；
5. 多数据集或多设置验证；
6. 报告多随机种子和方差；
7. 消融实验对应核心设计；
8. 鲁棒性、泛化或效率分析覆盖相关主张；
9. 错误分析和失败案例解释边界；
10. 可复现信息充分；
11. 结论不超过实验支持。
```

### 强证据组合示例

| 主张 | 强证据组合 |
|---|---|
| 方法性能更好 | 强 baseline + 多数据集 + 多随机种子 + 显著性 / 方差 |
| 模块有效 | 消融 + 替代模块 + 分组分析 |
| 更鲁棒 | 噪声、扰动、分布偏移实验 |
| 更泛化 | 跨域、跨数据、低资源或 OOD 测试 |
| 更高效 | 参数、FLOPs、延迟、显存、训练时间 |
| 新数据集有价值 | 数据统计 + 标注一致性 + baseline + 难度分析 |
| benchmark 合理 | 能区分模型能力 + 任务多样 + 指标有效 |
| 可解释 | 解释指标 + 案例 + 人类或诊断验证 |

## 7.2 弱证据的特征

弱证据常表现为：

- 只比较弱 baseline；
- 只报告一个数据集；
- 只跑一次；
- 不报告方差；
- 不说明数据划分；
- 不说明训练细节；
- 不做消融；
- 没有错误分析；
- 指标与主张不匹配；
- 声称泛化但无跨域；
- 声称鲁棒但无扰动；
- 声称高效但不报成本；
- 声称可解释但只有 attention 图；
- 使用测试集调参；
- 存在数据泄漏风险；
- 只报告 SOTA，不解释原因。

## 7.3 baseline 公平性标准

Agent 应检查：

```text
1. 是否包含当前强 baseline？
2. 是否包含简单强 baseline？
3. 是否包含与本文最接近的方法？
4. baseline 是否使用相同训练数据？
5. 是否使用官方实现或合理复现？
6. 是否对 baseline 进行合理调参？
7. 是否在相同硬件和设置下比较效率？
8. 是否说明不可比条件？
9. 是否避免 cherry-picking？
```

## 7.4 数据泄漏检查

数据泄漏是计算论文高风险问题。

常见泄漏包括：

- 训练集和测试集重复；
- 同一用户、文档、图像、序列泄漏到不同划分；
- 预处理使用了测试集统计；
- 数据增强产生近重复样本；
- 预训练语料包含测试集；
- benchmark 公开后被模型训练污染；
- prompt 或检索系统访问测试标签；
- 时间序列随机划分导致未来信息泄漏；
- 图学习中测试节点信息被训练阶段不当使用。

Agent 检查规则：

```text
1. 数据是否去重？
2. 划分单位是否正确？
3. 是否按时间划分而非随机划分？
4. 预处理是否只在训练集拟合？
5. 外部数据是否包含测试信息？
6. LLM 或预训练模型是否可能见过 benchmark？
7. 是否需要 contamination analysis？
```

## 7.5 结论强度与证据匹配

| 证据情况 | 合适表达 | 避免表达 |
|---|---|---|
| 多数据集、多 baseline、多种子 | 结果表明方法在这些任务上稳定优于 baseline | 全面解决该任务 |
| 单数据集提升 | 在该数据集上表现更好 | 方法具有普遍优势 |
| 无跨域实验 | 同分布设置下有效 | 泛化能力强 |
| 无鲁棒实验 | 性能较高 | 鲁棒 |
| 无效率指标 | 准确率提升 | 高效 |
| 有可视化 | 提供定性解释 | 证明模型可解释 |
| 小样本用户评估 | 初步显示 | 用户普遍认可 |

## 8. 文献综述写法

### 8.1 Related Work 的真正功能

计算类论文的文献综述应服务于三个目标：

```text
1. 说明本文任务或问题从哪里来；
2. 说明已有方法路线和局限；
3. 说明本文 novelty 的位置。
```

### 8.2 组织文献的四种常用方式

#### 方式一：方法路线型

适合模型 / 算法论文。

```text
传统方法
→ 深度学习方法
→ 预训练模型方法
→ 结构增强方法
→ 本文方法
```

#### 方式二：任务演化型

适合新任务或应用论文。

```text
早期任务定义
→ 扩展任务定义
→ 当前 benchmark
→ 仍未覆盖的真实需求
→ 本文任务
```

#### 方式三：能力诊断型

适合鲁棒性、泛化、可解释等论文。

```text
已有研究如何评价准确率
→ 如何评价鲁棒性
→ 如何评价泛化
→ 现有评价仍缺少什么
→ 本文协议
```

#### 方式四：数据资源型

适合数据集论文。

```text
已有数据集 A 类
→ 已有数据集 B 类
→ 标注、规模、领域和偏差问题
→ 本文数据集补充什么
```

### 8.3 Related Work 段落模板

```text
一类研究通过 X 解决 Y。它们的优势是 A，但通常依赖 B 假设，因此在 C 场景下表现受限。本文与这些研究相同之处在于……，不同之处在于……，并通过……评价这种差异。
```

### 8.4 常见错误

- 把文献按年份或作者罗列；
- 不解释已有方法为什么不够；
- 只强调本文优点；
- 忽略最相近工作；
- Related Work 中说某方法不足，但实验不比较；
- 混淆模型、数据集、benchmark 和应用论文；
- 不说明本文与预训练模型、基础模型或大模型的关系；
- 夸大已有方法缺陷。

### Agent 质量判断

```text
1. 是否围绕任务和方法缺口组织？
2. 是否比较主要方法路线？
3. 是否定位本文 novelty？
4. 是否为 baseline 选择提供依据？
5. 是否覆盖最相关和最新的强方法？
6. 是否避免文献堆砌？
```

## 9. 结果、分析或论证主体的组织方式

计算论文主体应围绕评价问题组织，而不是按实验运行顺序组织。

### 9.1 推荐 Results / Analysis 结构

```text
6.1 Main Results
6.2 Comparison with Baselines
6.3 Ablation Study
6.4 Robustness Analysis
6.5 Generalization Analysis
6.6 Efficiency Analysis
6.7 Error Analysis
6.8 Case Study / Qualitative Analysis
6.9 Failure Cases
```

或按评价问题组织：

```text
RQ1：本文方法是否优于强 baseline？
RQ2：哪些模块带来主要提升？
RQ3：方法是否能泛化到新域？
RQ4：方法是否对噪声鲁棒？
RQ5：方法的计算代价是什么？
RQ6：方法在哪些情况下失败？
```

### 9.2 主结果叙事

主结果不应只写：

```text
Our method achieves SOTA.
```

应写：

```text
1. 主要提升在哪里；
2. 与哪个 baseline 比较；
3. 在哪些数据集上提升更明显；
4. 哪些数据集提升有限；
5. 结果是否与方法设计预期一致；
6. 后续分析将解释什么。
```

### 9.3 消融叙事

消融实验应明确对应方法贡献。

弱消融：

```text
去掉模块 A 后性能下降。
```

强消融：

```text
去掉全局记忆模块后，长文档数据集上的 F1 下降 4.2，而短文本数据集仅下降 0.8。这表明该模块主要贡献于跨段落依赖建模，与本文设计动机一致。
```

### 9.4 鲁棒性与泛化叙事

不要只报告一个平均数，应展示趋势：

```text
随着噪声比例从 0% 增加到 30%，baseline 性能下降 18 个点，而本文方法下降 9 个点，说明一致性正则项提升了对输入扰动的稳定性。
```

### 9.5 误差分析叙事

推荐表格：

| 错误类型 | 比例 | 示例 | 可能原因 | 改进方向 |
|---|---:|---|---|---|

误差分析应连接方法假设：

```text
大多数错误集中在需要外部知识的样本上，说明本文模型虽然改善了上下文建模，但仍无法弥补知识缺失。
```

### 9.6 失败案例叙事

失败案例不是削弱论文，而是限定边界。

```text
在跨领域测试中，模型常将领域特定术语误判为已知类别。这表明模型对训练域词汇仍存在依赖，也解释了其在 Domain C 上提升有限。
```

### 9.7 定性案例

定性案例应避免只挑成功案例。

应包括：

- 成功案例；
- 失败案例；
- 与 baseline 对比；
- 模型输出解释；
- 为什么该案例有代表性。

### 9.8 常见错误

- Results 与 Analysis 分离过度，结果没有解释；
- Analysis 与方法贡献不对应；
- 消融太弱；
- 错误分析只讲几个趣味例子；
- 只展示成功案例；
- 不报告计算成本；
- 不解释负结果；
- 所有结果都写成“验证有效”，缺少边界。

## 10. 讨论与结论写法

## 10.1 Discussion 的功能

Discussion 应把实验结果提升为计算研究贡献。它回答：

```text
实验结果说明了什么？
方法为什么有效？
哪些设计是关键？
结果与已有研究有什么关系？
哪些任务或条件下方法不适用？
计算成本、数据偏差和伦理风险是什么？
```

### 推荐 Discussion 结构

```text
1. 回顾关键结果；
2. 解释方法有效性的机制；
3. 讨论不同数据集 / 任务上的差异；
4. 讨论与已有方法的关系；
5. 讨论效率、鲁棒性、泛化和可复现；
6. 说明限制和风险；
7. 给出未来方向。
```

### 结果到贡献的提升

结果层：

```text
本文方法在三个数据集上平均提升 2.5 F1。
```

分析层：

```text
消融表明主要提升来自全局记忆模块。
```

解释层：

```text
长度分组分析显示提升集中在长距离依赖样本上。
```

贡献层：

```text
因此，本文的贡献不是单纯提升分数，而是证明了显式建模跨段落状态对长文档事件抽取具有实际价值。
```

### 与已有研究对话

弱写法：

```text
我们的结果优于已有方法。
```

强写法：

```text
与直接扩展上下文窗口的方法相比，本文方法在长文档上获得相近或更高性能，同时降低推理成本。这表明在该任务中，显式层次建模可能比单纯扩大上下文更有效。
```

### 局限写法

弱写法：

```text
未来可以扩展更多数据集。
```

强写法：

```text
当前实验主要覆盖新闻和百科领域，尚不能证明模型在对话、法律或医学长文本中同样有效。由于这些领域具有不同的实体密度和篇章结构，未来需要跨领域数据进一步验证。
```

### Conclusion 写法

结论应简洁回答：

```text
本文提出了什么？
解决了什么任务缺口？
实验如何支持？
分析揭示了什么？
边界是什么？
```

### 常见错误

- Discussion 重复 Results；
- 不解释为什么有效；
- 不讨论负结果；
- 不讨论计算成本；
- 不讨论数据偏差；
- 不讨论可复现；
- Conclusion 只说 SOTA；
- 结论过度推广。

## 11. 审稿人关注点

计算类论文审稿人通常会重点质疑以下问题。

### 11.1 任务是否清楚且重要

审稿人会问：

```text
1. 任务定义是否清楚？
2. 输入输出是否明确？
3. 任务是否有实际或学术意义？
4. 与已有任务相比新在哪里？
5. 是否只是人为构造的小问题？
```

### 11.2 novelty 是否足够

审稿人会问：

```text
1. 方法是否只是已有模块组合？
2. 新模块是否真正必要？
3. 与最相近工作相比差异在哪里？
4. 贡献是算法、模型、数据、benchmark、训练策略还是分析？
5. 是否只是小幅调参或工程技巧？
```

### 11.3 baseline 是否弱

审稿人会问：

```text
1. 是否比较了当前强 baseline？
2. 是否遗漏最相关方法？
3. baseline 是否合理调参？
4. 是否使用官方实现？
5. 是否在同等数据和资源下比较？
```

### 11.4 数据是否有问题

审稿人会问：

```text
1. 数据来源是否可靠？
2. 划分是否合理？
3. 是否存在数据泄漏？
4. 是否存在重复样本？
5. 是否有标签噪声？
6. 数据是否偏向某些群体或场景？
7. 数据是否有许可或伦理风险？
```

### 11.5 指标是否合适

审稿人会问：

```text
1. 指标是否对应任务目标？
2. 是否只选择对作者有利的指标？
3. 对不平衡数据是否使用合适指标？
4. 生成任务是否需要人工评价？
5. 效率主张是否有成本指标？
```

### 11.6 消融和分析是否充分

审稿人会问：

```text
1. 核心模块是否被消融？
2. 消融是否证明设计必要？
3. 是否有鲁棒性或泛化分析？
4. 是否有错误分析？
5. 是否解释了为什么有效？
```

### 11.7 实验是否可复现

审稿人会问：

```text
1. 是否报告超参数？
2. 是否报告随机种子？
3. 是否多次运行？
4. 是否报告硬件？
5. 是否提供代码、数据或伪代码？
6. 是否说明预处理和划分？
```

### 11.8 结论是否过强

审稿人会问：

```text
1. 是否把同分布结果说成泛化？
2. 是否把 benchmark 分数说成真实应用成功？
3. 是否把单一任务结果推广到所有任务？
4. 是否把小幅提升夸大成重大突破？
5. 是否忽略计算成本？
```

## 12. 常见失败模式

## 12.1 只写 SOTA，缺少解释

表现：

```text
本文方法在多个数据集上取得 SOTA。
```

问题：

- 不知道为什么有效；
- 不知道在哪些样本上有效；
- 不知道是否可泛化；
- 不知道设计是否必要。

修正：

```text
1. 增加消融实验；
2. 增加分组分析；
3. 增加误差分析；
4. 增加效率和鲁棒性分析；
5. 将“分数提升”连接到方法设计。
```

## 12.2 baseline 弱或不公平

表现：

- 只比较旧方法；
- 不比较最强方法；
- baseline 未调参；
- 使用不同训练数据。

修正：

```text
1. 增加强 baseline；
2. 使用官方代码或说明复现；
3. 统一数据、指标、资源；
4. 报告超参数；
5. 降低无法公平比较的结论。
```

## 12.3 贡献增量小

表现：

- 只是加一个模块；
- 只是组合已有方法；
- 提升很小；
- 没有解释为什么重要。

修正：

```text
1. 明确贡献类型；
2. 证明该模块解决特定任务难点；
3. 用诊断实验展示非平凡收益；
4. 说明在哪些条件下提升明显；
5. 避免夸大。
```

## 12.4 数据泄漏或划分不清

表现：

- 不说明划分；
- 训练和测试有重复；
- 时间序列随机划分；
- 使用测试集调参。

修正：

```text
1. 说明划分单位；
2. 去重；
3. 使用严格训练 / 验证 / 测试；
4. 对时间数据使用时间切分；
5. 报告 contamination 检查。
```

## 12.5 指标与任务不匹配

表现：

- 类别不平衡却只报 accuracy；
- 生成任务只报自动指标；
- 效率主张不报成本；
- 鲁棒性主张只报干净测试集。

修正：

```text
1. 选择与任务目标匹配的指标；
2. 增加补充指标；
3. 解释每个指标测量什么；
4. 不用单一指标支撑全部主张。
```

## 12.6 消融不足

表现：

- 只去掉无关模块；
- 不消融核心模块；
- 消融没有解释。

修正：

```text
1. 为每个方法主张设计消融；
2. 去掉、替换、简化关键模块；
3. 分析哪些样本受影响；
4. 将结果回扣设计动机。
```

## 12.7 可复现性不足

表现：

- 不写超参数；
- 不写随机种子；
- 不写硬件；
- 不说明预处理；
- 不提供代码或伪代码。

修正：

```text
1. 补充实验设置表；
2. 报告训练细节；
3. 报告计算资源；
4. 说明代码和数据可用性；
5. 在附录中放完整超参数。
```

## 12.8 过度泛化

表现：

- 单一 benchmark 结果推广到实际应用；
- 同分布测试推广到 OOD；
- 小模型实验推广到大模型；
- 低资源实验推广到所有资源条件。

修正：

```text
1. 限定结论范围；
2. 增加泛化实验；
3. 在 Limitations 中说明未验证条件；
4. 使用谨慎表达。
```

## 13. Agent 写作辅助策略

## 13.1 Agent 总体工作流程

Agent 辅助计算类论文时，应按以下流程诊断：

```text
1. 识别论文类型：算法、模型、数据集、benchmark、应用或分析论文。
2. 提取任务定义：输入、输出、目标、假设。
3. 判断贡献类型：新模型、新数据、新任务、新协议等。
4. 检查方法主张：方法到底声称改善什么。
5. 检查评价协议：数据、baseline、指标、训练设置是否支撑主张。
6. 检查实验结果：结果是否稳定、公平、充分。
7. 检查分析：消融、鲁棒性、泛化、误差、失败案例。
8. 检查可复现性：代码、数据、种子、超参数、资源。
9. 检查结论强度：是否超过实验范围。
10. 重构论文故事线。
```

## 13.2 Agent 如何检查实验设计

使用“主张—实验—证据”矩阵：

| 方法主张 | 必要实验 | 当前是否有 | 问题 | 建议 |
|---|---|---|---|---|
| 准确率提升 | 主结果 + 强 baseline |  |  |  |
| 模块有效 | 消融 |  |  |  |
| 鲁棒 | 扰动 / 噪声 / OOD |  |  |  |
| 泛化 | 跨域 / 跨数据集 |  |  |  |
| 高效 | 延迟 / 参数 / 显存 |  |  |  |
| 可解释 | 解释指标 / 案例 |  |  |  |

Agent 应优先找断裂：

```text
1. 声称了但没有实验；
2. 有实验但指标不对；
3. 有指标但 baseline 不公平；
4. 有结果但没有解释；
5. 有解释但没有证据。
```

## 13.3 Agent 如何连接方法贡献和实验结果

Agent 可使用以下模板：

```text
方法设计 X 旨在解决任务挑战 Y。
为了验证 X 是否有效，实验 Z 比较了完整模型与去掉 X 的变体。
结果显示……，说明……
进一步的分组 / 错误 / 鲁棒性分析表明……
因此，实验结果支持 / 部分支持 / 尚不足以支持该方法主张。
```

示例：

```text
全局记忆模块旨在解决长文档中的跨段落依赖问题。消融实验显示，去掉该模块后长文档数据集上的 F1 下降 4.2，而短文本数据集仅下降 0.8。长度分组分析进一步显示，提升主要来自最长 25% 样本。因此，实验结果支持该模块改善长距离依赖建模的主张。
```

## 13.4 Agent 如何重构 Results

Agent 可把 Results 从排行榜改写为评价问题：

```text
原结构：
6.1 Dataset A results
6.2 Dataset B results
6.3 Dataset C results

重构：
6.1 Overall performance against strong baselines
6.2 Contribution of each model component
6.3 Robustness under input perturbations
6.4 Generalization across domains
6.5 Error patterns and failure cases
```

## 13.5 Agent 如何降级过度声称

高风险词：

```text
SOTA
通用
鲁棒
泛化
高效
可解释
解决了
证明了
显著优于
无需标注
适用于所有
完全避免
```

改写规则：

| 过强表达 | 更稳妥表达 |
|---|---|
| 本文方法解决了跨域泛化问题 | 本文方法在所测试的跨域设置中缓解了性能下降 |
| 本文模型具有强鲁棒性 | 本文模型在特定噪声扰动下表现出更稳定性能 |
| 本文方法全面优于已有方法 | 本文方法在主要指标上优于所选 baseline，但在效率上存在额外成本 |
| 实验证明模块 X 有效 | 消融结果支持模块 X 对性能提升有贡献 |
| 该方法适用于所有任务 | 该方法可能适用于具有类似输入结构的任务 |

## 13.6 Agent 如何检查可复现性

Agent 应逐项检查：

```text
1. 数据集和划分是否清楚？
2. 预处理是否清楚？
3. baseline 实现是否清楚？
4. 超参数是否清楚？
5. 随机种子是否清楚？
6. 运行次数是否清楚？
7. 硬件和软件环境是否清楚？
8. 代码和数据可用性是否说明？
9. 结果是否包含方差？
10. 是否避免测试集调参？
```

## 13.7 Agent 如何辅助数据集 / benchmark 论文

Agent 应特别检查：

```text
1. 数据来源是否合法、透明；
2. 标注协议是否清楚；
3. 标注者、标注流程、质量控制是否说明；
4. 数据统计是否充分；
5. 数据偏差是否分析；
6. benchmark 是否真的测目标能力；
7. baseline 是否覆盖不同方法类型；
8. 数据泄漏和污染风险是否讨论；
9. 使用许可、隐私和伦理是否说明。
```
